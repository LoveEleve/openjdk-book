# 第1章：Launcher Chain —— 从命令行到 libjvm.so

上一卷最后你亲手编译了一个 JDK。敲 `java -version` 的时候，从屏幕上看到三行输出，觉得挺自然——JDK 嘛，一个命令启动一个 JVM，很正常。

但如果你稍微想一下：这个 `java` 可执行文件才 23KB，而 JVM 本体 `libjvm.so` 有 164MB。一个 23KB 的 C 程序，怎么启动一个 164MB 的 Java 虚拟机？

本章就顺着这个好奇走下去——从 `main()` 开始，到 `dlopen("libjvm.so")` 为止。

---

## java 没链接 libjvm.so

先验证一个反直觉的事实。用 `ldd` 看 java 链接了哪些库：

```
$ ldd build/linux-x86_64-normal-server-slowdebug/jdk/bin/java
	libjli.so => .../jdk/bin/./../lib/jli/libjli.so
	libc.so.6 => /lib64/libc.so.6
	libz.so.1 => /lib64/libz.so.1
	/lib64/ld-linux-x86-64.so.2
```

三个依赖：`libc`（标准 C 库）、`libz`（压缩）、`libjli.so`。没有 `libjvm.so`。

你用 `readelf` 再看一遍 RUNPATH——java 只需要找到 `libjli.so`：

```
$ readelf -d build/linux-x86_64-normal-server-slowdebug/jdk/bin/java | grep RUNPATH
  (RUNPATH)            Library runpath: [$ORIGIN/../lib/jli]
```

`$ORIGIN` 是 ELF 动态链接器的特殊变量，展开为 java 二进制所在的目录。所以运行时找 `libjli.so` 就是 `java所在目录/../lib/jli/`。这个 RUNPATH 里只有 jli——没有 server、没有 libjvm、什么都没有。

`libjvm.so` 不是静态链接进来的，也不是动态链接器自动加载的。它是 java 启动器在运行中**手动**加载的。

这就是本章要讲的：启动器怎么找到 libjvm.so、怎么打开它、怎么从里面掏出 `JNI_CreateJavaVM` 这个函数。

---

## main() 在做什么——不是在"执行 Java"

java 的 `main()` 在 `src/java.base/share/native/launcher/main.c`。和 Ch0.1 那个 hello world 一样——标准 C 程序入口。但它不解析 classpath、不创建虚拟机——它只是**准备参数**。

搞懂这 150 行代码之前，先认识所有变量。这段代码名字起得特别容易混，列出来：

| 变量 | 类型 | 从哪来 | 干什么 |
|------|------|--------|--------|
| `argc` | `int` | OS 传给 `main()` | argv 的元素个数 |
| `argv` | `char **` | OS 传给 `main()` | 原始命令行参数数组 |
| `margc` | `int` | 函数内计算 | 最终传给 JLI_Launch 的参数个数 |
| `margv` | `char **` | 函数内构建 | 最终传给 JLI_Launch 的参数数组 |
| `jargc` | `int` | 编译期计算 | 编译期 Java 参数个数 |
| `jargv` | `char **` | 编译期常量 | 编译期 Java 参数数组 |
| `args` | `JLI_List` | 函数内临时变量 | 构建 margv 过程中的中转站 |

一句话区分：`argc/argv` 是 OS 给的，`margc/margv` 是加工后给 JLI_Launch 的，`jargc/jargv` 是编译期写死的，`args` 是临时的。

比如你敲 `java -version`，OS 传给 `main()` 的就是：

```
argc = 2
argv[0] = "java"
argv[1] = "-version"
```

`argv[0]` 是程序名，参数从 `argv[1]` 开始——C 语言的标准约定。

搞清这些之后，看一眼整个 `main()` 的骨架：

```c
// main.c:97  — 函数入口
JNIEXPORT int
main(int argc, char **argv) {
    int margc, jargc;
    char **margv, **jargv;

    // ◇ 阶段A：构建 jargv（编译期参数）
    main_jargc = (sizeof(const_jargs) / sizeof(char *)) > 1
                 ? sizeof(const_jargs) / sizeof(char *) : 0;
    // ... 组合 EXTRA_JAVA_ARGS + JAVA_ARGS → jargv

    JLI_InitArgProcessing(jargc > 0, const_disable_argfile);

    // ◇ 阶段B：Windows 参数处理（本书跳过）
    #ifdef _WIN32
    // UTF-16 → ANSI 编码转换 + CmdToArgs，不展开
    #else

    // ◇ 阶段C★：构建 margv（Linux x86_64）
    JLI_List args = JLI_List_new(argc + 1);
    JLI_List_add(args, JLI_StringDup(argv[0]));       // 程序名
    JLI_AddArgsFromEnvVar(args, JDK_JAVA_OPTIONS);    // 环境变量注入
    for (i = 1; i < argc; i++) {                      // @argfile 展开
        JLI_List argsInFile = JLI_PreprocessArg(argv[i], JNI_TRUE);
        // ...
    }
    margc = args->size;
    margv = args->elements;

    #endif

    // ◇ 阶段D★：硬编码参数覆盖（jdk11u-copy 定制）
    if (const_progname != NULL && JLI_StrCmp(const_progname, "java") == 0) {
        static char *hardcoded_argv[] = { "java", "-version", NULL };
        margc = ...; margv = hardcoded_argv; jargc = 0; jargv = NULL;
    }

    // ◇ 阶段E：交接给 libjli
    return JLI_Launch(margc, margv, jargc, (const char**) jargv, ...);
}
```

下面按代码的实际执行顺序走——CPU 跑到的顺序。

### JNIEXPORT / JNICALL —— JNI 边界的标记

入口那行有个你写的 hello world 没有的关键字：

```c
// main.c:97
JNIEXPORT int
main(int argc, char **argv)
```

`JNIEXPORT` 是宏，在 `src/hotspot/os/linux/jni_md.h:34` 展开为：

```c
#define JNIEXPORT __attribute__((visibility("default")))
```

`__attribute__((visibility("default")))` 是 GCC 编译指令——"这个符号对外可见"。对可执行文件的 `main()` 没啥实际作用，但 OpenJDK 用它保持 JNI 边界代码的风格统一。另一个 `JNICALL` 同理——Linux 上空，Windows 上 `__stdcall`。

入口之后，声明三板斧：`int margc, jargc`、`char **margv, **jargv`、`const_javaw = JNI_FALSE`。然后进入第一个实际逻辑。

### 第一步（真实顺序）：构建 jargv —— 编译期写死的参数

`jargv` 不走命令行。它来自编译时常量 `JAVA_ARGS` 和 `EXTRA_JAVA_ARGS`：

```c
// main.c:105-139
    main_jargc = (sizeof(const_jargs) / sizeof(char *)) > 1
                 ? sizeof(const_jargs) / sizeof(char *) : 0;

    extra_jargc = (sizeof(const_extra_jargs) / sizeof(char *)) > 1
                  ? sizeof(const_extra_jargs) / sizeof(char *) : 0;
```

`sizeof(const_jargs)` 除以 `sizeof(char *)`——数组总字节数除以每个元素的字节数，得到元素个数。`> 1` 的检查是因为数组至少有两个元素：至少一个内容加上末尾的 NULL。如果只有一个元素（只有一个 NULL），说明这个启动器没有编译期 Java 参数。

`java` 启动器的 `const_jargs` 定义为：

```c
// defines.h (CMake 生成)
static const char* const_jargs[] = { NULL };
```

所以 `main_jargc = 0`，`jargv = NULL`。

但编 `javac` 时，CMake 传入了 `-DJAVA_ARGS='{ "-J-ms8m", "-m", "jdk.compiler/..." }'`：

```c
// javac 的 defines.h
static const char* const_jargs[] = { "-J-ms8m", "-m", "jdk.compiler/...", NULL };
```

`-J` 前缀的意思是"这个参数给 JVM，不给 Java 程序"。所以 `jargv` 里的东西最终变成 JVM 的启动参数，而 `margv` 里不带 `-J` 的才是传给 `main(String[] args)` 的。

如果 `extra_jargc` 也有值，两个数组合并——extra 在先、main 在后，后者覆盖前者。

### 第二步：InitArgProcessing —— 参数预处理的开关

```c
// main.c:141
    JLI_InitArgProcessing(jargc > 0, const_disable_argfile);
```

这个函数设置两个全局标志。第一个 `jargc > 0` = "有没有编译期 Java 参数"，决定了后续是否处理 `-J` 前缀。第二个 `const_disable_argfile` = jdk11u-copy 没设，所以 `@opts.txt` 展开正常支持。

### 第三步（平台分支）：Linux x86_64 路径

```c
// main.c:183-222
#ifdef _WIN32
    // UTF-16 → ANSI 编码转换，本书跳过
#else
    // Linux x86_64 走这里
```

`#ifdef _WIN32` 后面的代码是 Windows 专用的——UTF-16 命令行到 ANSI 的转换。本书不展开，直接看 `#else` 分支。

`margv` 构建用了一个自定义的数据结构，先把它看清楚。

**JLI_List**——不是 C 标准库的东西，是启动器在 `jli_util.h` 里手搓的动态数组：

```c
typedef struct {
    int       size;      // 当前元素数
    int       capacity;  // 总容量
    char    **elements;  // 字符串数组
} JLI_List;
```

> **`char **elements` 为什么是"字符串数组"？**
>
> C 里没有 `string` 类型。一个字符串是 `char *`——指向一串连续 `char` 的首地址。多个字符串放到一起，就是一个 `char *` 的数组，内存里是"指针的指针"，也就是 `char **`。
>
> 比如 `margv` 存了 `"java"`、`"-version"`、`NULL` 三个字符串：
>
> ```
> elements → [0] → "java\0"
>             [1] → "-version\0"
>             [2] → NULL
> ```
>
> `elements[0]` 是 `char *`，指向 `"java"`；`elements[1]` 指向 `"-version"`。末尾的 `elements[2]` 是 `NULL`（空指针），不指向任何字符串——它只标记数组终点。
>
> C 里所有参数传递（`main` 的 `argv`、`JLI_Launch` 的 `margv`）都用这个模式：二级指针 + NULL 哨兵。

三个操作：`JLI_List_new(capacity)` 分配、`JLI_List_add(list, str)` 追加（满了自动翻倍扩容）、`JLI_List_add(list, NULL)` 加哨兵。`JLI_StringDup` 内部调 `malloc + memcpy`——因为 `argv` 是操作系统分配的内存，启动器不拥有它，复制一份才能随意管理生命周期。

回到 margv 构建。这里的 `args` 就是上面速查表里的那个临时 JLI_List：

```c
    JLI_List args = JLI_List_new(argc + 1);
    JLI_List_add(args, JLI_StringDup(argv[0]));       // 从 argv[0] 复制 "java" 到 args
```

`JLI_StringDup(argv[0])` 把 `argv[0]`（就是 `"java"` 这个程序名）复制一份到堆上。`JLI_List_add` 把这个副本的指针放进 `args` 列表。注意方向——是从 `argv` 搬到 `args`，不是反过来。

此时 `args` 里第一个元素是 `"java"`。

```c
    if (JLI_AddArgsFromEnvVar(args, JDK_JAVA_OPTIONS)) {
```

注入 `JDK_JAVA_OPTIONS` 环境变量。内部是 `getenv` + 空格分割 + 逐个 `JLI_List_add`。环境变量没设就什么都不加。注意它**在 `argv[1]` 之前**——所以命令行上的同名参数会覆盖环境变量里的。

> 你大概率永远不会设 `JDK_JAVA_OPTIONS`——它只在 Docker 镜像、CI 流水线这种需要全局注入 JVM 参数的场景才用。理解"有这条路径就够了"，不用记细节。

举例：设了 `export JDK_JAVA_OPTIONS="-Xmx128m"`，然后敲 `java -Xmx256m -version`。margv 顺序：`"java"` → `"-Xmx128m"`（环境变量）→ `"-Xmx256m"`（命令行）→ `"-version"`。JVM 取最后一个 `-Xmx`——命令行 256m 生效，环境变量 128m 被覆盖。

```c
    for (i = 1; i < argc; i++) {
        JLI_List argsInFile = JLI_PreprocessArg(argv[i], JNI_TRUE);
```

逐个处理 `argv[1]`、`argv[2]`……`JLI_PreprocessArg` 负责 `@filename` 展开。如果你写 `java @opts.txt`，`opts.txt` 的内容会被拆成参数插入。嵌套展开也支持。

> `@argfile` 也很少用到——Java 命令行太长时才会把参数存文件里。日常敲 `java -version` 不用管它。

```c
    margc = args->size;
    margv = args->elements;
```

收尾：NULL 哨兵、返回数组指针。margv 构建完成。

### 第四步：硬编码参数覆盖 —— CLion 调试不用配参数

```c
// main.c:225-241
    if (const_progname != NULL && JLI_StrCmp(const_progname, "java") == 0) {
        static char *hardcoded_argv[] = { "java", "-version", NULL };
        margc = (int) (sizeof(hardcoded_argv) / sizeof(hardcoded_argv[0])) - 1;
        margv = hardcoded_argv;
        jargc = 0;
        jargv = NULL;
    }
```

`const_progname` 是 CMake 编译期常量——编 `java` 启动器时等于 `"java"`，编 `javac` 时等于 `"javac"`。所以只有 `java` 会走进这个 `if`。`JLI_StrCmp` 是启动器自己的字符串比较，等价于 `strcmp`。

这整段代码就是 CLion 里点三角号就能跑的秘密——不用去 Run Configuration 配参数。它覆盖了前面所有构建出来的 `margv` 和 `jargv`。

### 第五步：交接给 libjli

```c
// main.c:242-251
    return JLI_Launch(margc, margv,
                   jargc, (const char**) jargv,
                   0, NULL,
                   VERSION_STRING, DOT_VERSION,
                   (const_progname != NULL) ? const_progname : *margv,
                   (const_launcher != NULL) ? const_launcher : *margv,
                   jargc > 0, const_cpwildcard, const_javaw, 0);
```

11 个参数逐一解释：

| 参数 | 值 | 含义 |
|------|-----|------|
| `margc, margv` | 命令行参数 | 你敲的东西 |
| `jargc, jargv` | 编译期参数 | java 启动器为空 |
| `0, NULL` | appclassc/appclassv | applet 用的，JDK 11 已废弃 |
| `VERSION_STRING` | `"11.0.32-internal"` | 完整版本号 |
| `DOT_VERSION` | `"11.0"` | 点分主版本 |
| `const_progname` | `"java"` | 程序名 |
| `const_launcher` | `"openjdk"` | 启动器名 |
| `jargc > 0` | `JNI_FALSE` | 有没有编译期 Java 参数 |
| `const_cpwildcard` | `JNI_TRUE` | classpath 支持 `*` 通配符 |
| `const_javaw` | `JNI_FALSE` | Windows javax 模式，Linux 始终 false |
| 最后一个 `0` | 0 | 保留参数，源码标注 UNUSED |

`JLI_Launch` 在 `src/java.base/share/native/libjli/java.c`——从 `main.c` 跨到 libjli 库的第一个函数。

**到这里为止，`main()` 干了什么？** 把命令行参数拆成两套数组（用户敲的 `margv`、编译期写死的 `jargv`），在 Linux 平台上跳过 Windows 分支，最后如果 `progname == "java"` 就用硬编码数组覆盖全部。然后把这些参数连同版本号、启动器名、通配符开关等一起打包，调用 `JLI_Launch`。150 行 C 代码，没有一行和 JVM 有关——纯粹的准备参数。

## JLI_Launch() —— 11 步流水线

`main()` 最后一行的 `JLI_Launch` 在 `java.c` 中。和 `main()` 一样——先看结构全貌，再逐个深入。

这个函数比 `main()` 复杂得多——**11 个步骤**，跨 4 个 `.c` 文件。下面标注了每一步的重要程度，★（了解）、★★（概述）、★★★（详细走读）：

```c
// java.c:234-344 — 平台无关代码
JLI_Launch(argc, argv, jargc, jargv, appclassc, appclassv,
           fullversion, dotversion, pname, lname,
           javaargs, cpwildcard, javaw, ergo)
{
│
├─ Step 1: 设置全局状态                         (★)
│     _fVersion, _launcher_name, _program_name, _is_java_args, _wc_enabled
│
├─ Step 2: Launcher 初始化 + 追踪               (★★)
│     InitLauncher → DumpState → 打印 jargv/argv
│
├─ Step 3: JRE 版本选择                         (★★)
│     SelectVersion; JDK 9+ 拒绝 -version: 语法
│
├─ Step 4: 构建执行环境 ★★★                     核心
│     CreateExecutionEnvironment() → jrepath, jvmpath, jvmcfg
│     └→ SetExecname + GetJREPath + ReadKnownVMs + CheckJvmType + GetJVMPath
│
├─ Step 5: 设置 JVM 环境变量                     (★★)
│     SetJvmEnvironment(argc, argv) → JAVA_HOME, LD_LIBRARY_PATH
│
├─ Step 6: 加载 JVM 动态库 ★★★                  核心

│     └→ dlopen + dlsym × 3 (JNI_CreateJavaVM 等)
│     └→ 失败返回 exit code 6
│
├─ Step 7: 参数预处理                           (★★)
│     ++argv; --argc; (跳过程序名)

│
├─ Step 8: 解析命令行选项 ★★★                   核心

├─ Step 9: -jar 特殊处理                        (★)

│     sun.java.command / sun.java.launcher / sun.java.launcher.pid
│
└─ Step 11: 启动 JVM ★★★                       核心
```

下面从 Step 1 走起。非核心的步骤（★/★★）几句话带过，核心的（★★★）逐步详讲。

### Step 1-3：开场准备

Step 1 把五个模块级全局变量赋了初值——程序名、启动器名、版本号、是否工具启动器、通配符开关。没有什么逻辑，就是把传进来的参数存到全局，后面所有函数都能访问。

Step 2 调了 `InitLauncher` 做平台初始化（信号处理等），然后如果启用了追踪模式（`JLI_IsTraceLauncher()`），打印当前的 jargv 和 argv 内容到 stdout。日常跑没有追踪模式，这一步等于空操作。

Step 3 调 `SelectVersion`，检查用户有没有用 `-version:` 语法指定 JRE 版本。JDK 9+ 已经不支持多 JRE 共存，所以这个函数的主要功能就是拒绝这种调用。

> Step 1-3 日常敲 `java -version` 都不会触发任何实际动作。知道它们存在就行。

### Step 4★：构建执行环境 —— 找到 libjvm.so

这是第一个核心步骤。`CreateExecutionEnvironment` 在 `java_md_solinux.c:303-483`，负责确定三个关键路径：

```c
// java.c:278-281
    CreateExecutionEnvironment(&argc, &argv,
                               jrepath, sizeof(jrepath),
                               jvmpath, sizeof(jvmpath),
                               jvmcfg,  sizeof(jvmcfg));
```

`jrepath` 是 JRE 的根目录，`jvmcfg` 是 `jvm.cfg` 的内容，`jvmpath` 是 `libjvm.so` 的绝对路径。这三个值都不是编译期写死的——它们是在运行中通过六个子步骤推导出来的：

**第一步：SetExecname。** 读 `/proc/self/exe` 获取 java 可执行文件的实际路径。`/proc/self/exe` 总是指向当前进程的可执行文件，不管你的 `$PWD` 在哪。

**第二步：GetApplicationHome。** 从可执行文件路径往上找——先去掉 `/bin/java`，再去掉 `/lib/libjli.so`（如果是从 jli 启动的），剩下的就是 JDK 的根目录。

**第三步：GetJREPath。** 三层回退确认 JRE 目录：
1. 同目录下找 `lib/libjava.so`——存在就用这个目录
2. JDK 根目录下找 `jre/lib/libjava.so`——老版本 JDK 的布局
3. 用 `dladdr` 反查当前代码段（libjli.so）所在的目录，往上找到 `lib/libjava.so`

**第四步：ReadKnownVMs。** 打开 `lib/jvm.cfg`，逐行解析。每个 VM 类型（如 `-server`）对应一个描述结构体，支持 `KNOWN`（存在）、`ALIASED_TO`（别名）、`WARN`/`IGNORE`/`ERROR`（各种问题标记）。JDK 11 的 `jvm.cfg` 通常只有一行：

```
-server KNOWN
```

**第五步：CheckJvmType。** 从三类来源中确定 VM 类型。优先级：命令行 `-client`/`-server` 参数 > `JDK_ALTERNATE_VM` 环境变量 > jvm.cfg 的第一条记录。同时把 argv 中的 VM 类型参数移除（`&argv` 是传地址，会修改调用者的数据）。

**第六步：GetJVMPath。** 用确定的 JRE 路径和 VM 类型拼接出 `libjvm.so` 的完整路径：

```c
JLI_Snprintf(jvmpath, so_jvmpath, "%s/lib/%s/%s/libjvm.so",
             jrepath, GetArch(), jvmtype);
```

`GetArch()` 是编译期宏——x86-64 上返回 `"amd64"`。实际路径如：

```
/data/workspace/jdk11u-copy/build/linux-x86_64-normal-server-slowdebug/jdk/lib/amd64/server/libjvm.so
```

`stat()` 验证文件存在。到此，`jvmpath` 确定了一个真实存在的 `.so` 文件路径。

> 六步走完，`jrepath`、`jvmcfg`、`jvmpath` 三个变量都有了确定值。日常使用中不会失败——除非你把 JDK 目录结构搞坏了。

### Step 5：设置 JVM 环境变量

```c
// java.c:283-285
    if (!IsJavaArgs()) {
        SetJvmEnvironment(argc, argv);
    }
```

`IsJavaArgs()` 判断当前是不是工具启动器（javac/javap 等）。不是工具启动器时，从命令行参数里读 `-Djava.home` 来设置 `JAVA_HOME`，从 `-Djava.library.path` 设置 `LD_LIBRARY_PATH`。java 启动器通常没有这些参数——这一步日常等于空操作。

### Step 6★：加载 JVM 动态库

```c
// java.c:287-303
    ifn.CreateJavaVM = 0;
    ifn.GetDefaultJavaVMInitArgs = 0;

    if (!LoadJavaVM(jvmpath, &ifn)) {
        return(6);
    }
```

`LoadJavaVM` 是本章的核心——把它展开来讲。

#### InvocationFunctions：启动器与 JVM 的接口

`ifn` 是 `InvocationFunctions` 类型的变量，定义在 `java.h:79-87`：

```c
typedef jint (JNICALL *CreateJavaVM_t)(JavaVM **pvm, void **env, void *args);

typedef struct {
    CreateJavaVM_t CreateJavaVM;
    GetDefaultJavaVMInitArgs_t GetDefaultJavaVMInitArgs;
    GetCreatedJavaVMs_t GetCreatedJavaVMs;
} InvocationFunctions;
```

三个字段都是函数指针，初始化为 0（NULL）。`LoadJavaVM` 的任务是把它们填上：

```c
// java_md_solinux.c:553
jboolean
LoadJavaVM(const char *jvmpath, InvocationFunctions *ifn) {
    libjvm = dlopen(jvmpath, RTLD_NOW + RTLD_GLOBAL);
```

`dlopen` 在运行时打开 `libjvm.so`。两个 flag：`RTLD_NOW`（值 0x00002）要求立即解析所有符号——如果 libjvm.so 缺少任何依赖符号，现在就会失败，不会等到调用时才崩溃。`RTLD_GLOBAL`（值 0x00100）让 libjvm.so 的符号对后续 `dlopen` 加载的其他 .so 可见。

`dlopen` 失败最常见的三个原因：路径不存在（jvmpath 拼错了）、64/32 bit 不匹配、依赖库缺失。`dlerror()` 告诉你具体是哪个。

```c
    ifn->CreateJavaVM = (CreateJavaVM_t)
        dlsym(libjvm, "JNI_CreateJavaVM");
```

`dlsym` 的第二个参数是字符串 `"JNI_CreateJavaVM"`——在 libjvm.so 的符号表里搜索这个符号名，找到它的地址，转成函数指针。注意引号——不是变量名，不是宏，就是字符串。如果你写成 `dlsym(libjvm, JNI_CreateJavaVM)`（不带引号），编译器会把它当变量名处理，完全错误。

同样的模式重复三次：

```c
    ifn->GetDefaultJavaVMInitArgs = (GetDefaultJavaVMInitArgs_t)
        dlsym(libjvm, "JNI_GetDefaultJavaVMInitArgs");
    ifn->GetCreatedJavaVMs = (GetCreatedJavaVMs_t)
        dlsym(libjvm, "JNI_GetCreatedJavaVMs");
```

每个 `dlsym` 之后都检查 NULL——缺任何一个符号，JVM 不完整，必须失败。三次成功后，`ifn` 不再是一个空壳——它的三个字段都指向了 libjvm.so 里真正可调用的函数。`ifn->CreateJavaVM` 指向 `jni.cpp` 里的 `JNI_CreateJavaVM`——下一章就要走到那个函数里面。

### Step 7-10：参数拆解与系统属性

Step 7 跳过程序名（`++argv; --argc`），然后如果是工具启动器就调用 `TranslateApplicationArgs` 把 jargv 转换成 `-J` 前缀的参数，否则读取 `CLASSPATH` 环境变量。

Step 8 `ParseArguments` 是另一个大函数——~400 行，逐一消费命令行上的所有 `-` 选项（`-cp`、`-jar`、`-m` 等）。解析完成后，剩余的非选项参数就是 Java 程序的参数。返回值 `mode` 指示启动模式——`LM_CLASS`（普通类）、`LM_JAR`（jar 包）、`LM_MODULE`（模块）等。

Step 9 处理 `-jar` 场景：把 jar 路径设为 classpath。

Step 10 注入三个系统属性：`sun.java.command`（用户敲的完整命令）、`sun.java.launcher`（`SUN_STANDARD` 模式）、`sun.java.launcher.pid`（启动器 PID）。这些不是用户 `-D` 设置的——是启动器"伪造"的。

> Step 7-10 是启动器作为"命令行解析器"的主要工作。日常使用中它们确保你敲的 `-cp`、`-jar`、`MyClass` 被正确分类，系统属性被正确设置。理解它们做什么就行，不用深入内部细节。

### Step 11★：启动 JVM —— 在新线程中

```c
// java.c:343
    return JVMInit(&ifn, threadStackSize, argc, argv, mode, what, ret);
```

`JVMInit` 在 `java_md_solinux.c:806`，只有两行：

```c
    ShowSplashScreen();
    return ContinueInNewThread(ifn, threadStackSize, argc, argv, mode, what, ret);
```

`ShowSplashScreen`：如果有 `-splash` 参数，在一个独立窗口显示启动图片——JVM 启动完成后由 JVM 负责关掉它。

`ContinueInNewThread`：内部调 `pthread_create`，创建一个新线程执行 `JavaMain()`。为什么要开新线程？原始线程（调 `main()` 的那个）的栈大小由 OS 初始分配，信号处理器已绑定，这些都不可以改。JVM 需要精确控制栈大小和 guard pages——这些只能在 `pthread_create` 时设定。

JDK 6 早期有个 bug（JDK-6316197）就是因为主线程启动 JVM 导致 StackGuardPages 和 OS 默认栈大小冲突。修复方案就是永远在新线程中启动。

`JVMInit` 不会返回——它内部最终调 `exit()`。

## 本章总结

从你敲 `java -version`，到函数指针 `ifn->CreateJavaVM` 指向 `jni.cpp` 里的 `JNI_CreateJavaVM`，代码走完了三件事：

1. **准备参数**（main.c 150 行）：把命令行拆成 margv（用户的）和 jargv（编译期的），Linux 分支上构建参数数组，最后用硬编码覆盖。
2. **找到 JVM**（CreateExecutionEnvironment 六步）：`/proc/self/exe` → 回退找到 JDK 根目录 → 读 jvm.cfg → 拼出 libjvm.so 的绝对路径。
3. **打开 JVM 的门**（LoadJavaVM）：`dlopen` 加载 164MB 的 libjvm.so，`dlsym` 三个函数指针。

```c
// 三个关键文件，四个核心调用
main.c:      main() → JLI_Launch(margc, margv, ...)
java.c:      JLI_Launch() → CreateExecutionEnvironment() + LoadJavaVM() + JVMInit()
java_md_solinux.c:  LoadJavaVM() → dlopen(jvmpath) + dlsym × 3
```
