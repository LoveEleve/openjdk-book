# 第1章：Launcher Chain —— 从命令行到 libjvm.so

Ch0.1 告诉你 `java` 是一个 C 编译出来的可执行文件。本章顺着这个结论往下走——这个可执行文件从 `main()` 开始，到底干了什么，才会最终把你的 `HelloWorld.class` 跑起来？

不急着到 Java。先把它到 JVM 门口这段路走完。

---

## java 到底链接了什么

Ch0.1 用 `file` 验证了 java 是 ELF 可执行文件。再用 `ldd` 看它的依赖：

```
$ ldd build/linux-x86_64-normal-server-slowdebug/jdk/bin/java
	linux-vdso.so.1 (0x00007ffef4b7e000)
	libjli.so => .../jdk/bin/./../lib/jli/libjli.so (0x00007f75efa41000)
	libc.so.6 => /lib64/libc.so.6 (0x00007f75ef7ef000)
	libz.so.1 => /lib64/libz.so.1 (0x00007f75ef7d5000)
	/lib64/ld-linux-x86-64.so.2 (0x00007f75efa59000)
```

除了内核的 `vdso` 和标准 C 库，java 只依赖三个 .so：`libjli.so`、`libc.so.6`、`libz.so.1`。

**没有 `libjvm.so`。**

这不是意外。java 启动器不静态链接 JVM——它要到运行时才加载。`libjli.so`（JVM Launch Infrastructure）承担了这个角色：找到 libjvm.so、用 `dlopen` 打开它、用 `dlsym` 找到 `JNI_CreateJavaVM` 函数指针。

`readelf -d` 验证了这一点——java 的 RUNPATH 只指向 jli 目录：

```
$ readelf -d build/linux-x86_64-normal-server-slowdebug/jdk/bin/java | grep -i rpath
 0x000000000000001d (RUNPATH)            Library runpath: [$ORIGIN/../lib/jli]
```

`$ORIGIN` 是 ELF 的动态链接器变量，展开为 java 二进制所在的目录。所以运行时找 `libjli.so` 的路径是 `java所在目录/../lib/jli/libjli.so`。

## main() —— 不是"开始解释 Java"，是"准备参数"

java 的 `main()` 在 `src/java.base/share/native/launcher/main.c`。和 Ch0.1 里的 hello world 一样——标准的 C 程序入口：

```c
// main.c:97-104
JNIEXPORT int
main(int argc, char **argv) {
    int margc;
    char **margv;
    int jargc;
    char **jargv;
    const jboolean const_javaw = JNI_FALSE;
```

`margc`/`margv` 和 `jargc`/`jargv`——两套参数体系，这是理解启动器的第一个关键点。

### margv：来自命令行 + 环境变量

`margv` 是你敲的命令行参数，加上 `JDK_JAVA_OPTIONS` 环境变量注入：

```c
// main.c:183-222 — margv 构建
    JLI_List args = JLI_List_new(argc + 1);
    int i = 0;
    JLI_List_add(args, JLI_StringDup(argv[0]));    // 程序名

    if (JLI_AddArgsFromEnvVar(args, JDK_JAVA_OPTIONS)) {  // JDK_JAVA_OPTIONS 注入
        // ...
    }
    for (i = 1; i < argc; i++) {
        JLI_List argsInFile = JLI_PreprocessArg(argv[i], JNI_TRUE);
        if (NULL == argsInFile) {
            JLI_List_add(args, JLI_StringDup(argv[i]));
        } else {
            // @argfile 展开
            for (idx = 0; idx < cnt; idx++)
                JLI_List_add(args, argsInFile->elements[idx]);
        }
    }
    margc = args->size;
    JLI_List_add(args, NULL);
    margv = args->elements;
```

构建顺序：`argv[0]` → 环境变量注入 → `argv[1..n]` 逐个预处理（支持 `@argfile` 文件展开）→ 末尾 NULL 哨兵。

`JDK_JAVA_OPTIONS` 在 `argv[1]` 之前注入位置，意味着命令行参数可以覆盖环境变量中的选项——后者先入列，同名参数后者覆盖前者。

`JLI_PreprocessArg()` 处理 `@filename` 语法：把文件中的每一行当作一个参数展开到数组中。`java @opts.txt -version` 等价于把 `opts.txt` 的内容逐行插入到 `java` 和 `-version` 之间。

### jargv：编译期写死的 Java 参数

`jargv` 是另一套参数，来自编译时常量 `JAVA_ARGS` 和 `EXTRA_JAVA_ARGS`。它们通过宏定义写在源码里，**不走命令行**。

```c
// main.c:109-138 — jargv 构建
    main_jargc = (sizeof(const_jargs) / sizeof(char *)) > 1
                 ? sizeof(const_jargs) / sizeof(char *)
                 : 0;

    extra_jargc = (sizeof(const_extra_jargs) / sizeof(char *)) > 1
                  ? sizeof(const_extra_jargs) / sizeof(char *)
                  : 0;

    if (main_jargc > 0 && extra_jargc > 0) {
        jargc = main_jargc + extra_jargc;
        list = JLI_List_new(jargc + 1);
        for (i = 0; i < extra_jargc; i++)
            JLI_List_add(list, JLI_StringDup(const_extra_jargs[i]));
        for (i = 0; i < main_jargc; i++)
            JLI_List_add(list, JLI_StringDup(const_jargs[i]));
        JLI_List_add(list, NULL);
        jargv = list->elements;
    } else if (extra_jargc > 0) {
        fprintf(stderr, "EXTRA_JAVA_ARGS defined without JAVA_ARGS");
        abort();
    } else {
        jargc = main_jargc;
        jargv = (char **) const_jargs;
    }
```

`extra` 在 `main` 之前入列——这意味着 `const_jargs` 中的参数可以覆盖 `const_extra_jargs` 的默认值。

**margv 和 jargv 是完全独立的路径。** `java` 启动器本身没有编译期 JAVA_ARGS（`const_jargs` 为空），所以 `jargc = 0`。但 `javac`、`javap` 等 tools 有——编译期定义了 `-J-ms8m` 和 `-m module/class` 之类的东西。

这里很容易混淆：`-J` 前缀的参数是给 JVM 的（走 `jargv`），不带前缀的参数是给 Java 程序的（走 `margv`）。名字像，用途不同。

### 硬编码参数：CLion 调试的便利入口

jdk11u-copy 项目在 main.c 末尾加了一段调试代码：

```c
// main.c:225-241
    if (const_progname != NULL && JLI_StrCmp(const_progname, "java") == 0) {
        static char *hardcoded_argv[] = {
                "java",
                "-version",
                NULL,
        };
        margc = (int) (sizeof(hardcoded_argv) / sizeof(hardcoded_argv[0])) - 1;
        margv = hardcoded_argv;
        jargc = 0;
        jargv = NULL;
    }
```

`const_progname` 是 CMake 的编译期常量——每个 launcher（`java`、`javac`、`keytool` 等）在编译时定义了自己的 `PROGNAME`。只有 progname 匹配 `"java"` 时，硬编码参数才生效。所以 `javac` 不受影响。

这段代码让 CLion 调试时不需要配 Run Configuration 参数——直接点上三角就能跑 `java -version`。想跑一个类就改数组内容。注释里写明了"可整段删除还原"。

## JLI_Launch —— 五步流水线

`main()` 的最后一行把活交给了 `JLI_Launch()`。这个函数在 `src/java.base/share/native/libjli/java.c`，11 个参数：

```c
// java.c:221-233
JNIEXPORT int JNICALL
JLI_Launch(int argc, char ** argv,              /* 命令行参数 */
        int jargc, const char** jargv,          /* 编译期 Java 参数 */
        int appclassc, const char** appclassv,  /* app classpath */
        const char* fullversion,                /* 完整版本字符串 */
        const char* dotversion,                 /* 点分版本号（未使用） */
        const char* pname,                      /* 程序名 */
        const char* lname,                      /* 启动器名 */
        jboolean javaargs,                      /* 是否有 JAVA_ARGS */
        jboolean cpwildcard,                    /* classpath 通配符 */
        jboolean javaw,                         /* Windows javaw 模式 */
        jint ergo                               /* 未使用 */
)
```

12 个参数里 3 个标注了"未使用"——窗口期产物。函数内部是五步流水线：

**Step 1: 找环境。** `CreateExecutionEnvironment()` 确定三个路径：`jrepath`（JRE 根目录）、`jvmcfg`（JVM 配置文件）、`jvmpath`（libjvm.so 的绝对路径）。

**Step 2: 加载 JVM。** `LoadJavaVM(jvmpath, &ifn)`——dlopen 打开 libjvm.so，dlsym 取出 `JNI_CreateJavaVM` 函数指针。失败返回退出码 6。

**Step 3: 解析参数。** `ParseArguments(&argc, &argv, &mode, &what, &ret, jrepath)`——把 `-classpath`、`-jar`、`-version` 等启动器参数剥离，剩下的传给 Java 程序。`argv` 在此之后已经移位（`argv++` 跳过程序名）。

**Step 4: 设置伪属性。** 三条调用把命令行信息注入 JVM 参数：
```c
// java.c:335-341
    SetJavaCommandLineProp(what, argc, argv);    // -Dsun.java.command=
    SetJavaLauncherProp();                       // -Dsun.java.launcher=
    SetJavaLauncherPlatformProps();              // -Dsun.java.launcher.pid=
```

这不是用户设置的属性——是启动器"伪造"进去的，让 JVM 知道谁启动了它。

**Step 5: 启动 JVM。** `JVMInit(&ifn, threadStackSize, argc, argv, mode, what, ret)`——这是跨越边界的最后一步。返回的是 JVM 的退出码——如果 JVM 正常退出，这个值就是程序的 exit code。

## CreateExecutionEnvironment —— libjvm.so 在哪

启动器要加载 JVM，首先得知道 libjvm.so 放在哪。这不是硬编码的——JDK 可以在任何目录，`JAVA_HOME` 可能指向不同的安装。

`CreateExecutionEnvironment()` 在 `java_md_solinux.c:303-364`，做三件事：

### 找 JRE 根目录

函数开头调 `GetJREPath()`，用三层回退策略确定 jrepath：

1. **检查启动器所在目录**——从 `/proc/self/exe` 读 java 的真实路径，往回找 `jre/` 或 `../jre/`（JDK 9+ 已不再有单独的 jre 目录，但代码保留了兼容）
2. **检查 JAVA_HOME 环境变量**
3. **退到编译期路径**——configure 时记录的 `--prefix`

```c
// java_md_solinux.c:512-550 — GetJREPath 路径选择
    // 如果启动器路径下的 jre 目录不存在，退回到 JAVA_HOME
    if (JLI_StrLen(jrepath) == 0) {
        char *env_jhome = getenv("JAVA_HOME");
        if (env_jhome != NULL && JLI_StrLen(env_jhome) != 0) {
            JLI_Snprintf(jrepath, so_jrepath, "%s", env_jhome);
        }
    }
```

### 读 jvm.cfg 确定 VM 类型

有了 jrepath，就能打开 `lib/jvm.cfg`。这个配置文件指定了可用的 JVM 类型——通常只有一行 `-server KNOWN`：

```c
// jvm.cfg 典型内容
-server KNOWN
```

启动器根据这个文件和 `-client`/`-server` 命令行参数决定最终用哪个 VM 类型。如果没指定，默认 `server`。

### 拼接 libjvm.so 路径

`GetJVMPath()` 用 jrepath 和 VM 类型拼接完整路径：

```c
// java_md_solinux.c:486-507 — GetJVMPath
void
GetJVMPath(char *jrepath, char *jvmtype, char *jvmpath, jint so_jvmpath) {
    // 模板: <jrepath>/lib/<arch>/<jvmtype>/libjvm.so
    JLI_Snprintf(jvmpath, so_jvmpath, "%s/lib/%s/%s/libjvm.so",
                 jrepath, GetArch(), jvmtype);
}
```

实际运行中，`GetArch()` 返回 `amd64`，`jvmtype` 是 `server`。所以最终路径是：

```
/your/jdk/path/lib/server/libjvm.so
```

用 `strings` 验证——`libjli.so` 里确实嵌入了路径模板：

```
$ strings libjli.so | grep "libjvm"
libjvm.so
%s/libjvm.so
%s/lib/%s/libjvm.so
```

## InvocationFunctions —— 启动器和 JVM 的 ABI 契约

找到 libjvm.so 之后，启动器需要一个接口来和它对话。这个"接口"不是 C 虚函数表——而是三个 `dlsym` 出来的函数指针，封装在 `InvocationFunctions` 结构体中：

```c
// java.h:79-87
typedef jint (JNICALL *CreateJavaVM_t)(JavaVM **pvm, void **env, void *args);

typedef struct {
    CreateJavaVM_t CreateJavaVM;
    GetDefaultJavaVMInitArgs_t GetDefaultJavaVMInitArgs;
    GetCreatedJavaVMs_t GetCreatedJavaVMs;
} InvocationFunctions;
```

JNI 规范（JNI Invocation API Chapter 5）定义的最小接口——创建 VM、查询默认参数、获取已创建的 VM 列表。`HINSTANCE` 是 `void *`，存的是 `dlopen` 返回的句柄。

三个函数指针的初始化在 `LoadJavaVM()` 中——用 `dlsym` 按符号名查找。

---

## LoadJavaVM —— dlopen 打开 JVM 的大门

`LoadJavaVM()` 是启动器链最核心的函数——它把"死"的函数指针结构体变成"活"的：

```c
// java_md_solinux.c:552-633 — LoadJavaVM
jboolean
LoadJavaVM(const char *jvmpath, InvocationFunctions *ifn) {
    void *libjvm;

    // ...

    libjvm = dlopen(jvmpath, RTLD_NOW + RTLD_GLOBAL);
    if (libjvm == NULL) {
        JLI_ReportErrorMessage(DLL_ERROR1, __LINE__);
        JLI_ReportErrorMessage(DLL_ERROR2, jvmpath, dlerror());
        return JNI_FALSE;
    }

    ifn->CreateJavaVM = (CreateJavaVM_t)
        dlsym(libjvm, "JNI_CreateJavaVM");
    if (ifn->CreateJavaVM == NULL) {
        JLI_ReportErrorMessage(DLL_ERROR2, jvmpath, dlerror());
        return JNI_FALSE;
    }

    ifn->GetDefaultJavaVMInitArgs = (GetDefaultJavaVMInitArgs_t)
        dlsym(libjvm, "JNI_GetDefaultJavaVMInitArgs");
    if (ifn->GetDefaultJavaVMInitArgs == NULL) {
        JLI_ReportErrorMessage(DLL_ERROR2, jvmpath, dlerror());
        return JNI_FALSE;
    }

    ifn->GetCreatedJavaVMs = (GetCreatedJavaVMs_t)
        dlsym(libjvm, "JNI_GetCreatedJavaVMs");
    if (ifn->GetCreatedJavaVMs == NULL) {
        JLI_ReportErrorMessage(DLL_ERROR2, jvmpath, dlerror());
        return JNI_FALSE;
    }

    return JNI_TRUE;
}
```

几个关键细节：

**`RTLD_NOW + RTLD_GLOBAL`**——`RTLD_NOW` 要求立即解析所有符号引用（不是懒加载）。如果 libjvm.so 缺少任何一个依赖符号，`dlopen` 立刻失败，不会等到运行时才发现。`RTLD_GLOBAL` 让 libjvm.so 的符号对所有后续加载的 .so 可见——这很有用，因为 JVM 内部有多个共享库之间需要互相引用符号。

**三个 dlsym，一个一个检查**——不是批量赋值然后看结果。每个 `dlsym` 之后立即检查 NULL。因为 JNI 规范要求这三个符号都必须存在——如果缺任何一个，JVM 不可用。

**返回值语义**——返回 `JNI_FALSE`（非 0）表示失败，调用者返回退出码 6。但函数签名是 `jboolean`，不是 `jint`——真正的退出码由上层 `JLI_Launch` 控制。

## JVMInit —— 为什么要在新线程中启动 JVM

`LoadJavaVM` 返回后，`ifn->CreateJavaVM` 已经指向了 `JNI_CreateJavaVM`。但启动器不直接调用它——先调用 `JVMInit()`：

```c
// java_md_solinux.c:806-813
int
JVMInit(InvocationFunctions* ifn, jlong threadStackSize,
        int argc, char **argv, int mode, char *what, int ret) {
    ShowSplashScreen();
    return ContinueInNewThread(ifn, threadStackSize, argc, argv, mode, what, ret);
}
```

`ContinueInNewThread()` 先用 `pthread_create` 创建一个新线程，在新线程里执行 `JavaMain()`：

```c
// java_md_solinux.c:736-762 — 新线程创建
    rslt = ContinueInNewThread0(JavaMain, threadStackSize, (void*)&args);
    // ...
    if (rslt == 0)
        return 0;
```

**为什么要在新线程中？** 原始线程（调 main 的线程）需要继续管理信号和生命周期。更重要的是 JDK 6 早期的 bug（JDK-6316197）——线程栈大小和 JVM 的 Stack Guard Pages 机制冲突。原始线程的栈大小由操作系统初始分配决定，不能修改。新线程可以在 `pthread_create` 时指定 `threadStackSize`，让 JVM 有精确控制权。

`JVMInit` 不会返回——它内部调用 `exit()`。所以 `JLI_Launch` 的第 5 步写完，`main()` 也就结束了。

## 完整链路回顾

run 一次 `java -version`，从 `main()` 到 `return` 的完整路径：

```
main.c:97   main()
  ↓
main.c:242  JLI_Launch(margc, margv, jargc, jargv, ...)
  ↓
java.c:278  Step 1: CreateExecutionEnvironment() → jrepath, jvmpath
java.c:287  Step 2: if (!LoadJavaVM(jvmpath, &ifn)) return 6
            │  └→ java_md_solinux.c:559  dlopen(jvmpath, RTLD_NOW+RTLD_GLOBAL)
            │  └→ java_md_solinux.c:612  dlsym(libjvm, "JNI_CreateJavaVM")
            │  └→ java_md_solinux.c:618  dlsym(libjvm, "JNI_GetDefaultJavaVMInitArgs")
            │  └→ java_md_solinux.c:625  dlsym(libjvm, "JNI_GetCreatedJavaVMs")
java.c:325  Step 3: ParseArguments(&argc, &argv, &mode, &what, &ret, jrepath)
java.c:335  Step 4: SetJavaCommandLineProp() / SetJavaLauncherProp() / SetJavaLauncherPlatformProps()
java.c:343  Step 5: return JVMInit(&ifn, threadStackSize, argc, argv, mode, what, ret)
            │  └→ java_md_solinux.c:807  ShowSplashScreen()
            │  └→ java_md_solinux.c:808  ContinueInNewThread(...)
            │       └→ java_md_solinux.c:749  pthread_create(JavaMain, threadStackSize)
            │            └→ java.c:394  JavaMain()
            │                 └→ java.c:416  InitializeJVM(&vm, &env, &ifn)
            │                      └→ java.c:1529  ifn->CreateJavaVM(pvm, penv, &args)
            │                           └→ jni.cpp:4098  JNI_CreateJavaVM()
```

每个箭头都跨了一个 `.c` 文件或 `.so` 库。从 ELF 可执行文件到 `dlopen` 再到函数指针——启动器用 C 的动态加载能力，把"命令行"转换成了"JVM 入口函数调用"。
