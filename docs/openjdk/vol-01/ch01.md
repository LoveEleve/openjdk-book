# 第1章：Launcher Chain —— 从命令行到 libjvm.so

上一卷最后你亲手编译了一个 JDK。敲 `java -version` 的时候，从屏幕上看到三行输出，觉得挺自然——JDK 嘛，一个命令启动一个 JVM，很正常。

但如果你稍微想一下：这个 `java` 可执行文件才 23KB，而 JVM 本体 `libjvm.so` 有 164MB。一个 23KB 的 C 程序，怎么启动一个 164MB 的 Java 虚拟机？

本章就顺着这个好奇走下去——从 `main()` 开始，到 `dlopen("libjvm.so")` 为止。

---

## java 没链接 libjvm.so

先验证一个反直觉的事实。用 `ldd` 看 java 链接了哪些库：

```
$ ldd build/linux-x86_64-normal-server-slowdebug/jdk/bin/java
	libjli.so => .../jdk/bin/./../lib/jli/libjli.so
	libc.so.6 => /lib64/libc.so.6
	libz.so.1 => /lib64/libz.so.1
	/lib64/ld-linux-x86-64.so.2
```

三个依赖：`libc`（标准 C 库）、`libz`（压缩）、`libjli.so`。**没有 `libjvm.so`。**

你用 `readelf` 再看一遍 RUNPATH——java 只需要找到 `libjli.so`：

```
$ readelf -d build/linux-x86_64-normal-server-slowdebug/jdk/bin/java | grep RUNPATH
  (RUNPATH)            Library runpath: [$ORIGIN/../lib/jli]
```

`$ORIGIN` 是 ELF 动态链接器的特殊变量，展开为 java 二进制所在的目录。所以运行时找 `libjli.so` 就是 `java所在目录/../lib/jli/`。这个 RUNPATH 里只有 jli——没有 server、没有 libjvm、什么都没有。

`libjvm.so` 不是静态链接进来的，也不是动态链接器自动加载的。它是 java 启动器在运行中**手动**加载的。

这就是本章要讲的：启动器怎么找到 libjvm.so、怎么打开它、怎么从里面掏出 `JNI_CreateJavaVM` 这个函数。

---

## main() 在做什么——不是在"执行 Java"

java 的 `main()` 在 `src/java.base/share/native/launcher/main.c`。和 Ch0.1 那个 hello world 一样——标准 C 程序入口：

```c
// main.c:97
JNIEXPORT int
main(int argc, char **argv)
```

`JNIEXPORT` 是什么？你写的 hello world 的 `main()` 前面可没这个关键字。

它是一个宏，定义在 `jni_md.h` 里——不同平台展开成不同的东西。在 Linux x86-64 上：

```c
// jni_md.h
#define JNIEXPORT __attribute__((visibility("default")))
```

`__attribute__((visibility("default")))` 是 GCC 的编译指令——告诉链接器"这个符号对外可见"。对于可执行文件的 `main()` 来说这个没啥实际作用（main 本身就是默认可见的），但 OpenJDK 用它保持代码风格统一——所有 JNI 边界的函数都用 `JNIEXPORT` 修饰，一眼就能看出哪些函数是公开入口。Windows 上它展开成 `__declspec(dllexport)`，比 Linux 上的作用大得多。

另一个关键字 `JNICALL` 同理——不同平台展开成 `__stdcall`（Windows）或空（Linux）。它保证 JNI 函数的调用约定一致。

所以 `JNIEXPORT int JNICALL main(...)` 翻译过来就是：**"这个函数是公开入口，用标准 JNI 调用约定"**。它不是必须的——但它是 JNI 源码的一种身份标记。

但进去之后，你会发现它不解析 classpath、不创建虚拟机——它只是**准备参数**。参数分了两套：`margv` 和 `jargv`。

### margv：你敲的命令行

```c
// main.c:183
    int margc;
    char **margv;
```

`margv` 是"命令行参数"。比如你敲了这个：

```
java -Xmx256m -cp /my/app MyClass hello world
```

那么在 main 函数收到 `argc=6, argv={"java", "-Xmx256m", "-cp", "/my/app", "MyClass", "hello world"}` 之后，`margv` 构建过程会把它变成这样：

```
margv[0] = "java"
margv[1] = "-Xmx256m"        ← 环境变量 JDK_JAVA_OPTIONS 如果有值也会插在这里
margv[2] = "-cp"
margv[3] = "/my/app"
margv[4] = "MyClass"
margv[5] = "hello"
margv[6] = "world"
margv[7] = NULL               ← 哨兵
```

构建过程是这样的：

```c
// main.c:190
    JLI_List args = JLI_List_new(argc + 1);
    JLI_List_add(args, JLI_StringDup(argv[0]));    // 1. 程序名
```

先放进 `argv[0]`——也就是 `java` 这个程序名。

```c
// main.c:194
    if (JLI_AddArgsFromEnvVar(args, JDK_JAVA_OPTIONS)) {
```

然后注入 `JDK_JAVA_OPTIONS` 环境变量。注意它在 `argv[1]` 之前——所以如果你用命令行指定了同名参数，命令行上的会覆盖环境变量里的。环境变量在前，命令行在后，后入的盖先入的。

```c
// main.c:206
    for (i = 1; i < argc; i++) {
        JLI_List argsInFile = JLI_PreprocessArg(argv[i], JNI_TRUE);
```

最后逐个处理 `argv[1]`、`argv[2]`……`JLI_PreprocessArg` 负责展开 `@filename` 语法——如果你写 `java @opts.txt`，`opts.txt` 里的每一行会被当作一个参数展开插入。

```c
// main.c:216
    margc = args->size;
    margv = args->elements;
```

末尾插一个 NULL 哨兵，`margv` 构建完成。

### jargv：编译期写死的参数

```c
// main.c:100
    int jargc;
    char **jargv;
```

`jargv` 不走命令行。它来自编译时常量 `JAVA_ARGS` 和 `EXTRA_JAVA_ARGS`。

```c
// main.c:109
    main_jargc = (sizeof(const_jargs) / sizeof(char *)) > 1
                 ? sizeof(const_jargs) / sizeof(char *) : 0;
```

`sizeof` 在编译期就知道数组大小——这些数组定义在 `defines.h` 或 CMake 的 `target_compile_definitions` 里。`java` 启动器本身没有编译期 Java 参数（`const_jargs` 是空的），所以 `jargc = 0`。但如果把同样的代码编成 `javac`，`const_jargs` 里就有 `-J-ms8m`、`-m jdk.compiler/...` 这些东西了。

**margv 和 jargv 是完全独立的两条路径。** 混了就全错了。`margv` 是用户决定的（命令行 + 环境变量），`jargv` 是编译期写死的。它们最后分别传给 `JLI_Launch` 的不同参数位置。

### 硬编码参数：调试时不用配参数

Ch0.4 里你在 CLion 里跑过 `java -version`——右上角点了就出结果，不用配 Run Configuration。秘密在这里：

```c
// main.c:225
    if (const_progname != NULL && JLI_StrCmp(const_progname, "java") == 0) {
        static char *hardcoded_argv[] = {
                "java",
                "-version",
                NULL,
        };
        margc = (int) (sizeof(hardcoded_argv) / sizeof(hardcoded_argv[0])) - 1;
        margv = hardcoded_argv;
        jargc = 0;
        jargv = NULL;
    }
```

`const_progname` 是 CMake 定义的编译期常量——编 `java` 启动器时它等于 `"java"`，编 `javac` 时等于 `"javac"`。所以只有 `java` 启动器会走进这个 `if`，`javac` 不会。你在数组里写什么，就等于敲了什么命令行。

这整段代码只有一个目的：CLion 里直接点三角号就能调试，不用每次去 Run Configuration 里敲 `-version`。它是一个开发便利，和 Java 本身毫无关系。

## JLI_Launch —— 把 args 交给启动器库

`main()` 最后一行：

```c
// main.c:242
    return JLI_Launch(margc, margv,
                   jargc, (const char**) jargv,
                   0, NULL,
                   VERSION_STRING, DOT_VERSION,
                   (const_progname != NULL) ? const_progname : *margv,
                   (const_launcher != NULL) ? const_launcher : *margv,
                   jargc > 0, const_cpwildcard, const_javaw, 0);
```

11 个参数，但很多你一看就懂：`VERSION_STRING` 是 `"11.0.32-internal"`，`DOT_VERSION` 是 `"11.0"`，`const_progname` 是 `"java"`，`jargc > 0` 告诉 JLI_Launch "这个启动器有没有编译期 Java 参数"。

`JLI_Launch` 本身在 `src/java.base/share/native/libjli/java.c`——它是 libjli 库的函数。从 `main.c` 到 `java.c`，你跨了第一个 `.c` 文件。

它的内部结构就是五步：

```c
// java.c:278 — 第一步：找环境
    CreateExecutionEnvironment(&argc, &argv, jrepath, sizeof(jrepath),
                               jvmpath, sizeof(jvmpath), jvmcfg, sizeof(jvmcfg));
```

三个输出变量：`jrepath`（JDK 根目录）、`jvmcfg`（VM 类型配置）、`jvmpath`（libjvm.so 的绝对路径）。这是下一节的内容。

```c
// java.c:287 — 第二步：加载 JVM
    ifn.CreateJavaVM = 0;
    ifn.GetDefaultJavaVMInitArgs = 0;

    if (!LoadJavaVM(jvmpath, &ifn)) {
        return(6);
    }
```

先清零，再加载。`LoadJavaVM` 如果失败，直接 `return 6`——退出码 6 就是"JVM 加载失败"。阶段越早退出码越小：2 是 JRE 路径无效、4 是 VM 类型无效、6 是 dlopen 失败。

```c
// java.c:325 — 第三步：解析参数
    if (!ParseArguments(&argc, &argv, &mode, &what, &ret, jrepath)) {
        return(ret);
    }
```

`ParseArguments` 把 `-classpath`、`-jar`、`-version` 这些启动器自己的参数剥离出来。剩下来的 `argv` 就是传给 Java 程序的——现在如果用户敲的是 `java MyClass arg1 arg2`，`argv` 里就是 `MyClass`、`arg1`、`arg2`。

```c
// java.c:335 — 第四步：注入"伪属性"
    SetJavaCommandLineProp(what, argc, argv);
    SetJavaLauncherProp();
    SetJavaLauncherPlatformProps();
```

这三行往 JVM 参数里塞了三个系统属性：`sun.java.command`（用户敲了啥）、`sun.java.launcher`（走 SUN_STANDARD 模式）、`sun.java.launcher.pid`（启动器进程的 PID）。注意——这**不是**用户通过 `-D` 指定的属性。这是启动器"伪造"的，所以叫伪属性。JVM 用它们来知道是谁启动了它。

```c
// java.c:343 — 第五步：启动 JVM
    return JVMInit(&ifn, threadStackSize, argc, argv, mode, what, ret);
```

`JVMInit` 不会返回——它内部会 `exit()`。所以 `JLI_Launch` 的第五步写出来，接下来就是另一个世界了。

## CreateExecutionEnvironment —— libjvm.so 到底在哪

`JLI_Launch` 的第一步是找到 `libjvm.so` 的路径。这不能写死——JDK 可以装在任意目录，`JAVA_HOME` 可以指向任何地方。

`CreateExecutionEnvironment` 在 `src/java.base/unix/native/libjli/java_md_solinux.c:303`，逻辑分三步。

### 第一步：确定 JRE 根目录

```c
// java_md_solinux.c:315
    if (!GetJREPath(jrepath, so_jrepath, GetArch(), JNI_FALSE)) {
        JLI_ReportErrorMessage(CFG_ERROR1);
        exit(1);
    }
```

`GetJREPath` 怎么找到 JDK 根目录？它从 `/proc/self/exe` 读 java 可执行文件的实际路径：

```c
// java_md_solinux.c:512
    char *env_jhome = getenv("JAVA_HOME");
    char *exe_path = getenv("_");
    // 如果启动器路径下的 ../jre 存在，用它
    // 否则：检查 JAVA_HOME 环境变量
    // 再否则：退回到编译期硬编码的路径
```

`/proc/self/exe` 是 Linux 的 `/proc` 文件系统提供的——它总是指向当前进程的可执行文件，无论你的当前目录在哪。所以即使用户在 `/tmp` 下敲 `/opt/jdk/bin/java`，启动器也知道自己实际在 `/opt/jdk/bin/`。

### 第二步：读 jvm.cfg 确定 VM 类型

有了 JRE 根目录，就能打开 `lib/jvm.cfg`：

```c
// java_md_solinux.c:335
    if (ReadKnownVMs(jvmcfg, JNI_FALSE) < 1) {
        JLI_ReportErrorMessage(CFG_ERROR7);
        exit(1);
    }
```

这个文件通常就一行：`-server KNOWN`。`KNOWN` 表示这个 VM 类型存在；如果写的是 `-server IGNORE`，启动器就跳过它。从 jvm.cfg 可以知道用户有没有指定 `-client`（现在基本没有了，JDK 11 连 client VM 都不编译）。

### 第三步：拼接 jvmpath

现在有了 JRE 根目录和 VM 类型，`GetJVMPath` 拼接出完整路径：

```c
// java_md_solinux.c:486
    JLI_Snprintf(jvmpath, so_jvmpath, "%s/lib/%s/%s/libjvm.so",
                 jrepath, GetArch(), jvmtype);
```

用 `strings` 验证——`libjli.so` 里确实嵌着这个路径模板：

```
$ strings build/.../lib/jli/libjli.so | grep libjvm
libjvm.so
%s/libjvm.so
%s/lib/%s/libjvm.so
```

`GetArch()` 在 x86-64 上返回 `amd64`，`jvmtype` 是 `server`。所以最终路径就是 `<JDK>/lib/amd64/server/libjvm.so`——或者简写 `<JDK>/lib/server/libjvm.so`（实际目录结构用的是简写）。

## InvocationFunctions —— 三个函数指针的接口

找到 libjvm.so 的路径，启动器需要一个方法去"调用"它。这不是 C++ 的虚函数表——而是一个纯 C 的结构体，装三个函数指针：

```c
// java.h:79
typedef jint (JNICALL *CreateJavaVM_t)(JavaVM **pvm, void **env, void *args);

typedef struct {
    CreateJavaVM_t CreateJavaVM;
    GetDefaultJavaVMInitArgs_t GetDefaultJavaVMInitArgs;
    GetCreatedJavaVMs_t GetCreatedJavaVMs;
} InvocationFunctions;
```

`CreateJavaVM` 是你马上要见到的核心——从启动器跨进 JVM 的唯一入口。`GetDefaultJavaVMInitArgs` 查询默认 VM 参数。`GetCreatedJavaVMs` 查询当前进程里已经有多少个 JVM 实例。

这三个函数不是静态链接的——它们是空的，结构体里全是 0。下一节讲怎么"填"它们。

这个结构体就是 JNI 规范第 5 章"Invocation API"的最小接口。HotSpot 只需要实现这三个函数，任何启动器都可以用这套接口来创建和查询 JVM。

---

## LoadJavaVM —— dlopen + dlsym 的灵魂三连

`LoadJavaVM` 在 `java_md_solinux.c:552`。它是本章最核心的函数——把三个空函数指针变成真正可调用的执行入口。

```c
// java_md_solinux.c:553
jboolean
LoadJavaVM(const char *jvmpath, InvocationFunctions *ifn) {
```

参数就两个：`jvmpath`（`CreateExecutionEnvironment` 拼出来的 libjvm.so 路径）、`ifn`（那个空的结构体指针）。返回 `JNI_TRUE` 表示全成功了，返回 `JNI_FALSE` 表示某一步失败了。

### dlopen：打开 libjvm.so

```c
// java_md_solinux.c:559
    libjvm = dlopen(jvmpath, RTLD_NOW + RTLD_GLOBAL);
```

`dlopen` 是 Linux 动态链接器的 API——它的作用和 `ldd` 看到的"自动加载"完全相反：自动加载发生在程序启动时，`dlopen` 发生在运行时。你可以在程序的任意时刻打开一个 .so 并用里面的函数。

两个 flag：`RTLD_NOW` 要求立即解析 libjvm.so 的所有符号引用——如果缺少任何依赖符号，`dlopen` 立刻失败，不会等到调用某个 JVM 函数时才崩溃。`RTLD_GLOBAL` 让 libjvm.so 的符号对后续用 `dlopen` 加载的其他 .so 可见——JVM 内部有多个 .so 互相引用，这个 flag 必须加上。

```c
// java_md_solinux.c:560
    if (libjvm == NULL) {
        JLI_ReportErrorMessage(DLL_ERROR1, __LINE__);
        JLI_ReportErrorMessage(DLL_ERROR2, jvmpath, dlerror());
        return JNI_FALSE;
    }
```

`dlopen` 失败最常见的三个原因：路径不存在（`jvmpath` 拼错了）、64/32 bit 不匹配（32 位程序试图加载 64 位 .so）、依赖库缺失（libjvm.so 依赖的某个 .so 没装）。`dlerror()` 会告诉你具体是哪个。

### 三个 dlsym：掏出函数指针

```c
// java_md_solinux.c:612
    ifn->CreateJavaVM = (CreateJavaVM_t)
        dlsym(libjvm, "JNI_CreateJavaVM");
```

`dlsym` 的第二个参数是**字符串**——`"JNI_CreateJavaVM"`。没错，不是符号表枚举、不是宏——就是一个字符串。你在 libjvm.so 里找到这个符号，把它的地址转成 `CreateJavaVM_t` 函数指针，赋值给 `ifn->CreateJavaVM`。

调用成功之后，`ifn->CreateJavaVM` 不再指向 NULL——它指向 `jni.cpp` 里的 `JNI_CreateJavaVM` 函数。下一章就要走到这个函数里面去。

```c
// java_md_solinux.c:618
    ifn->GetDefaultJavaVMInitArgs = (GetDefaultJavaVMInitArgs_t)
        dlsym(libjvm, "JNI_GetDefaultJavaVMInitArgs");
```

```c
// java_md_solinux.c:625
    ifn->GetCreatedJavaVMs = (GetCreatedJavaVMs_t)
        dlsym(libjvm, "JNI_GetCreatedJavaVMs");
```

同样的模式，再来两次。每一个 `dlsym` 之后都是 NULL 检查——缺少任何一个符号，JVM 就不完整，必须失败。

三个函数都成功后，返回 `JNI_TRUE`。此时 `ifn` 不再是一个空壳——它的三个字段都指向了 libjvm.so 里真正可调用的函数。

## 为什么要在新线程中启动 JVM

`LoadJavaVM` 返回后，启动器手里有了 `ifn->CreateJavaVM`，理论上可以直接调用。但它没有——先调了 `JVMInit`：

```c
// java.c:343
    return JVMInit(&ifn, threadStackSize, argc, argv, mode, what, ret);
```

`JVMInit` 在 `java_md_solinux.c:806`，就两行：

```c
// java_md_solinux.c:807
    ShowSplashScreen();
    return ContinueInNewThread(ifn, threadStackSize, argc, argv, mode, what, ret);
```

第一行画启动画面（如果有 -splash 参数）。第二行 `ContinueInNewThread` 内部调 `pthread_create`——创建一个全新的线程，在新线程里执行 `JavaMain()`。

为什么要开新线程？原始线程（调 `main()` 的那个线程）有几个不可以改的属性：栈大小由操作系统初始分配决定、信号处理器已经绑定、栈空间不可扩展。JVM 需要精确控制这些事情——特别是 Java 线程的栈 guard pages（防止栈溢出覆盖其他内存），必须在 `pthread_create` 时就能设定。

JDK 6 早期有个 bug（JDK-6316197）就是因为直接在主线程启动 JVM，导致 JVM 的 StackGuardPages 和操作系统的默认栈大小冲突。修复方案就是永远在新线程中启动。

`JVMInit` 不会返回——它内部最终会调用 `exit()`。所以 `main()` 的 `return JLI_Launch(...)` 实际上也不会执行到。整个进程的生命周期从这一行开始，交给了新线程里的 JVM。

## 完整链路

从你敲 `java -version` 到 `dlopen` 完成，代码走过的完整路径：

```
main.c:97      main()
                │
                ├─ 构建 margv（命令行 + JDK_JAVA_OPTIONS 注入）
                ├─ 构建 jargv（编译期 JAVA_ARGS，java 启动器为空）
                └─ main.c:242  JLI_Launch(margc, margv, jargc, jargv, ...)
                                   │
  java.c:278                       ├─ Step 1: CreateExecutionEnvironment()
                                   │           └─ java_md_solinux.c:303
                                   │                ├─ 从 /proc/self/exe 找 JRE 根目录
                                   │                ├─ 读 lib/jvm.cfg 确定 VM 类型
                                   │                └─ 拼接 jvmpath = <JDK>/lib/server/libjvm.so
                                   │
  java.c:287                       ├─ Step 2: LoadJavaVM(jvmpath, &ifn)
                                   │           └─ java_md_solinux.c:552
                                   │                ├─ dlopen(jvmpath, RTLD_NOW+RTLD_GLOBAL)
                                   │                ├─ dlsym(libjvm, "JNI_CreateJavaVM")
                                   │                ├─ dlsym(libjvm, "JNI_GetDefaultJavaVMInitArgs")
                                   │                └─ dlsym(libjvm, "JNI_GetCreatedJavaVMs")
                                   │
  java.c:325                       ├─ Step 3: ParseArguments（剥离启动器参数）
  java.c:335                       ├─ Step 4: SetJavaCommandLineProp() 等（伪属性注入）
                                   │
  java.c:343                       └─ Step 5: JVMInit()
                                               └─ java_md_solinux.c:806
                                                    └─ ContinueInNewThread() → pthread_create
                                                         └─ JavaMain() → InitializeJVM()
                                                              └─ ifn->CreateJavaVM(pvm, penv, &args)
```
