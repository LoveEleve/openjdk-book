# 第1章：Launcher Chain —— 从命令行到 libjvm.so

上一卷最后你亲手编译了一个 JDK。敲 `java -version` 的时候，从屏幕上看到三行输出，觉得挺自然——JDK 嘛，一个命令启动一个 JVM，很正常。

但如果你稍微想一下：这个 `java` 可执行文件才 23KB，而 JVM 本体 `libjvm.so` 有 164MB。一个 23KB 的 C 程序，怎么启动一个 164MB 的 Java 虚拟机？

本章就顺着这个好奇走下去——从 `main()` 开始，到 `dlopen("libjvm.so")` 为止。

---

## java 没链接 libjvm.so

先验证一个反直觉的事实。用 `ldd` 看 java 链接了哪些库：

```
$ ldd build/linux-x86_64-normal-server-slowdebug/jdk/bin/java
	libjli.so => .../jdk/bin/./../lib/jli/libjli.so
	libc.so.6 => /lib64/libc.so.6
	libz.so.1 => /lib64/libz.so.1
	/lib64/ld-linux-x86-64.so.2
```

三个依赖：`libc`（标准 C 库）、`libz`（压缩）、`libjli.so`。没有 `libjvm.so`。

你用 `readelf` 再看一遍 RUNPATH——java 只需要找到 `libjli.so`：

```
$ readelf -d build/linux-x86_64-normal-server-slowdebug/jdk/bin/java | grep RUNPATH
  (RUNPATH)            Library runpath: [$ORIGIN/../lib/jli]
```

`$ORIGIN` 是 ELF 动态链接器的特殊变量，展开为 java 二进制所在的目录。所以运行时找 `libjli.so` 就是 `java所在目录/../lib/jli/`。这个 RUNPATH 里只有 jli——没有 server、没有 libjvm、什么都没有。

`libjvm.so` 不是静态链接进来的，也不是动态链接器自动加载的。它是 java 启动器在运行中**手动**加载的。

这就是本章要讲的：启动器怎么找到 libjvm.so、怎么打开它、怎么从里面掏出 `JNI_CreateJavaVM` 这个函数。

---

## main() 在做什么——不是在"执行 Java"

java 的 `main()` 在 `src/java.base/share/native/launcher/main.c`。和 Ch0.1 那个 hello world 一样——标准 C 程序入口。但它不解析 classpath、不创建虚拟机——它只是**准备参数**。

先看一眼整个 `main()` 的骨架，把你要关注的部分标出来：

```c
// main.c:97  — 函数入口
JNIEXPORT int
main(int argc, char **argv) {
    int margc, jargc;
    char **margv, **jargv;

    // ◇ 阶段A：构建 jargv（编译期参数）
    main_jargc = (sizeof(const_jargs) / sizeof(char *)) > 1
                 ? sizeof(const_jargs) / sizeof(char *) : 0;
    // ... 组合 EXTRA_JAVA_ARGS + JAVA_ARGS → jargv

    JLI_InitArgProcessing(jargc > 0, const_disable_argfile);

    // ◇ 阶段B：Windows 参数处理（本书跳过）
    #ifdef _WIN32
    // UTF-16 → ANSI 编码转换 + CmdToArgs，不展开
    #else

    // ◇ 阶段C★：构建 margv（Linux x86_64）
    JLI_List args = JLI_List_new(argc + 1);
    JLI_List_add(args, JLI_StringDup(argv[0]));       // 程序名
    JLI_AddArgsFromEnvVar(args, JDK_JAVA_OPTIONS);    // 环境变量注入
    for (i = 1; i < argc; i++) {                      // @argfile 展开
        JLI_List argsInFile = JLI_PreprocessArg(argv[i], JNI_TRUE);
        // ...
    }
    margc = args->size;
    margv = args->elements;

    #endif

    // ◇ 阶段D★：硬编码参数覆盖（jdk11u-copy 定制）
    if (const_progname != NULL && JLI_StrCmp(const_progname, "java") == 0) {
        static char *hardcoded_argv[] = { "java", "-version", NULL };
        margc = ...; margv = hardcoded_argv; jargc = 0; jargv = NULL;
    }

    // ◇ 阶段E：交接给 libjli
    return JLI_Launch(margc, margv, jargc, (const char**) jargv, ...);
}
```

下面按代码的实际执行顺序走——不是骨架标注的顺序，而是 CPU 跑到的顺序。前面几步跑在 `#ifdef _WIN32` 之前，和平台无关。

### JNIEXPORT / JNICALL —— JNI 边界的标记

入口那行有个你写的 hello world 没有的关键字：

```c
// main.c:97
JNIEXPORT int
main(int argc, char **argv)
```

`JNIEXPORT` 是宏，在 `src/hotspot/os/linux/jni_md.h:34` 展开为：

```c
#define JNIEXPORT __attribute__((visibility("default")))
```

`__attribute__((visibility("default")))` 是 GCC 编译指令——"这个符号对外可见"。对可执行文件的 `main()` 没啥实际作用，但 OpenJDK 用它保持 JNI 边界代码的风格统一。另一个 `JNICALL` 同理——Linux 上空，Windows 上 `__stdcall`。

入口之后，声明三板斧：`int margc, jargc`、`char **margv, **jargv`、`const_javaw = JNI_FALSE`。然后进入第一个实际逻辑。

### 第一步（真实顺序）：构建 jargv —— 编译期写死的参数

`jargv` 不走命令行。它来自编译时常量 `JAVA_ARGS` 和 `EXTRA_JAVA_ARGS`：

```c
// main.c:105-139
    main_jargc = (sizeof(const_jargs) / sizeof(char *)) > 1
                 ? sizeof(const_jargs) / sizeof(char *) : 0;

    extra_jargc = (sizeof(const_extra_jargs) / sizeof(char *)) > 1
                  ? sizeof(const_extra_jargs) / sizeof(char *) : 0;
```

`sizeof(const_jargs)` 除以 `sizeof(char *)`——数组总字节数除以每个元素的字节数，得到元素个数。`> 1` 的检查是因为数组至少有两个元素：至少一个内容加上末尾的 NULL。如果只有一个元素（只有一个 NULL），说明这个启动器没有编译期 Java 参数。

`java` 启动器的 `const_jargs` 定义为：

```c
// defines.h (CMake 生成)
static const char* const_jargs[] = { NULL };
```

所以 `main_jargc = 0`，`jargv = NULL`。

但编 `javac` 时，CMake 传入了 `-DJAVA_ARGS='{ "-J-ms8m", "-m", "jdk.compiler/..." }'`：

```c
// javac 的 defines.h
static const char* const_jargs[] = { "-J-ms8m", "-m", "jdk.compiler/...", NULL };
```

`-J` 前缀的意思是"这个参数给 JVM，不给 Java 程序"。所以 `jargv` 里的东西最终变成 JVM 的启动参数，而 `margv` 里不带 `-J` 的才是传给 `main(String[] args)` 的。

如果 `extra_jargc` 也有值，两个数组合并——extra 在先、main 在后，后者覆盖前者。

### 第二步：InitArgProcessing —— 参数预处理的开关

```c
// main.c:141
    JLI_InitArgProcessing(jargc > 0, const_disable_argfile);
```

这个函数设置两个全局标志。第一个 `jargc > 0` = "有没有编译期 Java 参数"，决定了后续是否处理 `-J` 前缀。第二个 `const_disable_argfile` = jdk11u-copy 没设，所以 `@opts.txt` 展开正常支持。

### 第三步（平台分支）：Linux x86_64 路径

```c
// main.c:183-222
#ifdef _WIN32
    // UTF-16 → ANSI 编码转换，本书跳过
#else
    // Linux x86_64 走这里
```

`#ifdef _WIN32` 后面的代码是 Windows 专用的——UTF-16 命令行到 ANSI 的转换、`GetCommandLineW`、`WideCharToMultiByte`。本书不展开，直接看 `#else` 分支。

`margv` 构建用了一个自定义的数据结构，先把它看清楚。

**JLI_List**——不是 C 标准库的东西，是启动器在 `jli_util.h` 里手搓的动态数组：

```c
typedef struct {
    int       size;      // 当前元素数
    int       capacity;  // 总容量
    char    **elements;  // 字符串数组
} JLI_List;
```

> **`char **elements` 为什么是"字符串数组"？**
>
> C 里没有 `string` 类型。一个字符串是 `char *`——指向一串连续 `char` 的首地址。多个字符串放到一起，就是一个 `char *` 的数组，内存里是"指针的指针"，也就是 `char **`。
>
> 比如 `margv` 存了 `"java"`、`"-version"`、`NULL` 三个字符串：
>
> ```
> elements → [0] → "java\0"
>             [1] → "-version\0"
>             [2] → NULL
> ```
>
> `elements[0]` 是 `char *`，指向 `"java"`；`elements[1]` 指向 `"-version"`。末尾的 `elements[2]` 是 `NULL`（空指针），不指向任何字符串——它只标记数组终点。
>
> C 里所有参数传递（`main` 的 `argv`、`JLI_Launch` 的 `margv`）都用这个模式：二级指针 + NULL 哨兵。

三个操作：`JLI_List_new(capacity)` 分配、`JLI_List_add(list, str)` 追加（满了自动翻倍扩容）、`JLI_List_add(list, NULL)` 加哨兵。`JLI_StringDup` 内部调 `malloc + memcpy`——因为 `argv` 是操作系统分配的内存，启动器不拥有它，复制一份才能随意管理生命周期。

回到 margv 构建：

```c
    JLI_List args = JLI_List_new(argc + 1);
    JLI_List_add(args, JLI_StringDup(argv[0]));       // 程序名 "java"
```

先放进 `argv[0]`——也就是 `java` 这个程序名。

```c
    if (JLI_AddArgsFromEnvVar(args, JDK_JAVA_OPTIONS)) {
```

注入 `JDK_JAVA_OPTIONS` 环境变量。内部是 `getenv` + 空格分割 + 逐个 `JLI_List_add`。环境变量没设就什么都不加。注意它**在 `argv[1]` 之前**——所以命令行上的同名参数会覆盖环境变量里的。

举例：设了 `export JDK_JAVA_OPTIONS="-Xmx128m"`，然后敲 `java -Xmx256m -version`。margv 顺序：`"java"` → `"-Xmx128m"`（环境变量）→ `"-Xmx256m"`（命令行）→ `"-version"`。JVM 取最后一个 `-Xmx`——命令行 256m 生效，环境变量 128m 被覆盖。

```c
    for (i = 1; i < argc; i++) {
        JLI_List argsInFile = JLI_PreprocessArg(argv[i], JNI_TRUE);
```

逐个处理 `argv[1]`、`argv[2]`……`JLI_PreprocessArg` 负责 `@filename` 展开。如果你写 `java @opts.txt`，`opts.txt` 的内容（比如三行 `-Xmx256m`、`-cp`、`/my/app`）会被拆成三个参数插入。嵌套展开也支持——`opts.txt` 里再写 `@more.txt` 也会递归展开。

```c
    margc = args->size;
    margv = args->elements;
```

收尾：NULL 哨兵、返回数组指针。margv 构建完成。

### 第四步：硬编码参数覆盖 —— CLion 调试不用配参数

```c
// main.c:225-241
    if (const_progname != NULL && JLI_StrCmp(const_progname, "java") == 0) {
        static char *hardcoded_argv[] = { "java", "-version", NULL };
        margc = (int) (sizeof(hardcoded_argv) / sizeof(hardcoded_argv[0])) - 1;
        margv = hardcoded_argv;
        jargc = 0;
        jargv = NULL;
    }
```

`const_progname` 是 CMake 编译期常量——编 `java` 启动器时等于 `"java"`，编 `javac` 时等于 `"javac"`。所以只有 `java` 会走进这个 `if`。`JLI_StrCmp` 是启动器自己的字符串比较，等价于 `strcmp`。

这整段代码就是 CLion 里点三角号就能跑的秘密——不用去 Run Configuration 配参数。它覆盖了前面所有构建出来的 `margv` 和 `jargv`。

### 第五步：交接给 libjli

```c
// main.c:242-251
    return JLI_Launch(margc, margv,
                   jargc, (const char**) jargv,
                   0, NULL,
                   VERSION_STRING, DOT_VERSION,
                   (const_progname != NULL) ? const_progname : *margv,
                   (const_launcher != NULL) ? const_launcher : *margv,
                   jargc > 0, const_cpwildcard, const_javaw, 0);
```

11 个参数逐一解释：

| 参数 | 值 | 含义 |
|------|-----|------|
| `margc, margv` | 命令行参数 | 你敲的东西 |
| `jargc, jargv` | 编译期参数 | java 启动器为空 |
| `0, NULL` | appclassc/appclassv | applet 用的，JDK 11 已废弃 |
| `VERSION_STRING` | `"11.0.32-internal"` | 完整版本号 |
| `DOT_VERSION` | `"11.0"` | 点分主版本 |
| `const_progname` | `"java"` | 程序名 |
| `const_launcher` | `"openjdk"` | 启动器名 |
| `jargc > 0` | `JNI_FALSE` | 有没有编译期 Java 参数 |
| `const_cpwildcard` | `JNI_TRUE` | classpath 支持 `*` 通配符 |
| `const_javaw` | `JNI_FALSE` | Windows javax 模式，Linux 始终 false |
| 最后一个 `0` | 0 | 保留参数，源码标注 UNUSED |

`JLI_Launch` 在 `src/java.base/share/native/libjli/java.c`——从 `main.c` 跨到 libjli 库的第一个函数。

## JLI_Launch 内部做了什么

前面你看到 `JLI_Launch` 被调用。现在进入它的内部——一段五步流水线。每一步都要详细走。

它的内部结构就是五步：

```c
// java.c:278 — 第一步：找环境
    CreateExecutionEnvironment(&argc, &argv, jrepath, sizeof(jrepath),
                               jvmpath, sizeof(jvmpath), jvmcfg, sizeof(jvmcfg));
```

三个输出变量：`jrepath`（JDK 根目录）、`jvmcfg`（VM 类型配置）、`jvmpath`（libjvm.so 的绝对路径）。这是下一节的内容。

```c
// java.c:287 — 第二步：加载 JVM
    ifn.CreateJavaVM = 0;
    ifn.GetDefaultJavaVMInitArgs = 0;

    if (!LoadJavaVM(jvmpath, &ifn)) {
        return(6);
    }
```

先清零，再加载。`LoadJavaVM` 如果失败，直接 `return 6`——退出码 6 就是"JVM 加载失败"。阶段越早退出码越小：2 是 JRE 路径无效、4 是 VM 类型无效、6 是 dlopen 失败。

```c
// java.c:325 — 第三步：解析参数
    if (!ParseArguments(&argc, &argv, &mode, &what, &ret, jrepath)) {
        return(ret);
    }
```

`ParseArguments` 把 `-classpath`、`-jar`、`-version` 这些启动器自己的参数剥离出来。剩下来的 `argv` 就是传给 Java 程序的——现在如果用户敲的是 `java MyClass arg1 arg2`，`argv` 里就是 `MyClass`、`arg1`、`arg2`。

```c
// java.c:335 — 第四步：注入"伪属性"
    SetJavaCommandLineProp(what, argc, argv);
    SetJavaLauncherProp();
    SetJavaLauncherPlatformProps();
```

这三行往 JVM 参数里塞了三个系统属性：`sun.java.command`（用户敲了啥）、`sun.java.launcher`（走 SUN_STANDARD 模式）、`sun.java.launcher.pid`（启动器进程的 PID）。注意——这**不是**用户通过 `-D` 指定的属性。这是启动器"伪造"的，所以叫伪属性。JVM 用它们来知道是谁启动了它。

```c
// java.c:343 — 第五步：启动 JVM
    return JVMInit(&ifn, threadStackSize, argc, argv, mode, what, ret);
```

`JVMInit` 不会返回——它内部会 `exit()`。所以 `JLI_Launch` 的第五步写出来，接下来就是另一个世界了。

## CreateExecutionEnvironment —— libjvm.so 到底在哪

`JLI_Launch` 的第一步是找到 `libjvm.so` 的路径。这不能写死——JDK 可以装在任意目录，`JAVA_HOME` 可以指向任何地方。

`CreateExecutionEnvironment` 在 `src/java.base/unix/native/libjli/java_md_solinux.c:303`，逻辑分三步。

### 第一步：确定 JRE 根目录

```c
// java_md_solinux.c:315
    if (!GetJREPath(jrepath, so_jrepath, GetArch(), JNI_FALSE)) {
        JLI_ReportErrorMessage(CFG_ERROR1);
        exit(1);
    }
```

`GetJREPath` 怎么找到 JDK 根目录？它从 `/proc/self/exe` 读 java 可执行文件的实际路径：

```c
// java_md_solinux.c:512
    char *env_jhome = getenv("JAVA_HOME");
    char *exe_path = getenv("_");
    // 如果启动器路径下的 ../jre 存在，用它
    // 否则：检查 JAVA_HOME 环境变量
    // 再否则：退回到编译期硬编码的路径
```

`/proc/self/exe` 是 Linux 的 `/proc` 文件系统提供的——它总是指向当前进程的可执行文件，无论你的当前目录在哪。所以即使用户在 `/tmp` 下敲 `/opt/jdk/bin/java`，启动器也知道自己实际在 `/opt/jdk/bin/`。

### 第二步：读 jvm.cfg 确定 VM 类型

有了 JRE 根目录，就能打开 `lib/jvm.cfg`：

```c
// java_md_solinux.c:335
    if (ReadKnownVMs(jvmcfg, JNI_FALSE) < 1) {
        JLI_ReportErrorMessage(CFG_ERROR7);
        exit(1);
    }
```

这个文件通常就一行：`-server KNOWN`。`KNOWN` 表示这个 VM 类型存在；如果写的是 `-server IGNORE`，启动器就跳过它。从 jvm.cfg 可以知道用户有没有指定 `-client`（现在基本没有了，JDK 11 连 client VM 都不编译）。

### 第三步：拼接 jvmpath

现在有了 JRE 根目录和 VM 类型，`GetJVMPath` 拼接出完整路径：

```c
// java_md_solinux.c:486
    JLI_Snprintf(jvmpath, so_jvmpath, "%s/lib/%s/%s/libjvm.so",
                 jrepath, GetArch(), jvmtype);
```

三个占位符依次是 JRE 根目录、CPU 架构名、VM 类型。`GetArch()` 不是 Linux 系统调用——它是启动器自己实现的，编译期根据 `#ifdef __x86_64__` 等宏返回 `"amd64"`（不是 `"x86_64"`）。在你编译出来的 JDK 里，执行结果是：

```
/data/workspace/jdk11u-copy/build/linux-x86_64-normal-server-slowdebug/jdk/lib/amd64/server/libjvm.so
```

但实际目录结构里 `jdk/lib/server/libjvm.so` 就够了，没有 `amd64/server` 这个子目录。`GetArch()` 返回 `amd64`，但 JDK 9+ 把 `lib/<arch>/` 简化成了 `lib/`，代码里保留 `%s/lib/%s/` 是为了兼容老版本和 Solaris 这些架构目录差异大的平台。

用 `strings` 验证——`libjli.so` 里确实嵌着这个路径模板：

```
$ strings build/.../lib/jli/libjli.so | grep libjvm
libjvm.so
%s/libjvm.so
%s/lib/%s/libjvm.so
```

`GetArch()` 在 x86-64 上返回 `amd64`，`jvmtype` 是 `server`。所以最终路径就是 `<JDK>/lib/amd64/server/libjvm.so`——或者简写 `<JDK>/lib/server/libjvm.so`（实际目录结构用的是简写）。

## InvocationFunctions —— 三个函数指针的接口

找到 libjvm.so 的路径，启动器需要一个方法去"调用"它。这不是 C++ 的虚函数表——而是一个纯 C 的结构体，装三个函数指针：

```c
// java.h:79
typedef jint (JNICALL *CreateJavaVM_t)(JavaVM **pvm, void **env, void *args);

typedef struct {
    CreateJavaVM_t CreateJavaVM;
    GetDefaultJavaVMInitArgs_t GetDefaultJavaVMInitArgs;
    GetCreatedJavaVMs_t GetCreatedJavaVMs;
} InvocationFunctions;
```

`CreateJavaVM` 是你马上要见到的核心——从启动器跨进 JVM 的唯一入口。`GetDefaultJavaVMInitArgs` 查询默认 VM 参数。`GetCreatedJavaVMs` 查询当前进程里已经有多少个 JVM 实例。

这三个字段声明后默认是 0（C 的规则：静态或全局变量没显式初始化就是 0）。也就是说，`LoadJavaVM` 调用之前，`ifn.CreateJavaVM` 的值是 `NULL`。如果这时候就调用它——立刻段错误。这就是为什么 `LoadJavaVM` 必须先跑通。

这个结构体就是 JNI 规范第 5 章"Invocation API"的最小接口。HotSpot 只需要实现这三个符号（`JNI_CreateJavaVM`、`JNI_GetDefaultJavaVMInitArgs`、`JNI_GetCreatedJavaVMs`），任何启动器都可以用这套接口来创建和查询 JVM。

## LoadJavaVM —— dlopen + dlsym 的灵魂三连

`LoadJavaVM` 在 `java_md_solinux.c:552`。它是本章最核心的函数——把三个空函数指针变成真正可调用的执行入口。

```c
// java_md_solinux.c:553
jboolean
LoadJavaVM(const char *jvmpath, InvocationFunctions *ifn) {
```

参数就两个：`jvmpath`（`CreateExecutionEnvironment` 拼出来的 libjvm.so 路径）、`ifn`（那个空的结构体指针）。返回类型 `jboolean` 在 JNI 里等于 `unsigned char`——只有两个值：`JNI_TRUE`（1）和 `JNI_FALSE`（0）。这和 C 的 `bool` 不一样（`bool` 是 `_Bool`），它是 JNI 跨平台兼容的产物——Windows 的 JNI 头和 Linux 的可能不一样，用 `jboolean` 统一。

### dlopen：打开 libjvm.so

```c
// java_md_solinux.c:559
    libjvm = dlopen(jvmpath, RTLD_NOW + RTLD_GLOBAL);
```

`RTLD_NOW + RTLD_GLOBAL` 是两个 flag 的按位或（值分别是 `0x00002` 和 `0x00100`）。

`dlopen` 是 Linux 动态链接器的 API——它的作用和 `ldd` 看到的"自动加载"完全相反：自动加载发生在程序启动时，`dlopen` 发生在运行时。你可以在程序的任意时刻打开一个 .so 并用里面的函数。

两个 flag：`RTLD_NOW` 要求立即解析 libjvm.so 的所有符号引用——如果缺少任何依赖符号，`dlopen` 立刻失败，不会等到调用某个 JVM 函数时才崩溃。`RTLD_GLOBAL` 让 libjvm.so 的符号对后续用 `dlopen` 加载的其他 .so 可见——JVM 内部有多个 .so 互相引用，这个 flag 必须加上。

```c
// java_md_solinux.c:560
    if (libjvm == NULL) {
        JLI_ReportErrorMessage(DLL_ERROR1, __LINE__);
        JLI_ReportErrorMessage(DLL_ERROR2, jvmpath, dlerror());
        return JNI_FALSE;
    }
```

`dlopen` 失败最常见的三个原因：路径不存在（`jvmpath` 拼错了）、64/32 bit 不匹配（32 位程序试图加载 64 位 .so）、依赖库缺失（libjvm.so 依赖的某个 .so 没装）。`dlerror()` 会告诉你具体是哪个。

### 三个 dlsym：掏出函数指针

```c
// java_md_solinux.c:612
    ifn->CreateJavaVM = (CreateJavaVM_t)
        dlsym(libjvm, "JNI_CreateJavaVM");
```

`dlsym` 的第二个参数是字符串——`"JNI_CreateJavaVM"`，带引号。不是变量名、不是宏——就是编译器能看到的普通字符串。效果是：在 libjvm.so 的符号表里搜这个字符串，找到对应的地址，转成函数指针 `(CreateJavaVM_t)`。

如果你写成 `dlsym(libjvm, JNI_CreateJavaVM)`（不带引号），编译器会报错或者把它当变量名处理——那就完全错了。

调用成功之后，`ifn->CreateJavaVM` 不再指向 NULL——它指向 `jni.cpp` 里的 `JNI_CreateJavaVM` 函数。下一章就要走到这个函数里面去。

```c
// java_md_solinux.c:618
    ifn->GetDefaultJavaVMInitArgs = (GetDefaultJavaVMInitArgs_t)
        dlsym(libjvm, "JNI_GetDefaultJavaVMInitArgs");
```

```c
// java_md_solinux.c:625
    ifn->GetCreatedJavaVMs = (GetCreatedJavaVMs_t)
        dlsym(libjvm, "JNI_GetCreatedJavaVMs");
```

同样的模式，再来两次。每一个 `dlsym` 之后都是 NULL 检查——缺少任何一个符号，JVM 就不完整，必须失败。

三个函数都成功后，返回 `JNI_TRUE`。此时 `ifn` 不再是一个空壳——它的三个字段都指向了 libjvm.so 里真正可调用的函数。

## 为什么要在新线程中启动 JVM

`LoadJavaVM` 返回后，启动器手里有了 `ifn->CreateJavaVM`，理论上可以直接调用。但它没有——先调了 `JVMInit`：

```c
// java.c:343
    return JVMInit(&ifn, threadStackSize, argc, argv, mode, what, ret);
```

`JVMInit` 在 `java_md_solinux.c:806`，就两行：

```c
// java_md_solinux.c:807
    ShowSplashScreen();
    return ContinueInNewThread(ifn, threadStackSize, argc, argv, mode, what, ret);
```

第一行画启动画面（如果有 `-splash` 参数，`ShowSplashScreen` 在一个独立的窗口线程里显示一张图片，JVM 启动完成后由 JVM 负责关掉它）。第二行 `ContinueInNewThread` 内部调 `pthread_create`——创建一个全新的线程，在新线程里执行 `JavaMain()`。

为什么要开新线程？原始线程（调 `main()` 的那个线程）有几个不可以改的属性：栈大小由操作系统初始分配决定、信号处理器已经绑定、栈空间不可扩展。JVM 需要精确控制这些事情——特别是 Java 线程的栈 guard pages（防止栈溢出覆盖其他内存），必须在 `pthread_create` 时就能设定。

JDK 6 早期有个 bug（JDK-6316197）就是因为直接在主线程启动 JVM，导致 JVM 的 StackGuardPages 和操作系统的默认栈大小冲突。修复方案就是永远在新线程中启动。

`JVMInit` 不会返回——它内部最终会调用 `exit()`。所以 `main()` 的 `return JLI_Launch(...)` 实际上也不会执行到。整个进程的生命周期从这一行开始，交给了新线程里的 JVM。
