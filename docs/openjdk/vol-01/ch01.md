# 第1章：Launcher Chain —— 从命令行到 libjvm.so

上一卷最后你亲手编译了一个 JDK。敲 `java -version` 的时候，从屏幕上看到三行输出，觉得挺自然——JDK 嘛，一个命令启动一个 JVM，很正常。

但如果你稍微想一下：这个 `java` 可执行文件才 23KB，而 JVM 本体 `libjvm.so` 有 164MB。一个 23KB 的 C 程序，怎么启动一个 164MB 的 Java 虚拟机？

本章就顺着这个好奇走下去——从 `main()` 开始，到 `dlopen("libjvm.so")` 为止。

---

## java 没链接 libjvm.so

先验证一个反直觉的事实。用 `ldd` 看 java 链接了哪些库：

```
$ ldd build/linux-x86_64-normal-server-slowdebug/jdk/bin/java
	libjli.so => .../jdk/bin/./../lib/jli/libjli.so
	libc.so.6 => /lib64/libc.so.6
	libz.so.1 => /lib64/libz.so.1
	/lib64/ld-linux-x86-64.so.2
```

三个依赖：`libc`（标准 C 库）、`libz`（压缩）、`libjli.so`。没有 `libjvm.so`。

你用 `readelf` 再看一遍 RUNPATH——java 只需要找到 `libjli.so`：

```
$ readelf -d build/linux-x86_64-normal-server-slowdebug/jdk/bin/java | grep RUNPATH
  (RUNPATH)            Library runpath: [$ORIGIN/../lib/jli]
```

`$ORIGIN` 是 ELF 动态链接器的特殊变量，展开为 java 二进制所在的目录。所以运行时找 `libjli.so` 就是 `java所在目录/../lib/jli/`。这个 RUNPATH 里只有 jli——没有 server、没有 libjvm、什么都没有。

`libjvm.so` 不是静态链接进来的，也不是动态链接器自动加载的。它是 java 启动器在运行中**手动**加载的。

这就是本章要讲的：启动器怎么找到 libjvm.so、怎么打开它、怎么从里面掏出 `JNI_CreateJavaVM` 这个函数。

---

## main() 在做什么——不是在"执行 Java"

java 的 `main()` 在 `src/java.base/share/native/launcher/main.c`。和 Ch0.1 那个 hello world 一样——标准 C 程序入口。但它不解析 classpath、不创建虚拟机——它只是**准备参数**。

搞懂这 150 行代码之前，先认识所有变量。这段代码名字起得特别容易混，列出来：

| 变量 | 类型 | 从哪来 | 干什么 |
|------|------|--------|--------|
| `argc` | `int` | OS 传给 `main()` | argv 的元素个数 |
| `argv` | `char **` | OS 传给 `main()` | 原始命令行参数数组 |
| `margc` | `int` | 函数内计算 | 最终传给 JLI_Launch 的参数个数 |
| `margv` | `char **` | 函数内构建 | 最终传给 JLI_Launch 的参数数组 |
| `jargc` | `int` | 编译期计算 | 编译期 Java 参数个数 |
| `jargv` | `char **` | 编译期常量 | 编译期 Java 参数数组 |
| `args` | `JLI_List` | 函数内临时变量 | 构建 margv 过程中的中转站 |

一句话区分：`argc/argv` 是 OS 给的，`margc/margv` 是加工后给 JLI_Launch 的，`jargc/jargv` 是编译期写死的，`args` 是临时的。

比如你敲 `java -version`，OS 传给 `main()` 的就是：

```
argc = 2
argv[0] = "java"
argv[1] = "-version"
```

`argv[0]` 是程序名，参数从 `argv[1]` 开始——C 语言的标准约定。

搞清这些之后，直接把 `main.c` 的完整源码摆出来。**粗体注释**是阶段标注：

```c
/* === src/java.base/share/native/launcher/main.c === */
/* 文件头省略：GPL 声明 + #include "defines.h" / "jli_util.h" / "jni.h" */

/* _MSC_VER 和 JAVAW 块是 Windows 专用入口，Linux 不编译，省略 */

JNIEXPORT int
main(int argc, char **argv) {
    int margc;
    char **margv;
    int jargc;
    char **jargv;
    const jboolean const_javaw = JNI_FALSE;
    {
        /* ═══════ **阶段 1：构建 jargv（编译期 JVM 参数）** ═══════ */
        int i, main_jargc, extra_jargc;
        JLI_List list;

        main_jargc = (sizeof(const_jargs) / sizeof(char *)) > 1
                     ? sizeof(const_jargs) / sizeof(char *) : 0;
        extra_jargc = (sizeof(const_extra_jargs) / sizeof(char *)) > 1
                      ? sizeof(const_extra_jargs) / sizeof(char *) : 0;

        if (main_jargc > 0 && extra_jargc > 0) {
            jargc = main_jargc + extra_jargc;
            list = JLI_List_new(jargc + 1);
            for (i = 0; i < extra_jargc; i++)
                JLI_List_add(list, JLI_StringDup(const_extra_jargs[i]));
            for (i = 0; i < main_jargc; i++)
                JLI_List_add(list, JLI_StringDup(const_jargs[i]));
            JLI_List_add(list, NULL);
            jargv = list->elements;
        } else if (extra_jargc > 0) {
            fprintf(stderr, "EXTRA_JAVA_ARGS defined without JAVA_ARGS");
            abort();
        } else {
            jargc = main_jargc;
            jargv = (char **) const_jargs;
        }
    }

    /* ═══════ **阶段 2：参数预处理开关** ═══════ */
    JLI_InitArgProcessing(jargc > 0, const_disable_argfile);

    /* _WIN32 块是 Windows 参数解析，Linux 不编译，省略 */

    /* ═══════ **阶段 3：构建 margv（命令行 + 环境变量）** ═══════ */
    {
        JLI_List args = JLI_List_new(argc + 1);
        int i = 0;

        JLI_List_add(args, JLI_StringDup(argv[0]));
        if (JLI_AddArgsFromEnvVar(args, JDK_JAVA_OPTIONS)) {
            if (getenv(JLDEBUG_ENV_ENTRY)) {
                char *tmp = getenv("_JAVA_OPTIONS");
                if (NULL != tmp)
                    JLI_ReportMessage(ARG_INFO_ENVVAR, "_JAVA_OPTIONS", tmp);
            }
        }
        for (i = 1; i < argc; i++) {
            JLI_List argsInFile = JLI_PreprocessArg(argv[i], JNI_TRUE);
            if (NULL == argsInFile) {
                JLI_List_add(args, JLI_StringDup(argv[i]));
            } else {
                int cnt, idx;
                cnt = argsInFile->size;
                for (idx = 0; idx < cnt; idx++)
                    JLI_List_add(args, argsInFile->elements[idx]);
                JLI_MemFree(argsInFile->elements);
                JLI_MemFree(argsInFile);
            }
        }
        margc = args->size;
        JLI_List_add(args, NULL);
        margv = args->elements;
    }

    /* ═══════ **阶段 4：调试用硬编码参数覆盖** ═══════ */
    /* jdk11u-copy 定制功能，原版 OpenJDK 没有这段 */
    if (const_progname != NULL && JLI_StrCmp(const_progname, "java") == 0) {
        static char *hardcoded_argv[] = {
                "java",
                "-version",
                NULL,
        };
        margc = (int) (sizeof(hardcoded_argv) / sizeof(hardcoded_argv[0])) - 1;
        margv = hardcoded_argv;
        jargc = 0;
        jargv = NULL;
    }

    /* ═══════ **阶段 5：交接给 libjli** ═══════ */
    return JLI_Launch(margc, margv,
                      jargc, (const char **) jargv,
                      0, NULL,
                      VERSION_STRING,
                      DOT_VERSION,
                      (const_progname != NULL) ? const_progname : *margv,
                      (const_launcher != NULL) ? const_launcher : *margv,
                      jargc > 0,
                      const_cpwildcard, const_javaw, 0);
}
```

代码入口第一行有个关键字——`JNIEXPORT int main(...)`。`JNIEXPORT` 在 `src/hotspot/os/linux/jni_md.h:34`：

```c
#define JNIEXPORT __attribute__((visibility("default")))
```

GCC 编译指令——让符号对外可见。对可执行文件的 `main()` 没有实际效果，OpenJDK 用它保持 JNI 边界代码风格统一。另一个 `JNICALL` 同理——Linux 上空宏，Windows 上 `__stdcall`。

### 第一步（真实顺序）：构建 jargv —— 编译期写死的参数

`jargv` 不走命令行。它来自编译时常量 `JAVA_ARGS` 和 `EXTRA_JAVA_ARGS`：

```c
    main_jargc = (sizeof(const_jargs) / sizeof(char *)) > 1
                 ? sizeof(const_jargs) / sizeof(char *) : 0;

    extra_jargc = (sizeof(const_extra_jargs) / sizeof(char *)) > 1
                  ? sizeof(const_extra_jargs) / sizeof(char *) : 0;
```

`sizeof(const_jargs)` 除以 `sizeof(char *)`——数组总字节数除以每个元素的字节数，得到元素个数。`> 1` 的检查是因为数组至少有两个元素：至少一个内容加上末尾的 NULL。如果只有一个元素（只有一个 NULL），说明这个启动器没有编译期 Java 参数。

`java` 启动器的 `const_jargs` 定义为：

```c
// (file reference)
static const char* const_jargs[] = { NULL };
```

所以 `main_jargc = 0`，`jargv = NULL`。

但编 `javac` 时，CMake 传入了 `-DJAVA_ARGS='{ "-J-ms8m", "-m", "jdk.compiler/..." }'`：

```c
// (file reference)
static const char* const_jargs[] = { "-J-ms8m", "-m", "jdk.compiler/...", NULL };
```

`-J` 前缀的意思是"这个参数给 JVM，不给 Java 程序"。所以 `jargv` 里的东西最终变成 JVM 的启动参数，而 `margv` 里不带 `-J` 的才是传给 `main(String[] args)` 的。

如果 `extra_jargc` 也有值，两个数组合并——extra 在先、main 在后，后者覆盖前者。

### 第二步：InitArgProcessing —— 参数预处理的开关

```c
    JLI_InitArgProcessing(jargc > 0, const_disable_argfile);
```

这个函数设置两个全局标志。第一个 `jargc > 0` = "有没有编译期 Java 参数"，决定了后续是否处理 `-J` 前缀。第二个 `const_disable_argfile` = jdk11u-copy 没设，所以 `@opts.txt` 展开正常支持。

### 第三步（平台分支）：Linux x86_64 路径

```c
#ifdef _WIN32
    // UTF-16 → ANSI 编码转换，本书跳过
#else
    // Linux x86_64 走这里
```

`#ifdef _WIN32` 后面的代码是 Windows 专用的——UTF-16 命令行到 ANSI 的转换。本书不展开，直接看 `#else` 分支。

`margv` 构建用了一个自定义的数据结构，先把它看清楚。

**JLI_List**——不是 C 标准库的东西，是启动器在 `jli_util.h` 里手搓的动态数组：

```c
typedef struct {
    int       size;      // 当前元素数
    int       capacity;  // 总容量
    char    **elements;  // 字符串数组
} JLI_List;
```

> **`char **elements` 为什么是"字符串数组"？**
>
> C 里没有 `string` 类型。一个字符串是 `char *`——指向一串连续 `char` 的首地址。多个字符串放到一起，就是一个 `char *` 的数组，内存里是"指针的指针"，也就是 `char **`。
>
> 比如 `margv` 存了 `"java"`、`"-version"`、`NULL` 三个字符串：
>
> ```
> elements → [0] → "java\0"
>             [1] → "-version\0"
>             [2] → NULL
> ```
>
> `elements[0]` 是 `char *`，指向 `"java"`；`elements[1]` 指向 `"-version"`。末尾的 `elements[2]` 是 `NULL`（空指针），不指向任何字符串——它只标记数组终点。
>
> C 里所有参数传递（`main` 的 `argv`、`JLI_Launch` 的 `margv`）都用这个模式：二级指针 + NULL 哨兵。

三个操作：`JLI_List_new(capacity)` 分配、`JLI_List_add(list, str)` 追加（满了自动翻倍扩容）、`JLI_List_add(list, NULL)` 加哨兵。`JLI_StringDup` 内部调 `malloc + memcpy`——因为 `argv` 是操作系统分配的内存，启动器不拥有它，复制一份才能随意管理生命周期。

回到 margv 构建。这里的 `args` 就是上面速查表里的那个临时 JLI_List：

```c
    JLI_List args = JLI_List_new(argc + 1);
    JLI_List_add(args, JLI_StringDup(argv[0]));       // 从 argv[0] 复制 "java" 到 args
```

`JLI_StringDup(argv[0])` 把 `argv[0]`（就是 `"java"` 这个程序名）复制一份到堆上。`JLI_List_add` 把这个副本的指针放进 `args` 列表。注意方向——是从 `argv` 搬到 `args`，不是反过来。

此时 `args` 里第一个元素是 `"java"`。

```c
    if (JLI_AddArgsFromEnvVar(args, JDK_JAVA_OPTIONS)) {
```

注入 `JDK_JAVA_OPTIONS` 环境变量。内部是 `getenv` + 空格分割 + 逐个 `JLI_List_add`。环境变量没设就什么都不加。注意它**在 `argv[1]` 之前**——所以命令行上的同名参数会覆盖环境变量里的。

> 你大概率永远不会设 `JDK_JAVA_OPTIONS`——它只在 Docker 镜像、CI 流水线这种需要全局注入 JVM 参数的场景才用。理解"有这条路径就够了"，不用记细节。

举例：设了 `export JDK_JAVA_OPTIONS="-Xmx128m"`，然后敲 `java -Xmx256m -version`。margv 顺序：`"java"` → `"-Xmx128m"`（环境变量）→ `"-Xmx256m"`（命令行）→ `"-version"`。JVM 取最后一个 `-Xmx`——命令行 256m 生效，环境变量 128m 被覆盖。

```c
    for (i = 1; i < argc; i++) {
        JLI_List argsInFile = JLI_PreprocessArg(argv[i], JNI_TRUE);
```

逐个处理 `argv[1]`、`argv[2]`……`JLI_PreprocessArg` 负责 `@filename` 展开。如果你写 `java @opts.txt`，`opts.txt` 的内容会被拆成参数插入。嵌套展开也支持。

> `@argfile` 也很少用到——Java 命令行太长时才会把参数存文件里。日常敲 `java -version` 不用管它。

```c
    margc = args->size;
    margv = args->elements;
```

收尾：NULL 哨兵、返回数组指针。margv 构建完成。

### 第四步：硬编码参数覆盖 —— CLion 调试不用配参数

```c
    if (const_progname != NULL && JLI_StrCmp(const_progname, "java") == 0) {
        static char *hardcoded_argv[] = { "java", "-version", NULL };
        margc = (int) (sizeof(hardcoded_argv) / sizeof(hardcoded_argv[0])) - 1;
        margv = hardcoded_argv;
        jargc = 0;
        jargv = NULL;
    }
```

`const_progname` 是 CMake 编译期常量——编 `java` 启动器时等于 `"java"`，编 `javac` 时等于 `"javac"`。所以只有 `java` 会走进这个 `if`。`JLI_StrCmp` 是启动器自己的字符串比较，等价于 `strcmp`。

这整段代码就是 CLion 里点三角号就能跑的秘密——不用去 Run Configuration 配参数。它覆盖了前面所有构建出来的 `margv` 和 `jargv`。


### 第五步：交接给 libjli

```c
    return JLI_Launch(margc, margv,
                   jargc, (const char**) jargv,
                   0, NULL,
                   VERSION_STRING, DOT_VERSION,
                   (const_progname != NULL) ? const_progname : *margv,
                   (const_launcher != NULL) ? const_launcher : *margv,
                   jargc > 0, const_cpwildcard, const_javaw, 0);
```

11 个参数逐一解释：

| 参数 | 值 | 含义 |
|------|-----|------|
| `margc, margv` | 命令行参数 | 你敲的东西 |
| `jargc, jargv` | 编译期参数 | java 启动器为空 |
| `0, NULL` | appclassc/appclassv | applet 用的，JDK 11 已废弃 |
| `VERSION_STRING` | `"11.0.32-internal"` | 完整版本号 |
| `DOT_VERSION` | `"11.0"` | 点分主版本 |
| `const_progname` | `"java"` | 程序名 |
| `const_launcher` | `"openjdk"` | 启动器名 |
| `jargc > 0` | `JNI_FALSE` | 有没有编译期 Java 参数 |
| `const_cpwildcard` | `JNI_TRUE` | classpath 支持 `*` 通配符 |
| `const_javaw` | `JNI_FALSE` | Windows javax 模式，Linux 始终 false |
| 最后一个 `0` | 0 | 保留参数，源码标注 UNUSED |

`JLI_Launch` 在 `src/java.base/share/native/libjli/java.c`——从 `main.c` 跨到 libjli 库的第一个函数。

**到这里为止，`main()` 干了什么？** 把命令行参数拆成两套数组（用户敲的 `margv`、编译期写死的 `jargv`），在 Linux 平台上跳过 Windows 分支，最后如果 `progname == "java"` 就用硬编码数组覆盖全部。然后把这些参数连同版本号、启动器名、通配符开关等一起打包，调用 `JLI_Launch`。150 行 C 代码，没有一行和 JVM 有关——纯粹的准备参数。

## JLI_Launch() —— 11 步流水线

`main()` 最后一行的 `JLI_Launch` 在 `java.c` 中。和 `main()` 一样——先看源码全貌，再逐个深入。

```c
/* === src/java.base/share/native/libjli/java.c === */
JNIEXPORT int JNICALL
JLI_Launch(int argc, char ** argv,              /* main argc, argv */
        int jargc, const char** jargv,          /* java args */
        int appclassc, const char** appclassv,  /* app classpath */
        const char* fullversion,                /* full version defined */
        const char* dotversion,                 /* UNUSED dot version defined */
        const char* pname,                      /* program name */
        const char* lname,                      /* launcher name */
        jboolean javaargs,                      /* JAVA_ARGS */
        jboolean cpwildcard,                    /* classpath wildcard*/
        jboolean javaw,                         /* windows-only javaw */
        jint ergo                               /* unused */
)
{
    int mode = LM_UNKNOWN;
    char *what = NULL;
    char *main_class = NULL;
    int ret;
    InvocationFunctions ifn;
    jlong start = 0, end = 0;
    char jvmpath[MAXPATHLEN];
    char jrepath[MAXPATHLEN];
    char jvmcfg[MAXPATHLEN];

    /* == Step 1: 全局状态设置 == */
    _fVersion = fullversion;
    _launcher_name = lname;
    _program_name = pname;
    _is_java_args = javaargs;
    _wc_enabled = cpwildcard;

    /* == Step 2: 平台初始化 + 追踪 == */
    InitLauncher(javaw);
    DumpState();
    if (JLI_IsTraceLauncher()) {
        int i;
        printf("Java args:\n");
        for (i = 0; i < jargc ; i++) {
            printf("jargv[%d] = %s\n", i, jargv[i]);
        }
        printf("Command line args:\n");
        for (i = 0; i < argc ; i++) {
            printf("argv[%d] = %s\n", i, argv[i]);
        }
        AddOption("-Dsun.java.launcher.diag=true", NULL);
    }

    /* == Step 3: JRE 版本选择（JDK 9+ 拒绝 -version: 语法） == */
    SelectVersion(argc, argv, &main_class);

    /* == Step 4: 构建执行环境 — 找 libjvm.so 路径 == */
    CreateExecutionEnvironment(&argc, &argv,
                               jrepath, sizeof(jrepath),
                               jvmpath, sizeof(jvmpath),
                               jvmcfg,  sizeof(jvmcfg));

    /* == Step 5: 设置 JVM 环境变量 (JAVA_HOME / LD_LIBRARY_PATH) == */
    if (!IsJavaArgs()) {
        SetJvmEnvironment(argc,argv);
    }

    /* == Step 6: 加载 JVM 动态库 == */
    ifn.CreateJavaVM = 0;
    ifn.GetDefaultJavaVMInitArgs = 0;

    if (JLI_IsTraceLauncher()) {
        start = CounterGet();
    }
    if (!LoadJavaVM(jvmpath, &ifn)) {
        return(6);
    }
    if (JLI_IsTraceLauncher()) {
        end   = CounterGet();
    }
    JLI_TraceLauncher("%ld micro seconds to LoadJavaVM\n",
             (long)(jint)Counter2Micros(end-start));

    /* == Step 7: 参数预处理 — 跳过 argv[0] + 转换/CLASSPATH == */
    ++argv;
    --argc;
    if (IsJavaArgs()) {
        TranslateApplicationArgs(jargc, jargv, &argc, &argv);
        if (!AddApplicationOptions(appclassc, appclassv)) {
            return(1);
        }
    } else {
        char* cpath = getenv("CLASSPATH");
        if (cpath != NULL) {
            SetClassPath(cpath);
        }
    }

    /* == Step 8: 解析命令行选项 (-cp / -jar / -m 等) == */
    if (!ParseArguments(&argc, &argv, &mode, &what, &ret, jrepath)) {
        return(ret);
    }

    /* == Step 9: -jar 特殊处理 == */
    if (mode == LM_JAR) {
        SetClassPath(what);
    }

    /* == Step 10: 设置伪系统属性 == */
    SetJavaCommandLineProp(what, argc, argv);
    SetJavaLauncherProp();
    SetJavaLauncherPlatformProps();

    /* == Step 11: 在新线程中启动 JVM == */
    return JVMInit(&ifn, threadStackSize, argc, argv, mode, what, ret);
}
```

这个函数 123 行，11 个步骤按顺序走完。下面分三个重要度层级展开：★ 了解、★★ 概述、★★★ 详细走读。

### Step 1-3：开场准备

Step 1 把五个模块级全局变量赋了初值——程序名、启动器名、版本号、是否工具启动器、通配符开关。没有什么逻辑，就是把传进来的参数存到全局，后面所有函数都能访问。

Step 2 调了 `InitLauncher` 做平台初始化（信号处理等），然后如果启用了追踪模式（`JLI_IsTraceLauncher()`），打印当前的 jargv 和 argv 内容到 stdout。日常跑没有追踪模式，这一步等于空操作。

Step 3 调 `SelectVersion`，检查用户有没有用 `-version:` 语法指定 JRE 版本。JDK 9+ 已经不支持多 JRE 共存，所以这个函数的主要功能就是拒绝这种调用。

> Step 1-3 日常敲 `java -version` 都不会触发任何实际动作。知道它们存在就行。

### Step 4★：构建执行环境 —— 找到 libjvm.so

这是第一个核心步骤。调用点在 `java.c:278-281`：

```c
    CreateExecutionEnvironment(&argc, &argv,
                               jrepath, sizeof(jrepath),
                               jvmpath, sizeof(jvmpath),
                               jvmcfg,  sizeof(jvmcfg));
```

一共 8 个参数，构成 4 对：

| 参数对 | 作用 | 方向 |
|--------|------|------|
| `&argc, &argv` | 命令行参数 | `main()` 加工后传下来的 margc/margv。传地址是因为函数内部会检查并移除 `-server`/`-client` 等 VM 类型参数 |
| `jrepath, sizeof(jrepath)` | JRE 根目录缓冲区 | **输出**。`jrepath` 是调用方栈上的 `char[MAXPATHLEN]` 数组，`sizeof` 告诉函数缓冲区多大 |
| `jvmpath, sizeof(jvmpath)` | libjvm.so 路径缓冲区 | **输出**。同上模式——指针 + 容量，函数往里写结果 |
| `jvmcfg, sizeof(jvmcfg)` | jvm.cfg 路径缓冲区 | **输出**。同上，函数拼出 `%s/lib/jvm.cfg` 后写到这里 |

`jrepath`、`jvmpath`、`jvmcfg` 三个变量在 `JLI_Launch` 开头声明为 `char jrepath[MAXPATHLEN]`，栈上分配。`sizeof` 在编译期求值——不是运行时算出来的。三个值都不是编译期写死的，是运行时从文件系统推导出来的。

先看 `CreateExecutionEnvironment` 的完整实现。它在 `java_md_solinux.c:303-364`，六个子步骤依次调用，每个失败都有明确的退出码：

```c
void
CreateExecutionEnvironment(int *pargc, char ***pargv,
                           char jrepath[], jint so_jrepath,
                           char jvmpath[], jint so_jvmpath,
                           char jvmcfg[],  jint so_jvmcfg) {

    char * jvmtype = NULL;
    int argc = *pargc;
    char **argv = *pargv;

    /* Step A：获取可执行文件绝对路径 */
    SetExecname(*pargv);

    /* Step B：找到 JRE 根目录 */
    if (!GetJREPath(jrepath, so_jrepath, JNI_FALSE)) {
        JLI_ReportErrorMessage(JRE_ERROR1);
        exit(2);                             // 退出码 2：找不到 JRE
    }

    /* Step C：拼出 jvm.cfg 路径 */
    JLI_Snprintf(jvmcfg, so_jvmcfg, "%s%slib%sjvm.cfg",
            jrepath, FILESEP, FILESEP);

    /* Step D：读取并解析 jvm.cfg */
    if (ReadKnownVMs(jvmcfg, JNI_FALSE) < 1) {
        JLI_ReportErrorMessage(CFG_ERROR7);
        exit(1);                             // 退出码 1：jvm.cfg 无效
    }

    jvmpath[0] = '\0';

    /* Step E：确定 VM 类型（server/client） */
    jvmtype = CheckJvmType(pargc, pargv, JNI_FALSE);
    if (JLI_StrCmp(jvmtype, "ERROR") == 0) {
        JLI_ReportErrorMessage(CFG_ERROR9);
        exit(4);                             // 退出码 4：VM 类型无效
    }

    /* Step F：拼接 libjvm.so 的完整路径 */
    if (!GetJVMPath(jrepath, jvmtype, jvmpath, so_jvmpath)) {
        JLI_ReportErrorMessage(CFG_ERROR8, jvmtype, jvmpath);
        exit(4);
    }
}
```

六个子步骤的源码都在三个文件里：`java_md_solinux.c`、`java_md_common.c`（各平台共用）、`java_md.h`（函数声明）。逐一展开。

**Step A：SetExecname** —— 读 `/proc/self/exe`。`java_md_solinux.c:649-692`，只看 Linux 分支：

```c
static void
SetExecname(char **argv)
{
    char* exec_path = NULL;
    const char* self = "/proc/self/exe";
    char buf[PATH_MAX+1];
    int len = readlink(self, buf, PATH_MAX);
    if (len >= 0) {
        buf[len] = '\0';                  // readlink 不会自动加 '\0'
        exec_path = JLI_StringDup(buf);   // 复制到堆上
    }
    /* Solaris 分支用 dladdr(dlsym("main")) 兜底，Linux 跳过 */
}
```

关键代码就一行：`readlink("/proc/self/exe", buf, PATH_MAX)`

`readlink` 读符号链接的目标路径。`/proc/self/exe` 是 Linux 内核暴露的一个符号链接，永远指向读取它的那个进程的可执行文件。

在终端执行 `readlink /proc/self/exe`，`self` 是 bash，结果是 bash 的路径。但这段 C 代码跑在 java 进程里，`self` 是 java，读到的是 java 二进制的位置。本机 java 进程里，这行代码的结果是：

```
/data/workspace/jdk11u-copy/build/linux-x86_64-normal-server-slowdebug/jdk/bin/java
```

`readlink` 不会自动加 `\0`，下一行 `buf[len] = '\0'` 手动补上末尾。然后 `JLI_StringDup` 复制到堆，存进模块级变量 `exec_path`。

**Step B：GetJREPath** —— 拿 `exec_path` 逐层回退找 JRE 根目录。

```c
// java_md_common.c:79-90 — GetApplicationHome：截取 JDK 根目录
jboolean GetApplicationHome(char *buf, jint bufsize) {
    const char *execname = GetExecName();  // 取 Step A 存的 exec_path
    if (execname != NULL) {
        JLI_Snprintf(buf, bufsize, "%s", execname);
        buf[bufsize-1] = '\0';
    } else {
        return JNI_FALSE;
    }
    return TruncatePath(buf, JNI_FALSE);
}
```

`TruncatePath(buf, JNI_FALSE)` 找路径里最后一个 `/bin/`，截断。本机的结果：

```
exec_path = /data/workspace/jdk11u-copy/build/linux-x86_64-normal-server-slowdebug/jdk/bin/java
buf       = /data/workspace/jdk11u-copy/build/linux-x86_64-normal-server-slowdebug/jdk
```

`GetApplicationHome` 把可执行文件路径变成了 JDK 根目录。接下来 `GetJREPath` 用这个根目录去验证 `libjava.so` 是否存在：

```c
static jboolean GetJREPath(char *path, jint pathsize, jboolean speculative) {
    char libjava[MAXPATHLEN];

    if (GetApplicationHome(path, pathsize)) {
        JLI_Snprintf(libjava, sizeof(libjava), "%s/lib/" JAVA_DLL, path);
        if (access(libjava, F_OK) == 0) {
            return JNI_TRUE;
        }
        // 旧版私有 JRE：检查 path/jre/lib/libjava.so，正常不走
        // dladdr 反查 libjli.so：最终兜底，正常不走
    }
    JLI_ReportErrorMessage(JRE_ERROR8 JAVA_DLL);
    return JNI_FALSE;
}
```

`JAVA_DLL` 在 Linux 上是 `"libjava.so"`。在你的 JDK 里，这个文件就在 `lib/` 目录下：

```
$ ls jdk11u-copy/build/linux-x86_64-normal-server-slowdebug/jdk/lib/libjava.so
.../jdk11u-copy/build/.../jdk/lib/libjava.so
```

`access(libjava, F_OK)` 检查到文件存在，返回 0，`path` 就是 JRE 根目录。

**Step C：拼 jvm.cfg 路径** —— `JLI_Snprintf(jvmcfg, so_jvmcfg, "%s%slib%sjvm.cfg", jrepath, FILESEP, FILESEP)`。`FILESEP` 在 Linux 上是 `"/"`。本机上的结果：

```
/data/workspace/jdk11u-copy/build/linux-x86_64-normal-server-slowdebug/jdk/lib/jvm.cfg
```

**Step D：ReadKnownVMs** —— 打开 jvm.cfg，逐行解析。每行格式 `<vmtype> <flag>`：

| 标记 | 含义 |
|------|------|
| `KNOWN` | 可用 |
| `IGNORE` | 跳过（不兼容当前平台） |
| `ALIASED_TO` | 别名，指向另一个 VM |
| `WARN` | 可用但有兼容性问题 |
| `ERROR` | 不可用，尝试加载会失败 |

JDK 11 的 `jvm.cfg` 通常只有一行 `-server KNOWN`。ReadKnownVMs 是 ~200 行的文本解析器，在 jvm.cfg 只有一行时大部分路径走不到，不展开。

**Step E：CheckJvmType** —— 从三个来源确定 VM 类型。优先级：命令行 `-server`/`-client` > `JDK_ALTERNATE_VM` 环境变量 > jvm.cfg 第一条非 IGNORE 记录。这里会**修改 `argv`**（`pargc`/`pargv` 是传地址）——消费掉 VM 类型参数，后续不会重复解析到。

**Step F：GetJVMPath** —— 拼接 libjvm.so 的绝对路径。`java_md_solinux.c:486-507`：

```c
static jboolean GetJVMPath(const char *jrepath, const char *jvmtype,
                           char *jvmpath, jint jvmpathsize) {
    struct stat s;

    if (JLI_StrChr(jvmtype, '/')) {
        /* jvmtype 含 / 时当绝对路径用：-XXaltjvm=/custom/path */
        JLI_Snprintf(jvmpath, jvmpathsize, "%s/" JVM_DLL, jvmtype);
    } else {
        /* 标准模式：{jrepath}/lib/{jvmtype}/libjvm.so */
        JLI_Snprintf(jvmpath, jvmpathsize, "%s/lib/%s/" JVM_DLL,
                     jrepath, jvmtype);
    }

    if (stat(jvmpath, &s) == 0) {
        return JNI_TRUE;       // 文件存在
    } else {
        return JNI_FALSE;      // 路径存在但 .so 没了
    }
}
```

`JVM_DLL` 在 Linux 上展开为 `"libjvm.so"`。本机上的标准路径：

```
/data/workspace/jdk11u-copy/build/linux-x86_64-normal-server-slowdebug/jdk/lib/server/libjvm.so
```

最后用 `stat()` 确认文件真实存在——防御 jvm.cfg 列出 `-server KNOWN` 但 libjvm.so 被删除的情况。

> 六步走完，`jrepath`、`jvmcfg`、`jvmpath` 三个变量都有了确定值。

### Step 5：设置 JVM 环境变量

```c
    if (!IsJavaArgs()) {
        SetJvmEnvironment(argc, argv);
```

`IsJavaArgs()` 判断当前是不是工具启动器（javac/javap 等）。不是工具启动器时，从命令行参数里读 `-Djava.home` 来设置 `JAVA_HOME`，从 `-Djava.library.path` 设置 `LD_LIBRARY_PATH`。java 启动器通常没有这些参数——这一步日常等于空操作。

### Step 6★：加载 JVM 动态库

```c
    ifn.CreateJavaVM = 0;
    ifn.GetDefaultJavaVMInitArgs = 0;

    if (!LoadJavaVM(jvmpath, &ifn)) {
        return(6);
    }
```

`LoadJavaVM` 是本章的核心——把它展开来讲。

#### InvocationFunctions：启动器与 JVM 的接口

`ifn` 是 `InvocationFunctions` 类型的变量，定义在 `java.h:79-87`：

```c
typedef jint (JNICALL *CreateJavaVM_t)(JavaVM **pvm, void **env, void *args);

typedef struct {
    CreateJavaVM_t CreateJavaVM;
    GetDefaultJavaVMInitArgs_t GetDefaultJavaVMInitArgs;
    GetCreatedJavaVMs_t GetCreatedJavaVMs;
} InvocationFunctions;
```

三个字段都是函数指针，初始化为 0（NULL）。`LoadJavaVM` 的任务是把它们填上：

```c
jboolean
LoadJavaVM(const char *jvmpath, InvocationFunctions *ifn) {
    libjvm = dlopen(jvmpath, RTLD_NOW + RTLD_GLOBAL);
```

`dlopen` 在运行时打开 `libjvm.so`。两个 flag：`RTLD_NOW`（值 0x00002）要求立即解析所有符号——如果 libjvm.so 缺少任何依赖符号，现在就会失败，不会等到调用时才崩溃。`RTLD_GLOBAL`（值 0x00100）让 libjvm.so 的符号对后续 `dlopen` 加载的其他 .so 可见。

`dlopen` 失败最常见的三个原因：路径不存在（jvmpath 拼错了）、64/32 bit 不匹配、依赖库缺失。`dlerror()` 告诉你具体是哪个。

```c
    ifn->CreateJavaVM = (CreateJavaVM_t)
        dlsym(libjvm, "JNI_CreateJavaVM");
```

`dlsym` 的第二个参数是字符串 `"JNI_CreateJavaVM"`——在 libjvm.so 的符号表里搜索这个符号名，找到它的地址，转成函数指针。注意引号——不是变量名，不是宏，就是字符串。如果你写成 `dlsym(libjvm, JNI_CreateJavaVM)`（不带引号），编译器会把它当变量名处理，完全错误。

同样的模式重复三次：

```c
    ifn->GetDefaultJavaVMInitArgs = (GetDefaultJavaVMInitArgs_t)
        dlsym(libjvm, "JNI_GetDefaultJavaVMInitArgs");
    ifn->GetCreatedJavaVMs = (GetCreatedJavaVMs_t)
        dlsym(libjvm, "JNI_GetCreatedJavaVMs");
```

每个 `dlsym` 之后都检查 NULL——缺任何一个符号，JVM 不完整，必须失败。三次成功后，`ifn` 不再是一个空壳——它的三个字段都指向了 libjvm.so 里真正可调用的函数。`ifn->CreateJavaVM` 指向 `jni.cpp` 里的 `JNI_CreateJavaVM`——下一章就要走到那个函数里面。

### Step 7-10：参数拆解与系统属性

Step 7 跳过程序名（`++argv; --argc`），然后如果是工具启动器就调用 `TranslateApplicationArgs` 把 jargv 转换成 `-J` 前缀的参数，否则读取 `CLASSPATH` 环境变量。

Step 8 `ParseArguments` 是另一个大函数——~400 行，逐一消费命令行上的所有 `-` 选项（`-cp`、`-jar`、`-m` 等）。解析完成后，剩余的非选项参数就是 Java 程序的参数。返回值 `mode` 指示启动模式——`LM_CLASS`（普通类）、`LM_JAR`（jar 包）、`LM_MODULE`（模块）等。

Step 9 处理 `-jar` 场景：把 jar 路径设为 classpath。

Step 10 注入三个系统属性：`sun.java.command`（用户敲的完整命令）、`sun.java.launcher`（`SUN_STANDARD` 模式）、`sun.java.launcher.pid`（启动器 PID）。这些不是用户 `-D` 设置的——是启动器"伪造"的。

> Step 7-10 是启动器作为"命令行解析器"的主要工作。日常使用中它们确保你敲的 `-cp`、`-jar`、`MyClass` 被正确分类，系统属性被正确设置。理解它们做什么就行，不用深入内部细节。

### Step 11★：启动 JVM —— 在新线程中

```c
    return JVMInit(&ifn, threadStackSize, argc, argv, mode, what, ret);
```

`JVMInit` 在 `java_md_solinux.c:806`，只有两行：

```c
int JVMInit(InvocationFunctions* ifn, jlong threadStackSize,
            int argc, char **argv, int mode, char *what, int ret) {
    ShowSplashScreen();
    return ContinueInNewThread(ifn, threadStackSize, argc, argv, mode, what, ret);
}
```

`ShowSplashScreen`：如果有 `-splash` 参数，在一个独立窗口显示启动图片——JVM 启动完成后由 JVM 关闭它。日常不用管。

`ContinueInNewThread` 在 `java.c:2321`，准备 JavaMainArgs 后调平台层的线程创建函数：

```c
int ContinueInNewThread(InvocationFunctions* ifn, jlong threadStackSize,
                        int argc, char **argv, int mode, char *what, int ret) {
    if (threadStackSize == 0) {
        struct JDK1_1InitArgs args1_1;
        memset((void*)&args1_1, 0, sizeof(args1_1));
        args1_1.version = JNI_VERSION_1_1;
        ifn->GetDefaultJavaVMInitArgs(&args1_1);    // 用 JNI 1.1 协议查 VM 默认栈大小
        if (args1_1.javaStackSize > 0) {
            threadStackSize = args1_1.javaStackSize;
        }
    }

    JavaMainArgs args;
    args.argc = argc;
    args.argv = argv;
    args.mode = mode;
    args.what = what;
    args.ifn = *ifn;              // 把三个函数指针一起打包

    int rslt = CallJavaMainInNewThread(threadStackSize, (void*)&args);
    return (ret != 0) ? ret : rslt;
}
```

如果用户没指定 `-Xss`，用 JNI 1.1 协议向 JVM 查询默认栈大小。然后把所有参数打包进 `JavaMainArgs`，调用平台层的 `CallJavaMainInNewThread`：

```c
int CallJavaMainInNewThread(jlong stack_size, void* args) {
    int rslt;
    pthread_t tid;
    pthread_attr_t attr;
    pthread_attr_init(&attr);
    pthread_attr_setdetachstate(&attr, PTHREAD_CREATE_JOINABLE);

    if (stack_size > 0) {
        pthread_attr_setstacksize(&attr, stack_size);
    }
    pthread_attr_setguardsize(&attr, 0);    // Java 线程有自己的 guard page

    if (pthread_create(&tid, &attr, ThreadJavaMain, args) == 0) {
        void* tmp;
        pthread_join(tid, &tmp);            // 主线程阻塞等待 Java 线程结束
        rslt = (int)(intptr_t)tmp;
    } else {
        rslt = JavaMain(args);              // fallback：创建线程失败，当前线程继续
    }

    pthread_attr_destroy(&attr);
    return rslt;
}

// ThreadJavaMain —— pthread 签名适配器
static void* ThreadJavaMain(void* args) {
    return (void*)(intptr_t)JavaMain(args);
}
```

**为什么要开新线程？** `java.c:203-205` 的注释给了官方理由：`"Running Java code in primordial thread caused many problems."` 原始线程（调 `main()` 的那个）的栈大小由 OS 初始分配、信号处理器已绑定——这些都不可改。JVM 需要精确控制栈大小和 guard pages，这些只能在 `pthread_create` 时通过 `pthread_attr` 设定。JDK-6316197 记录了早期在主线程启动 JVM 导致的 StackGuardPages 冲突 bug。

`pthread_attr_setdetachstate(PTHREAD_CREATE_JOINABLE)` 保证 `pthread_join` 能等待 Java 线程结束。`pthread_attr_setguardsize(&attr, 0)` 关掉 pthread 自带的 4KB guard page——Java 线程有自己的 StackGuardPages（mprotect 映射），两个 guards 会打架。

如果 `pthread_create` 失败（内存耗尽），不走 `exit()`，而是 fallback 到当前线程直接调 `JavaMain`——大概率也会失败，但至少比静默崩溃好。

`JVMInit` 不会返回——`ContinueInNewThread` 里的 `pthread_join` 等到 `JavaMain` 执行完毕后，最终调 `exit()`。控制权永远不会回到 `JLI_Launch` 的调用者。

## 本章总结

从你敲 `java -version`，到函数指针 `ifn->CreateJavaVM` 指向 `jni.cpp` 里的 `JNI_CreateJavaVM`，代码走完了三件事：

1. **准备参数**（main.c 150 行）：把命令行拆成 margv（用户的）和 jargv（编译期的），Linux 分支上构建参数数组，最后用硬编码覆盖。
2. **找到 JVM**（CreateExecutionEnvironment 六步）：`/proc/self/exe` → 回退找到 JDK 根目录 → 读 jvm.cfg → 拼出 libjvm.so 的绝对路径。
3. **打开 JVM 的门**（LoadJavaVM）：`dlopen` 加载 164MB 的 libjvm.so，`dlsym` 三个函数指针。

```c
// 三个关键文件，四个核心调用
main.c:      main() → JLI_Launch(margc, margv, ...)
java.c:      JLI_Launch() → CreateExecutionEnvironment() + LoadJavaVM() + JVMInit()
java_md_solinux.c:  LoadJavaVM() → dlopen(jvmpath) + dlsym × 3
```
