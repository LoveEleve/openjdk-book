# ch10 写作规范 v1.0

## 核心原则

1. **从第一性原理出发**：每个概念从"如果没有它，JVM 会遇到什么问题"开始，而不是从类定义开始。
2. **每个设计决策都有 WHY**：不满足于"这个字段是 xxx"，必须追问"为什么需要这个字段""为什么不可以用别的方式"。
3. **自底向上建概念**：使用新概念前必须先解释它。如果 OopStorage 涉及 CAS，必须先解释 CAS 是什么、JVM 中哪里用到了它。
4. **自己定义基础概念**：需要用到 Placement New、mmap、NX bit、CAS、RCU 等 OS/CPU 层面的概念，在首次出现时必须解释——不假设读者都知道。
5. **禁止"流水账"**：不允许按函数调用顺序罗列（"首先调 X，然后调 Y，最后调 Z"）。只按知识点依赖关系组织——一个概念的 WHY 在讲清楚之前不跳到下一个概念。

## 写作结构（每篇文章必须遵守）

### 开头（3-5 行）
- 一句话说清楚本文在整个知识体系中的位置
- 一句话说清楚读者学完后能理解什么
- 一句话说清楚需要什么前置知识

### 第 0 节：源码清单
- 一个表格：文件 | 行号 | 关键词

### 正文：按图层组织（不是按函数调用顺序）

每篇文章分成 3-5 个 Layer，从底层往上：

```
Layer N:  为什么需要这个机制？（设计动机）
Layer N+1: 核心数据结构（最底层实体）
Layer N+2: 操作/算法（怎么分配/释放/查找）
Layer N+3: 对接上层（JVM 哪个子系统用它，解决什么问题）
Layer N+4: 并发/安全（线程安全设计，GC 协作）
```

**每层必须回答的问题**：
- 没有这层之前，上层是什么状态？
- 这层解决了什么问题？
- 具体怎么解决的？（贴源码，≤5 行）
- 为什么选择了这个方案而不是别的？

### 禁止事项
- 按函数调用顺序叙述（如"接着调用了 xxx，然后 yyy"）
- 说"前面已经讲过"但不回顾关键结论（每篇文章独立可读）
- Unicode 箭头/圈号/框线图
- "可能""似乎""大概"等模糊词
- "老规矩""说白了""你懂的"等口语
- 超过 5 行的代码块（必须拆成多段 + 解释）

### 长度要求
- 每个"最小知识点"至少 30-50 行的独立展开
- 每篇文章至少 400 行，目标 450-550 行
- 参考 ch09/03-bufferblob-create.md（520 行）作为标杆
