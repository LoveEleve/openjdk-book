# jdk11u-copy：裁剪、CMake 与 IDE

上一章的 Make 构建教会了你如何从源码编译 JDK。但这不够——你改一行 HotSpot 源码后，需要等 1 分 31 秒才能验证。`jdk11u-copy` 把这个循环缩短到 10 秒。

本章讲清楚这个项目做了什么——它不是一个简单的副本，而是从 jdk11u 基础上系统性重构出的成果：先裁出一个服务器专用 JDK，再给它配上 CMake 构建系统，最后用 CLion 实现秒级增量开发和源码跳转。

---

## 一、为什么要做 jdk11u-copy

### Make 的五个痛点

Ch0.3 讲了 Make 构建的完整流水线——12 行 Makefile wrapper -> Init.gmk 两次自调用 -> Main.gmk 10 层构建。这个过程你完整走通了，但你可能也察觉到了它的体验问题：

1. **IDE 不友好**。CLion / VSCode 无法直接理解 GNU Make 的宏模板。Make 用 `SetupJdkLibrary` -> `SetupNativeCompilation` 这种多层宏展开链动态生成编译规则，IDE 看不到这条链，也就无法提供代码跳转、自动补全、重构。打开 `src/hotspot/` 时满屏红色报错——不是因为代码有 bug，而是 IDE 不知道这些文件会被编成什么 target。

2. **错误信息晦涩**。Make 报错时的输出是宏展开后的原始命令，夹杂着 50 个 `-D` 宏、十几个 `-I` 路径、层层 `$(call ...)` 的结果。真正的错误信息（比如"这个头文件找不到"）埋在一行 3000 字符的命令中间，你要手动把输出分成多行才能找到。

3. **增量编译判断不够精确**。Make 的依赖扫描是文本时间戳级别——你改了一个头文件里的一行注释，make 会认为所有 include 过它的 `.cpp` 都要重编。Ch0.3 的 `abstractCompiler.o.cmdline` 里有个 `-MMD` 选项——Make 用它生成 `.d` 文件记录依赖关系，但这里的依赖判断是"头文件最近被 touch 了 -> 重编"，不是"头文件的内容变了 -> 重编"。touch 以下某个顶层头文件，几百个 .o 全重来。

4. **跨平台迁移成本高**。每增加一个新平台（比如 AIX、BSD），需要新写一套 `os_platform.m4` 和对应的 `.gmk` 文件。Make 的规则模板是手写的平台适配层——不是"加一个配置文件"就能搞定的事。

5. **并行编译控制弱**。Make 的 `-j` 只管进程级并行——它把 `.cpp` 分给 32 个 worker 进程各自编译，但它不知道哪个 `.o` 是哪个 `.so` 的前置依赖。而 CMake + Ninja 的图调度可以精确到"先编完这个 target 的前置依赖，再启动这个 target"——减少等待，避免"所有 worker 都在等某个热路径的 .h 编完"的瓶颈。

这 5 个痛点撑起了从 Make 迁移到 CMake 的技术理由。但还有一个更根本的问题——改了代码之后怎么最快验证。

### 改了 HotSpot 源码，怎么最快验证

这是 CMake 迁移的最核心动机。Ch0.3 揭示了 Make 的工作流程：`.so` 先编译到 `support/modules_libs/`，最后才 cp 到 `jdk/lib/`。Make 的流程是两步：

```
改了 g1CollectedHeap.cpp
  -> make hotspot JOBS=32（遍历所有 .cpp 判断哪些 .o 需要重编）
    -> 重新链接 libjvm.so 到 support/modules_libs/
      -> make exploded-image（cp libjvm.so 到 jdk/lib/server/）
        -> java -version（验证）
```

即使只重编一个 `.o`，`make hotspot` 也要重新遍历 928 个 `.cpp` 检查依赖——"改了一个文件 -> 重新判断全部文件要不要重编"是 Make 依赖管理的固有开销。加上重新链接 928 个 `.o` 的时间，整个循环约 1 分 30 秒。如果你只想测试一个 G1 参数调整，改动到验证的等待就是 1 分半。

CMake 的 `jvm` target 只需要 10 秒。

为什么能这么快？因为 libjvm.so 是通过 `dlopen` 在运行时加载的——这是 Ch0.3 第五节"加载链"揭示的关键机制。`bin/java` 启动器自身不链接 libjvm（它的 NE需要 只有 libz + libjli + libc），libjli 用 `dlopen("server/libjvm.so")` 在运行时加载 JVM。这意味着：

- 替换 `jdk/lib/server/libjvm.so` 文件后，JVM 下次启动时自动加载新版——不需要重新链接启动器
- 不需要重新生成 JNI 头文件（javac -h）——只要你不改 Java native 方法签名，JNI 头文件不变
- 不需要重新编译 Java 类——`java.lang.String.class` 和 `libjvm.so` 是独立产物

CMake 利用了这一点：把 `.so` 的输出目录直接指向 `jdk/lib/`，编完就能跑。Make 做不到这种粒度，因为它输出到 `support/modules_libs/` 再 cp——两步流转。所以 CMake 不是"Make 的替代品"，而是"改了 C++ 源码后最快验证的通道"。

```
CMake 流程：

改了 g1CollectedHeap.cpp
  -> cmake --build cmake-build --target jvm（只重编修改的 .o，~10 秒）
    -> libjvm.so 直接更新到 jdk/lib/server/
      -> java -version（验证）
```

改了哪个 `.cpp`，只编它一个 `.o`，链接时只替换这一个 `.o`，然后立即运行。1 分 30 秒变成 10 秒——这就是 CMake + dlopen 的核心价值。

同样的逻辑适用于所有 22 个 `.so`——`libjava.so`、`libnet.so`、`libzip.so` 等都是通过 `dlopen`、`System.loadLibrary` 或 JNI 动态加载的。改 `libjava/System.c` -> `cmake --build --target java` -> `java -version`，一样秒级。不是只有 libjvm.so 有这个能力。

## 二、裁剪：从源码森林到服务器专用 JDK

jdk11u 的源码树包含 82 个 src/ 条目——全 JDK 模块、多个 CPU 架构、多个操作系统适配层。jdk11u-copy 把它裁成了一个只在 Linux x86_64 上跑的服务器 JDK。

### 模块裁剪（82->41）

删掉了 41 个模块，按使用场景分组：

**桌面/图形 UI（2 个）**：`java.desktop`（AWT、Swing、字体、声音——服务器不需要 GUI）、`java.datatransfer`（剪贴板——依赖桌面）

**桌面辅助（3 个）**：`jdk.accessibility`（无障碍辅助）、`jdk.unsupported.desktop`、`jdk.jconsole`（JMX 监控 GUI）

**RMI 相关（3 个）**：`java.rmi`、`java.management.rmi`、`jdk.naming.rmi`——远程方法调用，现代服务用 gRPC/HTTP 替代

**脚本/Nashorn（4 个）**：`java.scripting`、`jdk.scripting.nashorn`（JavaScript 引擎，JDK 15+ 已移除）、`jdk.scripting.nashorn.shell`（jjs 命令行）、`jdk.dynalink`（动态语言链接）

**AOT / 编译器 / JVMCI（4 个）**：`jdk.aot`（AOT 预编译）、`jdk.internal.vm.compiler`（Graal——用 Java 写的 JIT 编译器，整个 JVMCI 生态链被移除）、`jdk.internal.vm.compiler.management`、`jdk.internal.vm.ci`（JVM Compiler Interface——Graal 的入口）

**已废弃/已移除功能（8 个）**：`jdk.pack`（pack200）、`jdk.rmic`（RMI 编译器）、`jdk.xml.dom`、`jdk.internal.ed`、`jdk.internal.le`、`jdk.editpad`、`jdk.jsobject`、`jdk.sctp`（SCTP 协议）

**调试/开发工具（3 个）**：`jdk.jdi`（调试器接口）、`jdk.jdwp.agent`（JPDA 调试代理——线上调试需要它，Arthas 类工具不依赖它）、`jdk.jshell`（Java REPL）

**开发工具（2 个）**：`jdk.javadoc`、`jdk.jstatd`

**平台限定（2 个）**：`jdk.crypto.mscapi`（Windows）、`jdk.crypto.ucrypto`（Solaris）——这两个在 Linux 上本来就不编，Ch0.3 第四节提过

**其他（2 个）**：`java.smartcardio`（智能卡 IO）、`java.se`（聚合模块——各子模块独立存在后不需要它）

**被清理的非代码目录（5 个）**：`bin/`（预构建工具）、`doc/`、`test/`（测试）、`sample/`、`demo/`、`bsd/`、`solaris/`、`utils/`

保留了 41 个模块——包括所有服务器运行的基石：`java.base`、加密（`jdk.crypto.ec`、`jdk.crypto.cryptoki`）、网络（`java.net.http`）、诊断（`jdk.jcmd`/`jps`/`jstack`/`jmap`/`jfr`/`jdk.hotspot.agent`）、管理（`java.management`、JMX）、压缩（`jdk.zipfs`）等。

### GC 裁剪（8->1，只留 G1）

删掉了除 G1 外的全部 GC 引擎：

```
删除 CMS (Concurrent Mark Sweep)    -> JDK 14 已正式移除
删除 Parallel Scavenge               -> 服务器已默认 G1
删除 Serial GC                       -> 桌面和嵌入式场景
删除 ZGC                             -> 需要彩色指针支持（JDK 15+）
删除 Epsilon GC                      -> 测试用 No-op GC
删除 Shenandoah                      -> 非标准，Red Hat 维护
```

对应的，`macros.hpp` 中 8 个 `INCLUDE_*` 宏从 `1` 改为 `0`：

```cpp
#define INCLUDE_CMSGC 0
#define INCLUDE_EPSILONGC 0
#define INCLUDE_PARALLELGC 0
#define INCLUDE_SERIALGC 0
#define INCLUDE_SHENANDOAHGC 0
#define INCLUDE_ZGC 0
#define INCLUDE_JVMCI 0
#define INCLUDE_AOT 0
```

其余 64 个 HotSpot 源文件加了 `#if INCLUDE_*` 条件编译守卫——比如之前在 `sharedRuntime.cpp` 里无条件 `#include "gc/cms/concurrentMarkSweepGeneration.hpp"`，现在变成：

```cpp
#if INCLUDE_CMSGC
#include "gc/cms/concurrentMarkSweepGeneration.hpp"
#endif
```

GC 目录也直接删了——`src/hotspot/share/gc/cms/`、`parallel/`、`serial/`、`z/`、`epsilon/`、`shenandoah/` 全部移除。`src/hotspot/share/jvmci/`（JVMCI）和 `src/hotspot/share/aot/` 同样移除。

### 平台裁剪（-> 仅 linux_x86_64）

| 维度 | jdk11u | jdk11u-copy |
|------|--------|-------------|
| CPU 架构 | 7 (aarch64, arm, ppc, s390, sparc, x86, zero) | 1 (x86) |
| 操作系统 | 6 (aix, bsd, linux, posix, solaris, windows) | 2 (linux, posix) |
| OS+CPU 组合 | 15 | 1 (linux_x86) |

`src/hotspot/cpu/` 下删了 aarch64、arm、ppc、s390、sparc、zero 六个目录。`src/hotspot/os/` 下删了 aix、bsd、solaris、windows 四个目录。`src/hotspot/os_cpu/` 下只保留 `linux_x86/`。`src/hotspot/cpu/x86/gc/` 下还删了 `shenandoah/` 和 `z/` 两个 GC 特定子目录。

## 三、CMake 迁移

### 双轨制设计

CMake 不是 Make 的全面替代——它**只接管 native（C/C++/ASM）编译**。Java 编译、JNI 头文件生成、ADLC/JFR/JVMTI 代码生成仍然走 Make。

构建流程：

```
bash configure
  -> make JOBS=32（生成 Java class + JNI 头文件，输出到 support/headers/）
    -> bash cmake/spec2cmake.sh build/*/spec.gmk
      -> cmake -B cmake-build -C build/*/cmake_config.cmake -DCMAKE_BUILD_TYPE=Debug
        -> cmake --build cmake-build -j32
```

Make 负责第一段——产生 JNI 头文件、生成代码、编译 Java 类。这些产出被 CMake 作为 `-I` 路径引入。spec2cmake.sh 把 Make configure 探测出的编译参数（编译器路径、CFLAGS、系统库路径）转换成 CMake 变量。

### spec2cmake.sh：桥接 Make 和 CMake

`cmake/spec2cmake.sh` 只有 101 行。它的工作很直接——从 `build/*/spec.gmk` 中 grep 关键变量，然后生成为 `set()` 命令到 `cmake_config.cmake`：

```
CC=/usr/bin/gcc
CXX=/usr/bin/g++
JDK_C_FLAGS=-std=c99 -fPIC -fstack-protector ...
JVM_CXX_FLAGS=-std=gnu++98 -fno-rtti -fno-exceptions -fvisibility=hidden ...
JVM_LIBS=-lm -ldl -lpthread
```

根 CMakeLists.txt 第 5-8 行加载这个文件：

```cmake
set(CMAKE_CONFIG_FILE "${CMAKE_SOURCE_DIR}/build/.../cmake_config.cmake")
if(EXISTS "${CMAKE_CONFIG_FILE}")
    include("${CMAKE_CONFIG_FILE}")
endif()
```

### 19 个 CMakeLists.txt 的组织

根 CMakeLists.txt 把构建分成 3 个阶段 + HotSpot 前置：

```
Phase 0 — HotSpot JVM（必须最先编，被后续 .so 依赖）
  add_subdirectory(src/hotspot)                    -> libjvm.so (~795 .cpp, G1 only)

Phase 1 — java.base 独立库
  add_subdirectory(src/java.base/share/native/libfdlibm)  -> libfdlibm.a
  add_subdirectory(src/java.base/share/native/libverify)   -> libverify.so
  add_subdirectory(src/java.base/unix/native/libjsig)     -> libjsig.so
  add_subdirectory(src/java.base)                          -> libjli/java/zip/jimage/net/nio

Phase 4 — 其他 JDK native 库（11 个模块）
  add_subdirectory(src/jdk.attach/native)          -> libattach.so
  add_subdirectory(src/java.instrument/native)     -> libinstrument.so
  add_subdirectory(src/java.management/native)     -> libmanagement.so
  add_subdirectory(src/jdk.management/native)      -> libmanagement_ext.so
  add_subdirectory(src/jdk.management.agent/native) -> libmanagement_agent.so
  add_subdirectory(src/jdk.crypto.cryptoki/native)  -> libj2pkcs11.so
  add_subdirectory(src/jdk.crypto.ec/native)       -> libsunec.so
  add_subdirectory(src/jdk.hotspot.agent/native)   -> libsaproc.so
  add_subdirectory(src/jdk.net/native)             -> libextnet.so
  add_subdirectory(src/jdk.security.auth/native)   -> libjaas.so
  add_subdirectory(src/java.security.jgss/native)  -> libj2gss.so
  add_subdirectory(src/java.prefs/native)          -> libprefs.so

Phase 5 — 启动器
  add_subdirectory(launcher)                       -> 19 个 ELF 可执行文件
```

为什么是 Phase 0/1/4/5？因为 Phase 2（GENDATA）和 Phase 3（COPY）在 Make 侧做了，CMake 不需要对应。`launcher/` 是手工创建的目录——`src/` 里没有单独放启动器的位置，`java.base/share/native/launcher/main.c` 是唯一源文件。

### 产物直接入镜像

根 CMakeLists.txt 第 24-26 行设置全局输出路径：

```cmake
set(CMAKE_LIBRARY_OUTPUT_DIRECTORY "${JDK_IMAGE}/lib")
set(CMAKE_ARCHIVE_OUTPUT_DIRECTORY "${JDK_IMAGE}/lib")
set(CMAKE_RUNTIME_OUTPUT_DIRECTORY "${JDK_IMAGE}/bin")
```

所有 .so 直接输出到 `jdk/lib/`，所有启动器直接输出到 `jdk/bin/`。例外只有两个：libjvm.so 通过 `src/hotspot/CMakeLists.txt` 单独指定输出到 `jdk/lib/server/`，libjli.so 通过 `src/java.base/CMakeLists.txt` 单独指定输出到 `jdk/lib/jli/`（对应 RPATH `$ORIGIN/../lib/jli`）。

这和第一节讲的核心价值一致——编完就能跑。Make 的 `support/modules_libs/` 工作区在 CMake 里不存在。

## 四、CMake 实现审查：对照 Make 的 10 项检查

经过 Ch0.3 对 Make 构建全流程的深入理解，重新审查 CMake 的实现是否完整：

### 已实现的 8 项

| # | 检查项 | CMake 的实现 |
|---|--------|------------|
| 1 | abstract_vm_version.cpp 的 VERSION_* 宏 | `src/hotspot/CMakeLists.txt:135-138`——`set_source_files_properties` 覆盖 20+ 个宏 |
| 2 | mapfile 版本脚本 | `src/hotspot/CMakeLists.txt:128`——`-Wl,--version-script=jvm_sym.ver`，手动编写静态版本 |
| 3 | libjava 链接 jvm + verify | `src/java.base/CMakeLists.txt:43`——`target_link_libraries(java PRIVATE jvm verify fdlibm dl pthread)` |
| 4 | libjli.so 输出到 jdk/lib/jli/ 子目录 | `src/java.base/CMakeLists.txt:14-16`——`LIBRARY_OUTPUT_DIRECTORY .../jdk/lib/jli` |
| 5 | fdlibm 静态链接 | `libfdlibm/CMakeLists.txt:7`——`add_library(fdlibm STATIC ...)`，libjava 链接它 |
| 6 | -fvisibility=hidden | `src/hotspot/CMakeLists.txt:113` |
| 7 | -fno-rtti / -fno-exceptions | `src/hotspot/CMakeLists.txt:111` |
| 8 | INCLUDE_* 条件编译宏 | `src/hotspot/CMakeLists.txt:39-46`——全部 GC 精准映射，G1=1 其余=0 |

### 需要修复的 2 项

**Launcher RPATH 缺第二条路径**。`launcher/CMakeLists.txt:36` 只设了 `INSTALL_RPATH "$ORIGIN/../lib/jli"`，但 Make 有两个 RPATH：

```make
ORIGIN_ARG := $(call SET_EXECUTABLE_ORIGIN,/../lib/jli)
ORIGIN_ARG += $(call SET_EXECUTABLE_ORIGIN,/../lib)
```

第二个 `$ORIGIN/../lib` 是给启动器加载 JNI 库（如 libjava.so）用的。不带这个路径，启动器在某些 JNI 加载场景下可能找不到 .so。修复：

```cmake
INSTALL_RPATH "$ORIGIN/../lib/jli:$ORIGIN/../lib"
```

**java 启动器缺 JAVA_ARGS 宏**。Make 中 java 启动器有 `-DJAVA_ARGS={ "-J-ms8m", }`（初始堆 8MB）。CMake 的 `launcher/CMakeLists.txt` 中 java 启动器不传 MAIN_CLASS，因此完全不设置 JAVA_ARGS。因为定义了只有 1 个 `-J` 前缀的参数，`defines.h` 计算出的 `main_jargc` 始终为 0，所以功能上等价——但严格一致是更好的实践。

## 五、源码改动

### main.c 的调试代码

`src/java.base/share/native/launcher/main.c` 中加了一段 CLion 调试用的代码（18 行）。当 `const_progname == "java"` 时，把 `margv` 硬编码为 `{"java", "-version", NULL}`，等价于用户敲了 `java -version`。这样在 CLion 里 Debug 运行不用每次手动配 Run Configuration 参数。这段代码在实际启动器中不会触发（只有 `const_progname` 是 "java" 时才进入），且 `-version` 对应的行为是打印版本信息后退出，不影响正常使用。

### jvm_sym.ver 符号版本脚本

Make 构建用 `JvmMapfile.gmk` 动态从 `.o` 文件中提取符号生成 mapfile。但 GNU 链接器 `ld` 不识别 Solaris mapfile 的 `SUNWprivate_X.X` 版本语法——传给它时静默忽略整个版本块，退化为导出全部符号。这导致 `operator new` 被全局导出，和 `libjimage.so` 的 `operator new` 冲突，运行时崩溃。

修复方案是创建 `src/hotspot/jvm_sym.ver`——GNU 版本的符号版本脚本：

```
SUNWprivate_1.1 {
  global:
    JNI_*;
    JVM_*;
    jio_*;
    AsyncGetCallTrace;
  local:
    *;
};
```

CMake 链接时通过 `-Wl,--version-script=${CMAKE_CURRENT_SOURCE_DIR}/jvm_sym.ver` 加载它。配合 `-fvisibility=hidden` 编译选项，确保 libjvm.so 只导出白名单中的符号。

### 65 个 HotSpot 文件的 #if INCLUDE_* 守卫

GC 裁剪（删了除 G1 外的所有 GC source 目录）后，剩下 65 个 HotSpot 文件中有对被删 GC 的 `#include` 引用——`classFileParser.cpp` 里 `#include "gc/cms/cmsArguments.hpp"` 这种代码在删掉 cms/ 目录后编译会报"找不到头文件"。

修复方式是加 `#if INCLUDE_* ... #endif` 守卫。`macros.hpp` 中 8 个 `INCLUDE_*` 宏从 `1` 改为 `0`，编译期 `-D` 展开这些宏后，预处理器自动跳过对应的 `#include`。64 个文件分布在 `classfile/`（6）、`code/`（2）、`compiler/`（2）、`gc/`（12）、`jfr/`（3）、`memory/`（2）、`oops/`（2）、`opto/`（10）、`prims/`（4）、`runtime/`（8+）、`utilities/`（1）。

## 七、CLion 集成

### 打开项目

CLion 打开 `/data/workspace/jdk11u-copy/` 时自动识别根目录的 `CMakeLists.txt`。CMake configure 完成后，Project 面板里会出现 22 个库 target（`jvm`、`java`、`jli`、`zip`...）和 19 个启动器 target（`java_launcher`、`javac_launcher`...）。代码跳转开箱即用——`goto definition` 直接进入 HotSpot 源码。

前置条件：必须先完成一次 Make 构建（`bash configure && make JOBS=32`），确保 `build/*/support/headers/` 下有 JNI 头文件，`cmake_config.cmake` 已由 `spec2cmake.sh` 生成。

### Run Configuration

CLion 创建 CMake Application 配置：

| 字段 | 值 |
|------|-----|
| Target | `java_launcher` |
| Executable | `build/linux-x86_64-normal-server-slowdebug/jdk/bin/java` |
| Program arguments | `-version` |

常用调试配置：

| 配置名 | Program arguments |
|--------|-------------------|
| java -version | `-version` |
| java -XX:+PrintGCDetails | `-XX:+PrintGCDetails -version` |
| java -cp /tmp HW | `-cp /tmp HW` |
| javac HelloWorld | Target 改用 `javac_launcher`，参数填 `/tmp/HelloWorld.java` |
