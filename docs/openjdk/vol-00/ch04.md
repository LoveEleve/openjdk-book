# jdk11u-copy

上一章你完整走通了 `jdk11u` 的 Make 构建——12 行 Makefile wrapper -> Init.gmk 两次自调用 -> Main.gmk 10 层构建。1 分 31 秒。那条管线是 OpenJDK 从 1990 年代沿用至今的标准构建方式。

本章的 `jdk11u-copy` 不是 jdk11u 的一个"副本"。它是一个独立的项目——从 jdk11u 分叉出来后，做了三件事：

1. **裁剪**：从完整的 OpenJDK 源码树裁出一个服务器专用子集
2. **CMake 迁移**：给 native 代码（22 个 .so + 19 个启动器）加上 CMake 构建
3. **CLion 集成**：用 CMake 让 CLion 原生索引源码，代码跳转、增量编译、断点调试开箱即用

整章按这个顺序展开。先看两个项目的物理差异，再讲每件事为什么做、怎么做的。

---

## 两个项目的目录结构

`jdk11u` 是从 OpenJDK 11u 官方仓库 `clone` 的干净源码。Ch0.2 用它做了 configure + make，Ch0.3 拆解了 make 的全过程。它的目录结构是标准的 OpenJDK 布局：

```
/data/workspace/jdk11u/

  configure         <- autoconf 生成的配置脚本（90KB+）
  make/             <- 全部 Makefile：Init.gmk（364 行）、Main.gmk（1307 行）、
                       Help.gmk、ModuleWrapper.gmk、以及 71 个模块的 Lib/Launcher .gmk
  src/              <- 源码目录（82 个条目）
    hotspot/        928 .cpp -> libjvm.so（8 个 GC 引擎、多 CPU/OS 适配）
    java.base/      3043 .java（跨平台 + unix/ + linux/ 三层优先级）
    java.desktop/   2786 .java（AWT/Swing/2D/字体）
    jdk.compiler/   398 .java（javac 本体）
    ...             共 73 个模块目录 + bsd/linux/solaris/demo/sample/utils
  build/            <- Make 构建产物目录（~1 GB）
    linux-x86_64-normal-server-slowdebug/
      spec.gmk          configure 探测结果
      hotspot/          HotSpot 中间产物（928 .o + gtest 版）
      support/          工作区（gensrc/ + headers/ + modules_libs/ + modules_cmds/）
      jdk/              最终镜像（33 启动器 + 38 .so + 71 模块）
  test/             <- jtreg 测试（hotspot/jdk/langtools/nashorn/jaxp 五个测试集）

  bin/              <- 预构建二进制工具
```

`jdk11u-copy` 是在 jdk11u 上做了 `cp -r` 副本后，**删了不该有的、加了需要的、改了必须改的**：

```

  configure         <- 无改动

  make/             <- 改了 8 个文件 + 删了 33 个模块定义文件

    删除：            Launcher-*.gmk（14 个删除模块的启动器定义）
                      Lib-*.gmk（12 个删除模块的 native 库定义）
                      make/data/* 下 7 个子目录 + devkit/ + ide/ + nb_native/ + test/ + vscode/

  launcher/         <- 新增：启动器 CMakeLists.txt（98 行，19 个 target）
  cmake-build/      <- 新增：CMake 输出（Makefile 生成器，1056 .o 文件，0 .so）
  cmake-build-debug/ <- 新增：CMake 输出（Ninja 生成器，空）

    jdk.attach/     CMakeLists.txt 已注入（6 行）
    jdk.crypto.ec/  CMakeLists.txt 已注入（5 行）

    hotspot/jvm_sym.ver    新增：符号版本脚本（GNU ld 兼容）

  build/            <- Make 构建产物（和 jdk11u 的 build/ 结构相同）
    linux-x86_64-normal-server-slowdebug/
```

## Make 和 CMake 在这个项目中的分工

jdk11u-copy 的构建是双轨制。Make 没有被替代——它管着整个构建流程的上半部分。CMake 专门管 native 编译的下半部分。

### Make 做的事

Make 在原构建系统中负责 8 个阶段（Ch0.3 完整拆解过）。jdk11u-copy 的 Make 保留了这个能力，但因为模块裁剪，编译量少了很多：

```
configure -> make JOBS=32

  阶段一：造工具
    -> 用 bootstrap JDK 编译 interim javac
    -> 编译 ADLC 工具（13 .cpp -> adlc 二进制）
    -> 编译 JVMTI 生成器、JFR 生成器

  阶段二：生成代码
    -> ADLC 从 .ad 文件生成 ad_x86.hpp/ad_x86.cpp/dfa_x86.cpp 等 11 个文件
    -> JFR 从 metadata.xml 生成 jfrEventClasses.hpp 等 5 个文件
    -> JVMTI 从 jvmti.xml 生成 jvmtiEnter.cpp 等 6 个文件
    -> DTrace 从 hotspot.d 生成 hotspot.h 等 3 个文件
    -> JDK 侧 gensrc 生成 2323 个 .java

  阶段三：编译 Java 类
    -> 用 interim javac 编译 41 个模块的 .java -> .class
    -> javac -h 生成 JNI 头文件支持后续 C++ 编译

  阶段四：编译 HotSpot（这个阶段 CMake 也能做）
  阶段五：编译原生库 + 启动器（这个阶段 CMake 也能做）
  阶段六：组装镜像
  阶段七：release 文件
```

### CMake 做的事

CMake 只接管了阶段四和阶段五中的 native 编译部分——22 个 `.so` + 1 个 `.a` + 19 个启动器。它的构建流程在 Make 完成上半部分后才启动：

```
make JOBS=32 完成后

  -> bash cmake/spec2cmake.sh build/*/spec.gmk
     （从 Make configure 产物 spec.gmk 中提取编译参数 -> cmake_config.cmake）

    -> cmake -B cmake-build -C build/*/cmake_config.cmake -DCMAKE_BUILD_TYPE=Debug
       （配置 CMake 项目，加载 Make 探测出的编译器路径、CFLAGS、系统库）

      -> cmake --build cmake-build -j32
         （编译 22 .so + 1 .a + 19 启动器，产物直接输出到 jdk/lib/ 和 jdk/bin/）
```

### 为什么不是 CMake 替代 Make

CMake 无法替代 Make 的上半部分，因为它依赖 Make 的三类产出：

1. **JNI 头文件**——`javac -h` 从已编译的 `.class` 中生成 `java_lang_Object.h` 等头文件。这些输出到 `build/*/support/headers/`，CMakeLists.txt 中多处 `-I` 路径指向这里。没有 Make 编译 Java 类，就没有 JNI 头文件，native 代码编译会报"找不到 JNI 函数声明"。

2. **生成代码**——ADLC/JFR/JVMTI 是 Make 编译和执行的工具（ADLC 是 C++ 写的、JFR/JVMTI 是 Java 写的用 bootstrap JDK 跑）。它们产出的 `ad_x86.cpp`、`jvmtiEnter.cpp` 等文件被 CMake 编进 `libjvm.so`。没有 Make 先跑代码生成，CMake 编译 HotSpot 时会缺少这些源文件。

3. **编译参数**——`spec.gmk` 是 Make configure 的产物，记录了 GCC 路径、`-D` 宏、`-std` 版本、`-f` 优化标志、系统库路径。`spec2cmake.sh` 从这里提取参数生成 `cmake_config.cmake`。CMake 不需要自己做平台探测——直接复用 Make configure 的结果。

**更重要的原因是个迭代流程**：改了 HotSpot 源码后，重新验证的步骤是：

| 方式 | 操作 | 时间 |
|------|------|------|
| Make | `make hotspot JOBS=32`（遍历全部依赖）-> 重新链接 -> `make exploded-image`（复制 .so） | ~1:30 |
| CMake | `cmake --build cmake-build --target jvm`（只编改动的 .o，产物已在 jdk/lib/） | ~10s |

Make 需要两步才能把 .so 送到 jdk/ 目录：先输出到 `support/modules_libs/`，然后 `make exploded-image` 时 cp 过去。CMake 直接输出到 `jdk/lib/`，编完就能跑。

**为什么 CMake 能直接输出到 jdk/lib/ 而 Make 不能**？因为 .so 是通过 `dlopen` 在运行时动态加载的。`bin/java` 启动器不链接 libjvm（它的 NEEDED 只有 libz + libjli + libc），libjli 用 `dlopen("server/libjvm.so")` 在运行时加载 JVM。所以替换 `jdk/lib/server/libjvm.so` 文件后，JVM 下次启动自然加载新版——不需要重新链接启动器、不需要重新生成 JNI 头文件。

这个原理对 22 个 .so 全部适用——`libjava.so`、`libnet.so`、`libzip.so` 等都是通过 `dlopen` 或 `System.loadLibrary` 动态加载的。改 `libjava/System.c` -> `cmake --build --target java` -> `java -version`，秒级验证。

## 裁剪

jdk11u 的 `src/` 包含全 JDK 的源码——从服务器核心到桌面 GUI，从现代 GC 到已废弃的实验性收集器。jdk11u-copy 做了三层面的裁剪。

### 模块裁剪

`src/` 从 82 个条目裁到 41 个模块。删除了六个类别的模块：

**桌面部**：`java.desktop`（AWT/Swing/2D/字体/声音——服务器没有显示器）、`java.datatransfer`（剪贴板——依赖桌面）、`jdk.accessibility`（无障碍辅助）、`jdk.unsupported.desktop`（桌面内部 API）、`jdk.jconsole`（JMX 监控 GUI）。这组是最大的裁剪——`java.desktop` 一个模块就有 2786 个 .java 文件。

**RMI**：`java.rmi`（远程方法调用）、`java.management.rmi`（JMX RMI 连接器）、`jdk.naming.rmi`（RMI 命名服务）。现代服务间通信用 gRPC/HTTP/REST，RMI 在 2026 年的服务器 JDK 上几乎不被使用。

**Nashorn 脚本引擎**：`java.scripting`（javax.script API）、`jdk.scripting.nashorn`（JavaScript 引擎，593 个 .java）、`jdk.scripting.nashorn.shell`（jjs 命令行）、`jdk.dynalink`（动态语言链接）。JDK 15 已经正式移除了 Nashorn，11u 里保留它是兼容性的考虑——在服务器场景下不需要。

**AOT + Graal + JVMCI**：`jdk.aot`（AOT 预编译，jaotc 工具）、`jdk.internal.vm.compiler`（Graal 编译器，用 Java 写的 JIT，1587 个 .java）、`jdk.internal.vm.compiler.management`（Graal 管理）、`jdk.internal.vm.ci`（JVM Compiler Interface——Graal 的入口）。这四者的关系是：JVMCI 是 HotSpot 暴露给外部编译器的接口，Graal 是用 Java 写的 JIT 编译器通过 JVMCI 接入 HotSpot，AOT 用 Graal 做预编译。删 Graal 后 JVMCI 没有消费者，所以整体移除。

**已废弃/已移除功能**：`jdk.pack`（pack200 压缩工具）、`jdk.rmic`（RMI 编译器）、`jdk.xml.dom`（XML DOM API）、`jdk.internal.ed`、`jdk.internal.le`、`jdk.editpad`、`jdk.jsobject`、`jdk.sctp`（SCTP 协议支持）。

**调试/开发工具**：`jdk.jdi`（JDI 调试器接口——IDE 调试时需要，Arthas 用 JVMTI/Instrumentation 不走 JDI）、`jdk.jdwp.agent`（JDWP 调试线协议代理——同上）、`jdk.jshell`（Java REPL）、`jdk.javadoc`（文档生成器）、`jdk.jstatd`（远程 jstat 守护进程——可以用 JMX 替代）。

**平台限定模块**：`jdk.crypto.mscapi`（Windows CryptoAPI，只有 `windows/` 目录没有 `share/`——Linux 上本来就不会编译，Ch0.3 第四节详述过）、`jdk.crypto.ucrypto`（Solaris Ucrypto 硬件加速——同样纯平台模块）。

**其他**：`java.smartcardio`（智能卡 IO）、`java.se`（聚合模块——各子模块独立存在后不需要它）。

**被删除的非模块目录**：`bin/`（预构建二进制工具）、`doc/`（文档）、`test/`（五个测试集——约 200MB 的测试代码和配置）、`sample/`、`demo/`、`bsd/`、`solaris/`、`utils/`、`README.md`、`SECURITY.md`。

### GC 裁剪：8 个减少到 1 个

jdk11u 编译 libjvm.so 时同时编入了 8 个垃圾回收器（Ch0.3 第四节 `Building JVM variant 'server' with features` 那一行里列了 22 个 feature，其中 GC 相关的占了 8 个）：

```
CMS (Concurrent Mark Sweep)      -> JDK 14 已正式移除
Parallel Scavenge                -> 专注于吞吐量，服务器场景 G1 已经满足
Serial GC                        -> 桌面和嵌入式场景
ZGC                             -> 需要彩色指针，JDK 15+ 才成熟
Epsilon GC                      -> No-op GC，测试用
Shenandoah                      -> Red Hat 维护的非标准 GC
G1 GC                           -> 保留——JDK 9+ 的默认 GC，生产验证充分
```

裁剪不只是不编译这些 GC——**源码目录被物理删除了**。`src/hotspot/share/gc/` 下只剩 `g1/` 和 `shared/`：

```
$ ls src/hotspot/share/gc/
g1/          <- 保留
shared/      <- 保留（所有 GC 的公共基础设施）
```

删除的目录：
```
src/hotspot/share/gc/cms/         <- concurrent mark sweep（已废弃）
src/hotspot/share/gc/parallel/     <- parallel scavenge
src/hotspot/share/gc/serial/       <- serial gc
src/hotspot/share/gc/z/            <- ZGC（60 个 .cpp）
src/hotspot/share/gc/epsilon/      <- epsilon（5 个 .cpp）
src/hotspot/share/gc/shenandoah/   <- shenandoah
src/hotspot/share/jvmci/           <- JVM 编译器接口（JVMCI）
src/hotspot/share/aot/             <- AOT 预编译
```

源码目录删除后，剩下 65 个 HotSpot 源文件中有对被删 GC 的 `#include` 引用——比如 `classFileParser.cpp` 里无条件 `#include "gc/cms/cmsArguments.hpp"`，但 `cms/` 目录已经不存在了，编译会报错。

修复方式是加 `#if INCLUDE_* ... #endif` 条件编译守卫。对应的 `macros.hpp` 中 8 个宏从 `1` 改为 `0`：

```cpp
#define INCLUDE_CMSGC 0
#define INCLUDE_EPSILONGC 0
#define INCLUDE_PARALLELGC 0
#define INCLUDE_SERIALGC 0
#define INCLUDE_SHENANDOAHGC 0
#define INCLUDE_ZGC 0
#define INCLUDE_JVMCI 0
#define INCLUDE_AOT 0
```

配合 CMakeLists.txt 中同步设置的 `-D` 编译选项：

```cmake
target_compile_definitions(jvm PRIVATE
    INCLUDE_G1GC=1
    INCLUDE_CMSGC=0      INCLUDE_EPSILONGC=0    INCLUDE_PARALLELGC=0
    INCLUDE_SERIALGC=0   INCLUDE_SHENANDOAHGC=0  INCLUDE_ZGC=0
    INCLUDE_JVMCI=0      INCLUDE_AOT=0
    ...
)
```

### 平台裁剪

jdk11u 的 `src/hotspot/` 支持 7 个 CPU 架构 + 6 个操作系统 + 15 个 OS+CPU 组合。jdk11u-copy 只保留 Linux x86_64 一个：

| 维度 | jdk11u | jdk11u-copy |
|------|--------|-------------|
| CPU | aarch64, arm, ppc, s390, sparc, x86, zero | x86 |
| OS | aix, bsd, linux, posix, solaris, windows | linux, posix |
| OS+CPU | 15 个组合 | linux_x86 |

`src/hotspot/cpu/` 下删了 aarch64/arm/ppc/s390/sparc/zero 六个目录和对应汇编器/寄存器文件。`src/hotspot/os/` 删了 aix/bsd/solaris/windows。`src/hotspot/os_cpu/` 只留 `linux_x86/`。

`posix/` 被保留是因为它是 Linux 的公共基础层——信号处理、线程、文件 IO 等 POSIX 标准 API 的封装都在这里。

## CMake 项目的内部结构

### 根 CMakeLists.txt

94 行，是整个 CMake 构建的入口。按功能分四部分：

```cmake
cmake_minimum_required(VERSION 3.14)
project(OpenJDK11u LANGUAGES C CXX ASM)

# 第一部分：加载 Make configure 的产物
set(CMAKE_CONFIG_FILE "${CMAKE_SOURCE_DIR}/build/.../cmake_config.cmake")
include("${CMAKE_CONFIG_FILE}")     # 从 spec.gmk 提取的 GCC路径、CFLAGS、LIBS

# 第二部分：全局编译设置
set(CMAKE_POSITION_INDEPENDENT_CODE ON)
set(CMAKE_C_STANDARD 99)
set(CMAKE_CXX_STANDARD 98)
set(CMAKE_CXX_EXTENSIONS OFF)

# 第三部分：JNI 头文件路径（依赖 Make 的 javac -h 产物）
set(JNI_HEADERS_BASE "${JDK_BUILD_DIR}/support/headers")
set(JNI_INCLUDE_DIR "${CMAKE_SOURCE_DIR}/${JDK_BUILD_DIR}/support/modules_include/java.base")
set(JNI_INCLUDE_DIR_LINUX "${CMAKE_SOURCE_DIR}/${JDK_BUILD_DIR}/support/modules_include/java.base/linux")

# 第四部分：输出目录（.so -> jdk/lib/, 可执行文件 -> jdk/bin/）
set(JDK_IMAGE "${CMAKE_SOURCE_DIR}/build/linux-x86_64-normal-server-slowdebug/jdk")
set(CMAKE_LIBRARY_OUTPUT_DIRECTORY "${JDK_IMAGE}/lib")
set(CMAKE_ARCHIVE_OUTPUT_DIRECTORY "${JDK_IMAGE}/lib")
set(CMAKE_RUNTIME_OUTPUT_DIRECTORY "${JDK_IMAGE}/bin")

# 第五部分：构建阶段
# Phase 0 — HotSpot JVM（必须先编，被后续所有 .so 依赖）
add_subdirectory(src/hotspot)

# Phase 1 — java.base 独立库（无依赖，可并行）
add_subdirectory(src/java.base/share/native/libfdlibm)   # libfdlibm.a（静态数学库）
add_subdirectory(src/java.base/share/native/libverify)    # libverify.so（字节码验证）
add_subdirectory(src/java.base/unix/native/libjsig)       # libjsig.so（信号链）
add_subdirectory(src/java.base)                            # libjli/java/zip/jimage/net/nio

# Phase 4 — 其他 JDK native 库（11 个模块，依赖 libjvm + libjava）
add_subdirectory(src/jdk.attach/native)                   # libattach.so
add_subdirectory(src/java.instrument/native)              # libinstrument.so
add_subdirectory(src/java.management/native)              # libmanagement.so
add_subdirectory(src/jdk.management/native)               # libmanagement_ext.so
add_subdirectory(src/jdk.management.agent/native)         # libmanagement_agent.so
add_subdirectory(src/jdk.crypto.cryptoki/native)          # libj2pkcs11.so
add_subdirectory(src/jdk.crypto.ec/native)                # libsunec.so
add_subdirectory(src/jdk.hotspot.agent/native)            # libsaproc.so
add_subdirectory(src/jdk.net/native)                      # libextnet.so
add_subdirectory(src/jdk.security.auth/native)            # libjaas.so
add_subdirectory(src/java.security.jgss/native)           # libj2gss.so
add_subdirectory(src/java.prefs/native)                   # libprefs.so

# Phase 5 — 启动器（依赖 libjli.so）
add_subdirectory(launcher)
```

Phase 2（GENDATA）和 Phase 3（COPY）只在 Make 里存在——它们不是 native 编译，CMake 没有对应阶段。所以编号跳过了。

### src/hotspot/CMakeLists.txt

这个文件 162 行，是 19 个 CMakeLists.txt 里最大、最复杂的一个。它处理 libjvm.so 的编译——约 795 个 .cpp（裁剪后，原版 928 个）：

```cmake
# 收集源文件（六个层次的源码目录）
file(GLOB_RECURSE JVM_SOURCES
    ${HOTSPOT_SHARE}/*.cpp         # 平台无关核心（~600 文件）
    ${HOTSPOT_CPU}/*.cpp           # x86 汇编器/栈帧（~60 文件）
```

## CMake 实现审查

经过 Ch0.3 对 Make 全流程的深入理解，现在可以审查 CMake 对关键细节的处理。10 项检查：

1. abstract_vm_version.cpp 的版本宏——已实现。CMake 用 `set_source_files_properties` 为这个文件注入 20+ 个版本宏（`-DHOTSPOT_VERSION_STRING`、`-DVERSION_FEATURE=11`、`-DVERSION_UPDATE=32` 等），和 Make 的 `CFLAGS_VM_VERSION` 完全对应。952 个 .cpp 中只有这一个文件需要版本号——其他只需要 feature 宏。

2. mapfile 版本脚本——已实现。Make 用 `JvmMapfile.gmk` 动态从 .o 提取符号生成 mapfile。CMake 用静态手写的 `jvm_sym.ver`（56 行）。两者的白名单等价——都只导出 `JNI_*`、`JVM_*`、`jio_*`、`AsyncGetCallTrace`。

3. libjava 依赖链——已实现。`target_link_libraries(java PRIVATE jvm verify fdlibm dl pthread)`——和 Make 的 `-ljvm -lverify` 一致。CMake 额外显式链接了 `dl` 和 `pthread`（Make 中间接通过 configure 的 LDFLAGS 提供）。

4. libjli.so 输出到 jdk/lib/jli/ 子目录——已实现。`set_target_properties(jli PROPERTIES LIBRARY_OUTPUT_DIRECTORY ".../jdk/lib/jli")`。

5. fdlibm 静态链接——已实现。`add_library(fdlibm STATIC ...)`。libjava 通过 `target_link_libraries` 链接它。

6. -fvisibility=hidden——已实现。所有 hotspot .cpp 编译时带上这个选项。

7. -fno-rtti / -fno-exceptions——已实现。和 Make 的 HotSpot 编译选项一致。

8. INCLUDE_* 宏——已实现。G1=1，其余 7 个 GC + JVMCI + AOT = 0。

9. Launcher RPATH——遗漏。CMake 只设了 `INSTALL_RPATH "$ORIGIN/../lib/jli"`，Make 有两个 RPATH：第二个 `$ORIGIN/../lib` 是启动器加载 JNI 库时需要的。修复：`INSTALL_RPATH "$ORIGIN/../lib/jli:$ORIGIN/../lib"`。

10. java 启动器的 JAVA_ARGS 宏——遗漏。Make 中 java 启动器有 `-DJAVA_ARGS={ "-J-ms8m", }`（初始堆 8MB）。CMake 的 java 启动器没有传 MAIN_CLASS，完全不设 JAVA_ARGS。因为只有 1 个 `-J` 前缀参数，`defines.h` 计算出的 `main_jargc` 为 0，功能等价。但严格一致更好。

### jvm_sym.ver 符号版本脚本

Make 构建用 `JvmMapfile.gmk` 动态生成 mapfile。但 GNU 链接器 `ld` 不识别 Solaris mapfile 的 `SUNWprivate_X.X` 版本语法——传给它时静默忽略整个版本块，退化为导出全部符号。这导致 `operator new` 被全局导出，和 `libjimage.so` 的 `operator new` 冲突，运行时崩溃。

修复方案是创建 GNU 版符号版本脚本 `src/hotspot/jvm_sym.ver`：

```
SUNWprivate_1.1 {
  global:
    JNI_*;
    JVM_*;
    jio_*;
    AsyncGetCallTrace;
  local:
    *;
};
```

CMake 链接时用 `-Wl,--version-script=jvm_sym.ver` 加载它，配合 `-fvisibility=hidden` 编译选项。

## CLion 集成

### 打开项目

CLion 打开 `/data/workspace/jdk11u-copy/` 时自动识别根目录的 `CMakeLists.txt`。因为 `CXX_STANDARD=98` 且 `C_STANDARD=99`，CLion 自动配置正确的 C++ 标准和索引。

前置条件：**必须先完成一次 Make 构建**（`bash configure && make JOBS=32`），确保 `support/headers/` 下有 JNI 头文件，`cmake_config.cmake` 已由 `spec2cmake.sh` 生成。第一次运行 CMake configure 在 CLion 里需要 30-60 秒（扫描 795 个 .cpp 文件）。之后是增量。

configure 完成后，Project 面板出现 41 个 target（22 库 + 19 启动器）。双击任何一个 .cpp 文件，Goto Definition（Ctrl+B）直接跳转——无论是同一个 target 内部，还是跨 target（比如 `g1CollectedHeap.cpp` -> `CollectedHeap.hpp` -> `oop.inline.hpp`）。不像 Make 构建那样 CLion 满屏红线报错。

### Run Configuration

为 `java_launcher` 创建 CMake Application 配置：

| 字段 | 值 |
|------|-----|
| Target | `java_launcher` |
| Executable | `build/linux-x86_64-normal-server-slowdebug/jdk/bin/java` |
| Program arguments | `-version` |
| Working directory | `/data/workspace/jdk11u-copy` |

main.c 中的调试代码（当 const_progname == "java" 时自动设 `-version`）让这个配置可以留空参数——但手动配置支持更复杂的参数（如 `-XX:+PrintGCDetails`）。

### 增量编译

| 操作 | 方式 | 说明 |
|------|------|------|
| 编译 libjvm.so | 右上角 Build Configurations 选 `jvm` -> Build | 触发 CMake 增量编译，只处理修改过的 .cpp |
| 编译全体 | Ctrl+F9 | 等价于 `cmake --build cmake-build-debug` |
| Build + Debug | Shift+F9 | 触发增量编译后自动启动 GDB |

改 `g1CollectedHeap.cpp` 的一个函数体后，Build `jvm` 只编译这一个 .o -> 重新链接 libjvm.so（因为依赖图检测到 .o 内容变了）。整个过程 ~10 秒。Make 的同样操作需要 ~90 秒——因为它要重新遍历全部 928 个 .cpp 的依赖关系。
