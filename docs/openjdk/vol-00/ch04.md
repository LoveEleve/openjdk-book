# jdk11u-copy：裁剪、CMake 与 IDE

上一章的 Make 构建教会了你如何从源码编译 JDK。但这不够——你改一行 HotSpot 源码后，需要等 1 分 31 秒才能验证。`jdk11u-copy` 把这个循环缩短到 10 秒。

本章讲清楚这个项目做了什么——它不是一个简单的副本，而是从 jdk11u 基础上系统性重构出的成果：先裁出一个服务器专用 JDK，再给它配上 CMake 构建系统，最后用 CLion 实现秒级增量开发和源码跳转。

---

## 一、为什么要做 jdk11u-copy

### Make 的五个痛点

Ch0.3 讲了 Make 构建的完整流水线——12 行 Makefile wrapper -> Init.gmk 两次自调用 -> Main.gmk 10 层构建。这个过程你完整走通了，但你可能也察觉到了它的体验问题：

1. **IDE 不友好**。CLion / VSCode 无法直接理解 GNU Make 的宏模板。Make 用 `SetupJdkLibrary` -> `SetupNativeCompilation` 这种多层宏展开链动态生成编译规则，IDE 看不到这条链，也就无法提供代码跳转、自动补全、重构。打开 `src/hotspot/` 时满屏红色报错——不是因为代码有 bug，而是 IDE 不知道这些文件会被编成什么 target。

2. **错误信息晦涩**。Make 报错时的输出是宏展开后的原始命令，夹杂着 50 个 `-D` 宏、十几个 `-I` 路径、层层 `$(call ...)` 的结果。真正的错误信息（比如"这个头文件找不到"）埋在一行 3000 字符的命令中间，你要手动把输出分成多行才能找到。

3. **增量编译判断不够精确**。Make 的依赖扫描是文本时间戳级别——你改了一个头文件里的一行注释，make 会认为所有 include 过它的 `.cpp` 都要重编。Ch0.3 的 `abstractCompiler.o.cmdline` 里有个 `-MMD` 选项——Make 用它生成 `.d` 文件记录依赖关系，但这里的依赖判断是"头文件最近被 touch 了 -> 重编"，不是"头文件的内容变了 -> 重编"。touch 以下某个顶层头文件，几百个 .o 全重来。

4. **跨平台迁移成本高**。每增加一个新平台（比如 AIX、BSD），需要新写一套 `os_platform.m4` 和对应的 `.gmk` 文件。Make 的规则模板是手写的平台适配层——不是"加一个配置文件"就能搞定的事。

5. **并行编译控制弱**。Make 的 `-j` 只管进程级并行——它把 `.cpp` 分给 32 个 worker 进程各自编译，但它不知道哪个 `.o` 是哪个 `.so` 的前置依赖。而 CMake + Ninja 的图调度可以精确到"先编完这个 target 的前置依赖，再启动这个 target"——减少等待，避免"所有 worker 都在等某个热路径的 .h 编完"的瓶颈。

这 5 个痛点撑起了从 Make 迁移到 CMake 的技术理由。但还有一个更根本的问题——改了代码之后怎么最快验证。

### 改了 HotSpot 源码，怎么最快验证

这是 CMake 迁移的最核心动机。Ch0.3 揭示了 Make 的工作流程：`.so` 先编译到 `support/modules_libs/`，最后才 cp 到 `jdk/lib/`。Make 的流程是两步：

```
改了 g1CollectedHeap.cpp
  -> make hotspot JOBS=32（遍历所有 .cpp 判断哪些 .o 需要重编）
    -> 重新链接 libjvm.so 到 support/modules_libs/
      -> make exploded-image（cp libjvm.so 到 jdk/lib/server/）
        -> java -version（验证）
```

即使只重编一个 `.o`，`make hotspot` 也要重新遍历 928 个 `.cpp` 检查依赖——"改了一个文件 -> 重新判断全部文件要不要重编"是 Make 依赖管理的固有开销。加上重新链接 928 个 `.o` 的时间，整个循环约 1 分 30 秒。如果你只想测试一个 G1 参数调整，改动到验证的等待就是 1 分半。

CMake 的 `jvm` target 只需要 10 秒。

为什么能这么快？因为 libjvm.so 是通过 `dlopen` 在运行时加载的——这是 Ch0.3 第五节"加载链"揭示的关键机制。`bin/java` 启动器自身不链接 libjvm（它的 NE需要 只有 libz + libjli + libc），libjli 用 `dlopen("server/libjvm.so")` 在运行时加载 JVM。这意味着：

- 替换 `jdk/lib/server/libjvm.so` 文件后，JVM 下次启动时自动加载新版——不需要重新链接启动器
- 不需要重新生成 JNI 头文件（javac -h）——只要你不改 Java native 方法签名，JNI 头文件不变
- 不需要重新编译 Java 类——`java.lang.String.class` 和 `libjvm.so` 是独立产物

CMake 利用了这一点：把 `.so` 的输出目录直接指向 `jdk/lib/`，编完就能跑。Make 做不到这种粒度，因为它输出到 `support/modules_libs/` 再 cp——两步流转。所以 CMake 不是"Make 的替代品"，而是"改了 C++ 源码后最快验证的通道"。

```
CMake 流程：

改了 g1CollectedHeap.cpp
  -> cmake --build cmake-build --target jvm（只重编修改的 .o，~10 秒）
    -> libjvm.so 直接更新到 jdk/lib/server/
      -> java -version（验证）
```

改了哪个 `.cpp`，只编它一个 `.o`，链接时只替换这一个 `.o`，然后立即运行。1 分 30 秒变成 10 秒——这就是 CMake + dlopen 的核心价值。

同样的逻辑适用于所有 22 个 `.so`——`libjava.so`、`libnet.so`、`libzip.so` 等都是通过 `dlopen`、`System.loadLibrary` 或 JNI 动态加载的。改 `libjava/System.c` -> `cmake --build --target java` -> `java -version`，一样秒级。不是只有 libjvm.so 有这个能力。
