# jdk11u-copy

上一章你完整走通了 `jdk11u` 的 Make 构建——12 行 Makefile wrapper -> Init.gmk 两次自调用 -> Main.gmk 10 层构建。1 分 31 秒。

本章的 `jdk11u-copy` 不是 jdk11u 的一个"副本"。它是对原项目的系统性重构——裁出服务器必需的最小源码集，然后用 CMake 给 native 代码加上秒级增量编译。

在进入细节之前，先看一眼两个项目的差异。

## 两个项目的目录对比

`jdk11u`（原项目）的顶层：

```
/data/workspace/jdk11u/
├── Makefile         <- make 默认入口，include make/Init.gmk
├── configure        <- autoconf 生成的配置脚本
├── make/            <- 全部 Makefile（Init.gmk, Main.gmk, 所有 .gmk 模块文件）
├── src/             <- 源码目录（82 个条目：java.*/jdk.*/hotspot/bsd/linux/solaris/...）
├── build/           <- 产物目录（1.0GB，Make 构建产物）
├── test/            <- 测试
├── doc/             <- 文档
└── bin/             <- 预构建工具
```

`jdk11u-copy`（改造后）的顶层：

```
/data/workspace/jdk11u-copy/
├── Makefile         <- 无改动（和 jdk11u 完全相同）
├── configure        <- 无改动
├── make/            <- 改了 8 个 .gmk + 删了 33 个 launcher/lib 模块定义
├── CMakeLists.txt   <- 新增：根 CMake 配置（94 行）
├── cmake/           <- 新增：spec2cmake.sh
├── launcher/        <- 新增：启动器 CMakeLists.txt（19 个 target）
├── cmake-build/     <- 新增：CMake 构建目录（Makefile 生成器）
├── cmake-build-debug/ <- 新增：CMake 构建目录（Ninja 生成器）
├── .idea/           <- 新增：CLion 项目配置
├── src/             <- 41 个模块 + 19 个 CMakeLists.txt + 删了 41 个模块
├── build/           <- Make 构建产物 + cmake_config.cmake
├── src/hotspot/jvm_sym.ver  <- 新增：符号版本脚本
└── jdk.hotspot.agent.bak/   <- 新增：SA 模块备份
```

关键差异：多了 `CMakeLists.txt`（19 个）、`cmake/`、`launcher/`、`.idea/`、`jvm_sym.ver`。少了 `src/` 下 41 个模块、`test/`、`doc/`、`bin/`。`Makefile` 和 `configure` 没改——Make 构建系统还在，但被裁了 33 个模块定义文件。

## Make 和 CMake 的分工

Make 没有被替代。`jdk11u-copy` 的构建是双轨：

```
          configure
              │
      ┌───────┴───────┐
      ▼               ▼
   make/           cmake/
  (autoconf)    (spec2cmake.sh)
      │               │
      ▼               ▼
  编译 Java        编译 C/C++
  生成 JNI 头       22 .so + 19 启动器
  生成 ADLC/JFR/
  JVMTI 代码
```

**Make 做的事**（Ch0.3 的 8 阶段流水线，全保留）：

```
configure -> make JOBS=32
  -> 阶段一：造工具（interim javac + ADLC）
  -> 阶段二：生成代码（ADLC/JFR/JVMTI/DTrace）
  -> 阶段三：编译 Java 类（71 模块 -> .class）
  -> 阶段四：编译 HotSpot（928 .cpp -> libjvm.so）   <- 裁后 ~795 .cpp
  -> 阶段五：编译原生库 + 启动器（35 .so + 33 启动器）
  -> 阶段六：组装镜像（support/ -> jdk/）
  -> 阶段七：release 文件
```

**CMake 做的事**（新增，覆盖阶段四和五的 native 部分）：

```
make JOBS=32 完成后
  -> bash cmake/spec2cmake.sh build/*/spec.gmk
    -> cmake -B cmake-build -C build/*/cmake_config.cmake
      -> cmake --build cmake-build -j32

产出：
  22 个 .so（直接覆盖 jdk/lib/ 下 Make 编译的同名文件）
  19 个启动器（直接覆盖 jdk/bin/）
```

CMake 依赖 Make 的前置产出——JNI 头文件（`javac -h` 生成在 `support/headers/`）、代码生成文件（ADLC/JFR/JVMTI 生成在 `gensrc/`）。这些 `-I` 路径在 CMakeLists.txt 里写死了指向 `build/.../` 下的 Make 产出。

为什么需要 CMake 替代 Make 编译 native？因为改一个 HotSpot `.cpp` 后，重新验证的流程是：

| 方式 | 操作 | 耗时 |
|------|------|------|
| Make | `make hotspot JOBS=32` -> 遍历 928 个依赖 -> 重新链接 -> `make exploded-image` | ~1m30s |
| CMake | `cmake --build cmake-build --target jvm` -> 只编改动的 .o -> 产物直接在 jdk/lib/ | ~10s |

快的原因有两个：CMake 的依赖图比 Make 精确（知道了哪个 `.o` 需要重编而不是遍历全部），以及产物直接输出到 `jdk/lib/`（因为 `.so` 是通过 `dlopen` 运行时加载的，替换文件后 JVM 启动时自然加载新版——不需要重新链接启动器、不需要重新生成 JNI 头文件）。

这个原理对 22 个 .so 全部适用。改 `libjava/System.c` -> `cmake --build --target java` -> `java -version`，一样秒级。

## CMake 项目的内部架构

`jdk11u-copy` 有 19 个 CMakeLists.txt，负责 22 个 .so + 1 个 .a + 19 个启动器。

根 CMakeLists.txt（94 行）总览全局：

```cmake
# 加载 Make configure 的产物（spec2cmake.sh 从 spec.gmk 提取）
include("${CMAKE_SOURCE_DIR}/build/.../cmake_config.cmake")

# 全局输出路径——.so 直接到 jdk/lib/，启动器直接到 jdk/bin/
set(CMAKE_LIBRARY_OUTPUT_DIRECTORY  "${CMAKE_SOURCE_DIR}/build/.../jdk/lib")
set(CMAKE_RUNTIME_OUTPUT_DIRECTORY  "${CMAKE_SOURCE_DIR}/build/.../jdk/bin")

# JNI 头文件路径——指向 Make 生成的 support/headers/
set(JNI_HEADERS_BASE  "${JDK_BUILD_DIR}/support/headers")

# Phase 0: HotSpot（必须最先编，被后续 .so 依赖）
add_subdirectory(src/hotspot)                            -> libjvm.so

# Phase 1: java.base 独立库
add_subdirectory(src/java.base/share/native/libfdlibm)   -> libfdlibm.a
add_subdirectory(src/java.base/share/native/libverify)   -> libverify.so
add_subdirectory(src/java.base/unix/native/libjsig)      -> libjsig.so
add_subdirectory(src/java.base)                          -> libjava/jli/zip/jimage/net/nio

# Phase 4: 其他 JDK native 库（11 个模块）
add_subdirectory(src/java.instrument/native)             -> libinstrument.so
add_subdirectory(src/java.management/native)             -> libmanagement.so
add_subdirectory(src/jdk.management/native)              -> libmanagement_ext.so
add_subdirectory(src/jdk.management.agent/native)        -> libmanagement_agent.so
add_subdirectory(src/jdk.attach/native)                  -> libattach.so
add_subdirectory(src/jdk.crypto.cryptoki/native)         -> libj2pkcs11.so
add_subdirectory(src/jdk.crypto.ec/native)               -> libsunec.so
add_subdirectory(src/jdk.hotspot.agent/native)           -> libsaproc.so
add_subdirectory(src/jdk.net/native)                     -> libextnet.so
add_subdirectory(src/jdk.security.auth/native)           -> libjaas.so
add_subdirectory(src/java.security.jgss/native)          -> libj2gss.so
add_subdirectory(src/java.prefs/native)                  -> libprefs.so

# Phase 5: 启动器
add_subdirectory(launcher)                               -> 19 个 ELF
```

Phase 2（GENDATA）和 Phase 3（COPY）只存在于 Make，CMake 没有对应阶段。

`launcher/` 是手工创建的独立模块——`src/` 里没有集中放启动器的目录。19 个启动器的源码是同一个文件：`src/java.base/share/native/launcher/main.c`（Ch0.3 第五节详述过）。CMake 为每个启动器创建一个 target，通过 `-DPROGNAME`、`-DJAVA_ARGS` 等编译期宏区分行为。java 启动器的 JAVA_ARGS 是空的——不预设主类，用户从命令行传入。

`cmake/spec2cmake.sh`（101 行）是 Make 和 CMake 之间的桥接器——从 `build/*/spec.gmk` 中 grep 关键变量，生成为 `set()` 命令写入 `cmake_config.cmake`：

```
CC=/usr/bin/gcc
CXX=/usr/bin/g++
JDK_C_FLAGS=-std=c99 -fPIC ...
JVM_CXX_FLAGS=-std=gnu++98 -fno-rtti -fno-exceptions -fvisibility=hidden ...
JVM_LIBS=-lm -ldl -lpthread
```

## CMake 实现审查

经过 Ch0.3 对 Make 构建全流程的深入理解，对照审查 CMake 对关键细节的处理。10 项检查中 8 项正确实现，2 项遗漏：

| # | 检查项 | 状态 | 位置 |
|---|--------|------|------|
| VERSION_* 宏 | 已实现 | `src/hotspot/CMakeLists.txt:135`——抽象编译器版本宏全部覆盖 |
| jvm_sym.ver 符号版本脚本 | 已实现 | 第 128 行——手动编写 GNU 版本，配合 `-fvisibility=hidden` |
| libjava 链接 jvm + verify | 已实现 | `src/java.base/CMakeLists.txt:43` |
| libjli 输出到 jdk/lib/jli/ | 已实现 | 第 14-16 行——单独子目录，对应 RPATH |
| fdlibm 静态链接 | 已实现 | `libfdlibm/CMakeLists.txt:7`——`add_library(fdlibm STATIC ...)` |
| -fvisibility=hidden | 已实现 | `src/hotspot/CMakeLists.txt:113` |
| -fno-rtti / -fno-exceptions | 已实现 | 第 111 行 |
| INCLUDE_* 编译宏 | 已实现 | 第 39-46 行——G1=1 其余=0 |
| **Launcher RPATH** | **遗漏** | 只设了 `$ORIGIN/../lib/jli`，缺 `$ORIGIN/../lib` |
| **java 启动器 JAVA_ARGS** | **遗漏** | 没有设 `-DJAVA_ARGS={ "-J-ms8m", }`（功能等价，但和 Make 不一致） |

遗漏的两项修复很简单：RPATH 加第二路径，JAVA_ARGS 加一行 `-D` 注入。

## 源码改动

除了增删文件和 CMakeLists.txt，实际修改过的源码文件有三类：

**main.c**：加了 18 行 CLion 调试代码。当 `const_progname == "java"` 时硬编码 `margv = {"java", "-version", NULL}`，免去每次 CLion Debug 时手动配 Run Configuration 参数。

**jvm_sym.ver**：新增的符号版本脚本。GNU 链接器 `ld` 不识别 Solaris mapfile 的 `SUNWprivate_X.X` 语法，导致 `operator new` 被全局导出、和 libjimage.so 冲突崩溃。这个文件用 GNU 语法重写了导出白名单。

**65 个 HotSpot 源文件**：加 `#if INCLUDE_* ... #endif` 守卫。删掉 CMS/Parallel/Serial/Z 等 GC 的源码目录后，这 65 个文件中对被删 GC 的 `#include` 引用不加守卫就会编译报错。

## CLion 集成

CLion 打开 `jdk11u-copy/` 时自动识别 `CMakeLists.txt`，CMake configure 后 Project 面板出现 41 个 target（22 库 + 19 启动器）。代码跳转开箱即用。

前置条件：先完成一次 Make 构建（`bash configure && make JOBS=32`），确保有 JNI 头文件。

| 操作 | 方式 | 耗时 |
|------|------|------|
| 编译 libjvm.so | 选 `jvm` target -> Build | ~10 秒 |
| Debug | `Shift+F9`，Run Configuration 选 `java_launcher` | GDB 自动 attach |

典型断点位置：
```
Threads::create_vm()           -- JVM 初始化总入口
JNI_CreateJavaVM()             -- libjli 通过 dlsym 调用的函数
G1CollectedHeap::initialize()  -- G1 GC 初始化
```

下一章 Ch0.5 用这套环境 GDB 追踪 java 命令的完整启动过程——从 main.c 到 JLI_Launch，到 dlopen libjvm，到 JNI_CreateJavaVM 再到 Threads::create_vm。
