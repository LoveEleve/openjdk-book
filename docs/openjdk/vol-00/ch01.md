# Ch0.1 java 命令到底是什么

## java 到底是个啥

你天天敲 `java -version`。想过没——`java` 这个命令本身，到底是个什么东西？

先看你肯定认识的一个 C 程序：

```c
#include <stdio.h>

int main() {
    printf("hello world\n");
    return 0;
}
```

编译，换个名字，跑起来：

```
$ gcc hello.c -o myjava
$ ./myjava
hello world
$ file /tmp/myjava
/tmp/myjava: ELF 64-bit LSB executable, x86-64, version 1 (SYSV), dynamically linked, interpreter /lib64/ld-linux-x86-64.so.2, BuildID[sha1]=cfba5628..., for GNU/Linux 3.2.0, not stripped
```

叫 `myjava` 还是 `java` 还是 `abc`，无所谓。它就是 `gcc` 从一个 `.c` 文件编译出来的可执行文件。

你机器上刚编译出来的 `java` 也一样：

```
$ file build/linux-x86_64-normal-server-slowdebug/jdk/bin/java
build/linux-x86_64-normal-server-slowdebug/jdk/bin/java: ELF 64-bit LSB executable, x86-64, version 1 (SYSV), dynamically linked, interpreter /lib64/ld-linux-x86-64.so.2, BuildID[sha1]=f4adc92e..., for GNU/Linux 3.2.0, with debug_info, not stripped
```

一样的格式——ELF 64-bit LSB executable, x86-64, dynamically linked。区别只有两个：`BuildID` 不一样（不同编译产物当然不一样），以及 `java` 多了 `with debug_info`（因为编的是 slowdebug 版本）。

当然，真正的 `java` 命令比你写的 `myjava` 复杂得多。它的 `main()` 在 `src/java.base/share/native/launcher/main.c`，最后一行调用的是 `JLI_Launch()`，把活交给了一个叫 `libjli.so` 的共享库。编译也不是手动敲 `gcc`，而是 Makefile 驱动的——在 `make/launcher/Launcher-java.base.gmk` 里定义了几个编译宏和链接库，最终输出到 `build/.../jdk/bin/java`。

但骨架没变：一个 `.c` 文件，gcc 编译，得到可执行文件。这个认知是后面所有章节的基础——JVM 不是从天上掉下来的运行时，它就是从 C 代码编译出来的东西，你能读它，能改它，能用 GDB 停住它。

## git clone 下来的代码长什么样

`git clone git@github.com:openjdk/jdk11u.git` 之后，你得到这么一个目录：

```
jdk11u/
├── src/            全部源码（80+ 个模块 + JVM 本体）
├── make/           编译脚本（Makefile）
├── configure       构建配置脚本（先跑它）
├── test/           测试
├── doc/            文档
├── bin/            编译辅助脚本
└── README.md
```

`build/` 目录不存在——编译后才会生成，编译产物全在里面。

### src/hotspot/ —— JVM 本体

`src/hotspot/` 就是 JVM 的 C++ 源码。四层结构：

```
src/hotspot/
├── share/           平台无关代码（本书 95% 的分析在这里，25 个子目录）
│   ├── gc/          GC 实现（g1/、shared/、cms/、parallel/ 等）
│   ├── runtime/     线程、锁、Safepoint
│   ├── oops/        对象模型（oop/Klass/markOop）
│   ├── interpreter/ 模板解释器
│   ├── c1/          C1 JIT 编译器
│   ├── opto/        C2 JIT 编译器
│   ├── classfile/   .class 文件解析
│   ├── prims/       JNI/JVMTI/Unsafe
│   ├── memory/      元空间/堆抽象
│   ├── services/    JMX/DCmd/Attach
│   ├── compiler/    CompileBroker（编译调度）
│   ├── code/        CodeCache/nmethod
│   ├── jfr/         Java Flight Recorder
│   ├── jvmci/       Graal JIT 接口
│   ├── aot/         AOT 编译（提前编译）
│   ├── asm/         汇编器辅助
│   ├── ci/          Compiler Interface（编译器视角的 JVM 接口）
│   ├── logging/     统一日志框架
│   ├── utilities/   内部工具类
│   └── ...          adlc/libsdt/metaprogramming/precompiled
├── os/              平台适配（linux/、bsd/、solaris/ 等）
├── cpu/             CPU 架构（x86/、aarch64/ 等）
└── os_cpu/          平台+架构组合（linux_x86/ 等）
```

这么多目录不用慌——不是要全记住。每卷开头会告诉你这卷读哪个目录。现在只需要一个印象：**JVM 的核心代码全在 `src/hotspot/share/` 下面**。

### src/ —— 80+ 个 Java 模块

`src/` 下除了 `hotspot`，还有 80+ 个 Java 模块目录。大部分本书不碰——`java.desktop`（AWT/Swing）、`java.security.jgss`（Kerberos）、`jdk.crypto.ec`（加密）这些和 JVM 源码没关系。

会真正用到的就这几个：

`java.base` 是头号重点。它编译出来的是 libjava.so、libjli.so、libverify.so、libzip.so、libjimage.so、libnet.so、libnio.so 这一批。从 JNI 桥梁、启动器、字节码验证、类加载、Socket 到 NIO Selector——全在 java.base 的 native 层。这也是为什么整本书有一卷专门讲 java.base 的 JNI 实现。

然后是几个独立的小模块：`java.instrument`（Agent 插桩，编出来是 libinstrument.so）、`jdk.attach`（libattach.so，让 jcmd 能连上正在跑的 JVM）、`jdk.hotspot.agent`（libsaproc.so，事后调试工具）。这仨各有各的卷。

纯 Java 实现的也有几个：`jdk.jcmd`（jcmd/jps/jstack 这些命令）、`jdk.jfr`（飞行记录器）、`jdk.management`（JMX）、`jdk.unsupported`（Unsafe/Cleaner）。在讲对应主题的章节里会展开，但不用提前记。

### make/ —— 编译脚本

`make/` 下面是一堆 `.gmk` 文件（GNU Make 的宏定义），管整条编译流水线。你不用全懂——后面章节用到时再展开。现在只需要知道：

1. 先跑 `bash configure --with-debug-level=slowdebug ...`。这个脚本检测你的系统环境——GCC 版本、头文件路径、系统调用是否存在。跑完在根目录生成一个 `build/` 配置目录
2. 再跑 `make`，用这个配置把 `src/` 下的源码编译链接成 JDK

编译细节后面有单独一章讲。现在脑子里有个概念就行：`configure` 是体检，`make` 是动手。

## clone 下来的代码和解压即用的 JDK 有什么区别

装 JDK 最常见的姿势是下载二进制包：

```
$ wget https://.../openjdk-11.0.17_linux-x64_bin.tar.gz
$ tar xzf openjdk-11.0.17_linux-x64_bin.tar.gz
$ ./jdk-11.0.17/bin/java -version
openjdk version "11.0.17" ...
```

解压就能用——里面已经有一个编译好的 `java` 可执行文件。

而上面 `git clone` 下来的只有源码。想跑起 `java`？先 `bash configure`，再 `make`，等几分钟编译完。

一个是成品，一个是原材料。本书用的是后者——我们要读的就是这些 `.c`/`.cpp`/`.hpp`，看它们是怎么一步一步变成 164MB 的 `libjvm.so` 和 23KB 的 `java` 命令的。
