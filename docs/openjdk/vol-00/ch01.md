# Ch0.1 java 命令到底是什么

## java 到底是个啥

你天天敲 `java -version`。想过没——`java` 这个命令本身，到底是个什么东西？

先看你肯定认识的一个 C 程序：

```c
#include <stdio.h>

int main() {
    printf("hello world\n");
    return 0;
}
```

编译，换个名字，跑起来：

```
$ gcc hello.c -o myjava
$ ./myjava
hello world
$ file /tmp/myjava
/tmp/myjava: ELF 64-bit LSB executable, x86-64, version 1 (SYSV), dynamically linked, interpreter /lib64/ld-linux-x86-64.so.2, BuildID[sha1]=cfba5628..., for GNU/Linux 3.2.0, not stripped
```

叫 `myjava` 还是 `java` 还是 `abc`，无所谓。它就是 `gcc` 从一个 `.c` 文件编译出来的可执行文件。

你机器上刚编译出来的 `java` 也一样：

```
$ file build/linux-x86_64-normal-server-slowdebug/jdk/bin/java
build/linux-x86_64-normal-server-slowdebug/jdk/bin/java: ELF 64-bit LSB executable, x86-64, version 1 (SYSV), dynamically linked, interpreter /lib64/ld-linux-x86-64.so.2, BuildID[sha1]=f4adc92e..., for GNU/Linux 3.2.0, with debug_info, not stripped
```

一样的格式——ELF 64-bit LSB executable, x86-64, dynamically linked。区别只有两个：`BuildID` 不一样（不同编译产物当然不一样），以及 `java` 多了 `with debug_info`（因为编的是 slowdebug 版本）。

当然，真正的 `java` 命令比你写的 `myjava` 复杂得多。它的 `main()` 在 `src/java.base/share/native/launcher/main.c`，最后一行调用的是 `JLI_Launch()`，把活交给了一个叫 `libjli.so` 的共享库。编译也不是手动敲 `gcc`，而是 Makefile 驱动的——在 `make/launcher/Launcher-java.base.gmk` 里定义了几个编译宏和链接库，最终输出到 `build/.../jdk/bin/java`。

但骨架没变：一个 `.c` 文件，gcc 编译，得到可执行文件。这个认知是后面所有章节的基础——JVM 不是从天上掉下来的运行时，它就是从 C 代码编译出来的东西，你能读它，能改它，能用 GDB 停住它。

## git clone 下来的代码长什么样

`git clone git@github.com:openjdk/jdk11u.git` 之后，你得到这么一个目录：

```
jdk11u/
├── src/          ← 全部源码
├── make/         ← 编译脚本（Makefile）
├── configure     ← 构建配置脚本
├── CMakeLists.txt ← jdk11u-copy 自研的 CMake 构建（配合 CLion 用）
└── cmake/        ← CMake 辅助脚本
```

### src/ —— 源码的主场

`src/` 下面有 42 个目录。其中 41 个是 JDK 的模块，1 个是 `hotspot`（JVM 本体）。按「和本书的关系」分四类：

**JVM 本体（本书核心，全覆盖）**

```
src/hotspot/
├── share/          ← 平台无关代码，本书 95% 的分析在这里
│   ├── gc/         ← GC 实现（g1/、shared/ 等）
│   ├── runtime/    ← 线程、锁、Safepoint
│   ├── oops/       ← 对象模型（oop/Klass）
│   ├── interpreter/← 模板解释器
│   ├── c1/         ← C1 JIT 编译器
│   ├── opto/       ← C2 JIT 编译器
│   ├── classfile/  ← .class 文件解析
│   ├── prims/      ← JNI/JVMTI/Unsafe
│   ├── memory/     ← 元空间/堆抽象
│   ├── services/   ← JMX/DCmd/Attach
│   ├── compiler/   ← CompileBroker（编译调度）
│   ├── code/       ← CodeCache/nmethod 管理
│   ├── jfr/        ← Java Flight Recorder
│   ├── logging/    ← 统一日志框架
│   ├── utilities/  ← 内部工具类
│   └── ...         ← 其他（asm/ci/libadt/metaprogramming/precompiled/Xusage.txt）
├── os/             ← 平台适配（linux/ 等）
├── cpu/            ← CPU 架构（x86/ 等）
└── os_cpu/         ← 平台+架构组合（linux_x86/ 等）
```

本书 13 卷按这个目录树展开：`gc/` → GC 卷，`runtime/` → 线程与锁卷，`oops/` → 对象模型卷。现在不用记住——每卷开头会告诉你读哪个目录。

**Java 核心库（本书部分覆盖 native 层）**

```
src/java.base/     ← String/Object/Thread/ClassLoader 等（native 层）
```

其他模块大部分是纯 Java 实现，本书不深入，但要知道它们存在：

```
src/java.compiler/   src/java.logging/    src/java.naming/
src/jdk.compiler/    src/jdk.jcmd/        src/jdk.jfr/
src/jdk.attach/      src/jdk.unsupported/ ...
```

总共 41 个模块，不是 41 个全部要读。本书聚焦热点模块——后面每卷开头会说明覆盖了哪些。

**编译器和工具**

```
src/java.compiler/   ← javax.tools 接口
src/jdk.compiler/    ← javac 实现（纯 Java）
src/jdk.jcmd/        ← jcmd/jps/jstack 等诊断命令
src/jdk.jfr/         ← JFR 事件定义
src/jdk.hotspot.agent/ ← Serviceability Agent（事后调试）
```

**安全 / 加密 / GUI（本书不覆盖）**

```  
src/java.security.jgss/  src/jdk.crypto.ec/   ...
```

这些模块和 JVM 源码理解关系不大，忽略即可。

### make/ —— 怎么把这堆源码变成能跑的 JDK

`make/` 下面是 Makefile 脚本，管整个编译流程。你不用全看懂——后面章节用到时再展开。现在只需要知道两件事：

1. `configure` 脚本先跑，检测你的系统环境（GCC 版本、头文件路径、系统调用），生成 `build/` 配置
2. `make` 拿这个配置，把 `src/` 下的 `.c`/`.cpp`/`.java` 编译链接成最终的 JDK

### configure 和 CMakeLists.txt

`configure` 是 OpenJDK 原生的构建入口：`bash configure --with-debug-level=slowdebug ...` 跑完才能编译。

`CMakeLists.txt` 和 `cmake/` 是 jdk11u-copy 自研的，方便用 CLion 打开项目、跳转代码。本书推荐日常用 CMake + CLion（快），但全量编译走 Make（稳）。

## clone 下来的代码和解压即用的 JDK 有什么区别

装 JDK 最常见的姿势是下载二进制包：

```
$ wget https://.../openjdk-11.0.17_linux-x64_bin.tar.gz
$ tar xzf openjdk-11.0.17_linux-x64_bin.tar.gz
$ ./jdk-11.0.17/bin/java -version
openjdk version "11.0.17" ...
```

解压就能用——里面已经有一个编译好的 `java` 可执行文件。

而上面 `git clone` 下来的只有源码。想跑起 `java`？先 `bash configure`，再 `make`，等几分钟编译完。

一个是成品，一个是原材料。本书用的是后者——我们要读的就是这些 `.c`/`.cpp`/`.hpp`，看它们是怎么一步一步变成 164MB 的 `libjvm.so` 和 23KB 的 `java` 命令的。
