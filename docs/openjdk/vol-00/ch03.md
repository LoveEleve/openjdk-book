# make 到底做了什么

上一章的最后，你跑了 `make JOBS=32 LOG=debug`，1 分 31 秒后 `build/` 目录下多了一个 311MB 的 JDK。

本章逐项拆开这 1 分 31 秒：make 按什么顺序做了什么、每个阶段产出了什么、那些产物和你的 `src/` 源码之间是什么关系。

---

## 1 分 31 秒里发生了什么

先回顾上一章那条命令：

```
$ cd /data/workspace/jdk11u
$ make JOBS=32 LOG=debug
```

`make` 后面没跟任何目标。那它默认做了什么？

### 你敲 make，make 读的是哪个文件

`make` 后面没跟任何目标。make 的默认行为是找当前目录下名为 `Makefile`（或 `makefile`、`GNUmakefile`）的文件。你根目录下确实有一个 `Makefile`——但它不是真正的构建逻辑。文件头注释自己说得很清楚：

```
###
### This file is just a very small wrapper needed to run the real make/Init.gmk.
### It also performs some sanity checks on make.
###
```

这个 `Makefile` 做三件事：

1. **检查是不是 GNU Make**——`.TEST_FOR_NON_GNUMAKE:sh=echo You are not using GNU Make/gmake...`，Solaris 的 `/usr/bin/make` 会执行这行报错。GNU Make 直接跳过。
2. **检查 make 版本**——`ifeq ($(strip $(foreach v, 3.81% 3.82% 4.%,$(filter $v,$(MAKE_VERSION)))),)` 要求版本 ≥ 3.81。
3. 算出 `topdir` 后 `include $(topdir)/make/Init.gmk`——把控制权交给 `make/Init.gmk`。

根目录 Makefile 全文就 12 行有效代码，没有任何构建规则——它只是个**门卫 + 跳板**。

### Init.gmk：为什么要被调用两次

`make/Init.gmk` 有 364 行，是整个构建的调度中心。但它有一个反直觉的设计——**同一个文件被 make 调用了两次**，走完全不同的代码分支。为什么要这样？

核心矛盾是：**Init.gmk 需要 spec.gmk 里的变量才能工作，但 spec.gmk 的路径它一开始不知道**。

spec.gmk 是 configure 产出的，路径是 `build/linux-x86_64-normal-server-slowdebug/spec.gmk`——这个 `linux-x86_64-normal-server-slowdebug` 是你 configure 时选的配置名，不同配置名路径不同。Init.gmk 不能把这个路径写死，必须运行时去 `build/*/` 下扫描找到它。

但找到了还不够——Init.gmk 想用 spec.gmk 里的变量（比如 `JOBS`、`SPEC`、`TOPDIR`），必须先 `include` 它。而 include 之前，make 的变量作用域里还没有这些值。所以解法是：

1. **第一次调用**：不 include spec.gmk。只扫描 `build/*/` 找到 spec.gmk 的路径，然后**带上这个路径重新调用自己**。
2. **第二次调用**：这次知道 spec.gmk 在哪了，`include $(SPEC)` 把变量加载进来，然后真正开始工作。

Init.gmk 用一个叫 `HAS_SPEC` 的变量区分这两次调用——第一次它是空的，第二次它是 `true`。

#### 第一次调用：找 SPEC

根目录 Makefile `include` Init.gmk 时，`HAS_SPEC` 是空的。Init.gmk:42 检测到这一点，走第一分支。这个分支做三件事：

1. **扫描测试组**——日志开头就是 5 次 `FindJtregGroups` 调用：

```
make[1]: Entering directory '/data/workspace/jdk11u'
FindJtregGroups /data/workspace/jdk11u/test/hotspot/jtreg
+ /usr/bin/sed -n -e 's/\[\(.*\)\]/\1/g' -e 's/^groups\w*=//p' .../test/hotspot/jtreg/TEST.ROOT
+ /usr/bin/sed -n -e 's/^\#.*//g' -e 's/\([^ ]*\)\w*=.*/\1/gp' .../test/hotspot/jtreg/TEST.groups ...
+ /usr/bin/sort -u
FindJtregGroups /data/workspace/jdk11u/test/jdk
+ /usr/bin/sed ... .../test/jdk/TEST.ROOT
FindJtregGroups /data/workspace/jdk11u/test/langtools
FindJtregGroups /data/workspace/jdk11u/test/nashorn
FindJtregGroups /data/workspace/jdk11u/test/jaxp
```

5 次分别扫 `test/hotspot/jtreg`、`test/jdk`、`test/langtools`、`test/nashorn`、`test/jaxp`，每次用 `sed` 解析 `TEST.ROOT` 和 `TEST.groups` 提取测试组名。你这次没跑测试——这个扫描只是 Init.gmk 的固定动作，给 `make test` 预备的。

2. **找 SPEC 文件**——Init.gmk 在 `build/*/` 下找 `spec.gmk`：

```makefile
ifeq ($(wildcard $(SPEC)), )
  main_targets_file := $(firstword $(wildcard $(build_dir)/*/make-support/main-targets.gmk))
  ifneq ($(main_targets_file), )
    SPEC := $(patsubst %/make-support/main-targets.gmk, %/spec.gmk, $(main_targets_file))
  else
    SPEC := $(firstword $(all_spec_files))
```

如果你 configure 过多次（多个配置目录），它会列出所有 spec。你只有一个配置 `linux-x86_64-normal-server-slowdebug`，所以找到 `build/linux-x86_64-normal-server-slowdebug/spec.gmk`。

3. **生成 main target list**——日志里紧接着输出：

```
Generating main target list
ALL_MAIN_TARGETS := all all-bundles all-bundles-only all-images all-only bootcycle-images ...
```

后面跟着几百个 target 名字——包括每个模块的 clean、java、libs、launchers 等子目标。这个变量是给 `make help` 用的，让 `make help` 知道有哪些目标可构建。

做完这三件事，第一分支发起自调用——重新调用自己，但这次带上 `SPEC=.../spec.gmk` 和 `HAS_SPEC=true`：

```
$(MAKE) -r -R -j 1 -f make/Init.gmk SPEC=.../spec.gmk HAS_SPEC=true main
```

注意 `-j 1`（单线程）——这只是准备阶段，不需要并行。日志里紧接着回显了这条命令，然后再次进入目录——这是第二次进入：

```
Running make as 'make JOBS=32 LOG=debug'
make[1]: Entering directory '/data/workspace/jdk11u'
```

#### 第二次调用：真正工作

`HAS_SPEC=true` 了，Init.gmk 走 `else` 分支。第一件事就是：

```makefile
include $(SPEC)
```

**这是整个构建过程中第一次有人读取 spec.gmk**。spec.gmk 是 configure 的产物——里面记录了所有探测结果：编译器路径、平台标识、版本号、CFLAGS、BOOT_JDK 等。include 它之后，Init.gmk 就有了所有需要的变量。

有了变量之后，Init.gmk 提供 `main` 目标。这个 `main` 目标的 recipe 是整个构建的真正发起者——它调用 Main.gmk 开始 32 路并行编译。

日志里紧接着就是这条命令的真实执行（从 Init.gmk 的 recipe 展开而来）：

```
( cd /data/workspace/jdk11u && \
    /usr/bin/nice /usr/bin/gmake \
        -r -R -I /data/workspace/jdk11u/make/common \
        SPEC=/data/workspace/jdk11u/build/.../spec.gmk \
        LOG_LEVEL=debug -Onone -j 32 \
        -f make/Main.gmk default \
    > >(/usr/bin/tee -a .../build.log) 2> >(/usr/bin/tee -a .../build.log >&2) || \
    ( exitcode=$? && ... on-failure ) )
```

逐个参数拆开看：

| 参数 | 值 | 作用 |
|------|-----|------|
| `nice` | `/usr/bin/nice` | 降低进程优先级，构建不抢占系统 |
| `-r -R` | | 禁用 make 所有内置规则和变量，OpenJDK 自己定义一切 |
| `-I` | `.../make/common` | include 搜索路径，`include Utils.gmk` 之类从这里找 |
| `SPEC=` | `.../spec.gmk` | 把 spec.gmk 路径传给 Main.gmk（Main.gmk 第一件事是 include 它） |
| `-Onone` | | 禁用 make 4.0+ 的输出同步，OpenJDK 用自己的 tee 管道收集日志 |
| `-j 32` | | 32 路并行 |
| `-f` | `make/Main.gmk` | 用 Main.gmk 作为入口——这是第三次文件跳转 |
| `default` | | 构建 default 目标（= exploded-image） |
| `> >(tee...)` | | 所有输出 tee 到 build.log |

末尾的 `|| ( exitcode=$? ... on-failure )` 是失败回退——如果 32 路并行构建中某个模块失败，make 会用单线程重新跑 `Init.gmk on-failure` 目标，整理错误信息。你这次没失败，这段没执行。

这条命令真正执行时，日志记录的是：

```
+ cd /data/workspace/jdk11u
+ /usr/bin/nice /usr/bin/gmake -r -R -I .../make/common SPEC=.../spec.gmk ... -Onone -j 32 -f make/Main.gmk default
```

紧接着日志里出现 `gmake[2]: Entering directory`——这是 Main.gmk 的执行。从此往下 55,000 多行日志，都是 `gmake[2]` 干的事。

### Main.gmk：真正的构建入口

Main.gmk 第一件事就是 `include $(SPEC)`——再次读 spec.gmk（Init.gmk 第二分支已经 include 过一次，这里再 include 一次，确保所有变量在 Main.gmk 作用域内可见）。

然后它定义所有构建目标。你关心的默认目标在 `Main.gmk:1187`：

```makefile
default: $(DEFAULT_MAKE_TARGET)
```

`DEFAULT_MAKE_TARGET` 的值不是写死在 Main.gmk 里的——它在 configure 阶段由 autoconf 决定。看 `make/autoconf/basic.m4:498-510`：

```m4
AC_ARG_WITH(default-make-target, [AS_HELP_STRING([--with-default-make-target],
    [set the default make target @<:@exploded-image@:>@])])
if test "x$with_default_make_target" = "x" \
    || test "x$with_default_make_target" = "xyes"; then
  DEFAULT_MAKE_TARGET="exploded-image"
elif test "x$with_default_make_target" = "xno"; then
  AC_MSG_ERROR([--without-default-make-target is not a valid option])
else
  DEFAULT_MAKE_TARGET="$with_default_make_target"
fi
```

`@<:@exploded-image@:>@` 是 autoconf 转义后的方括号，意思是默认值 `exploded-image`。如果你 configure 时加 `--with-default-make-target=docs-image`，默认目标就变成了文档镜像。你没有加这个参数，所以 configure 把 `DEFAULT_MAKE_TARGET="exploded-image"` 写进了 spec.gmk：

```
$ grep DEFAULT_MAKE_TARGET build/*/spec.gmk
DEFAULT_MAKE_TARGET:=exploded-image
```

Main.gmk include spec.gmk 后，`$(DEFAULT_MAKE_TARGET)` 展开成 `exploded-image`。所以你敲的 `make` 等价于 `make exploded-image`。

### exploded-image 是什么

看 `Main.gmk:1089`：

```makefile
exploded-image: exploded-image-base release-file
```

它依赖两个东西：`exploded-image-base`（把所有产物组装成 jdk/ 目录）和 `release-file`（写 release 文件）。最终的 `build/*/jdk/` 目录——311MB、33 个启动器、38 个 .so、71 个模块——就是 `exploded-image` 的产物。

它和 `make jdk-image` 不同。`jdk-image` 会跑 jlink 把 71 个模块打包成单一的 `lib/modules` 文件、生成 `images/jdk/`——那是分发给用户的版本。`exploded-image` 是开发者的版本：模块是目录展开的（`jdk/modules/java.base/` 下直接放 .class 文件），改一行 Java 源码后下次 make 只编译那一个文件，然后直接覆盖到 jdk/modules/——不用 jlink 重新打包，迭代最快。

`Main.gmk:1188-1192` 还列了几个别名：

```makefile
jdk: exploded-image
images: product-images
docs: docs-image
bundles: all-bundles
all: all-images
```

`make jdk` 等价于 `make exploded-image`（旧式叫法）；`make images` 是打包发布版；`make all` 是所有产物（含 docs、test image）。

### 完整的文件跳转链路

回顾一下，从你敲 `make` 到 Main.gmk 真正开始构建，经过了 4 次文件跳转 + 3 次 make 调用：

```
你敲 make JOBS=32 LOG=debug
  ↓ make 找当前目录的 Makefile
根目录 Makefile（wrapper，检查 GNU Make 版本）
  ↓ include make/Init.gmk
make/Init.gmk 第一次（HAS_SPEC 为空）
  扫测试组、找 SPEC、生成 main target list
  ↓ $(MAKE) -j 1 -f make/Init.gmk SPEC=... HAS_SPEC=true main
make/Init.gmk 第二次（HAS_SPEC=true）
  include $(SPEC) 读 spec.gmk
  提供 main 目标
  ↓ $(NICE) $(MAKE) -j 32 -f make/Main.gmk default
make/Main.gmk
  include $(SPEC) 再读 spec.gmk
  default: $(DEFAULT_MAKE_TARGET) → exploded-image
```

日志里对应三层 `Entering directory`：
- 第一次 `make[1]: Entering`——进 Init.gmk（无 SPEC）
- 第二次 `make[1]: Entering`——进 Init.gmk（带 SPEC）
- `gmake[2]: Entering`——进 Main.gmk

紧接着日志输出 `Building target 'default (exploded-image)'`——这是 Init.gmk 的 main 目标 recipe 打印的，宣告准备工作完成、正式开始构建。

1 分 31 秒后，日志收尾：

```
Finished building target 'default (exploded-image)' in configuration 'linux-x86_64-normal-server-slowdebug'
```

中间这 55,000 多行都是 Main.gmk 干的事。

### 八阶段流水线 + 日志头尾

Main.gmk 用 `DeclareRecipesForPhase` 宏定义了 10 层构建依赖，按实际执行顺序是：

| 阶段 | 做什么 | 关键标记 |
|------|--------|---------|
| 0. 包装层 | Init.gmk 扫测试组、准备日志 | `make[1]: Entering directory` |
| 1. 造工具 | bootstrap JDK 编译 interim javac、ADLC | `Compiling 8 files for BUILD_TOOLS_LANGTOOLS` |
| 2. 生成代码 | ADLC/JFR/JVMTI/DTrace 四套生成器 | `Building JVM variant 'server' with features ...` |
| 3. interim 编译器 | 用 bootstrap JDK 编译 interim langtools | `Compiling 117 files for BUILD_java.compiler.interim` |
| 4. 编译 libjvm.so | 928 个 .cpp → libjvm.so | `Creating libjvm.so from 928 file(s)` |
| 5. 编译 Java 类 | 71 个模块的 .java → .class | `Compiling 3043 files for java.base` |
| 6. 编译原生库 + 启动器 | 35 个 .so + 33 个启动器 | `Linking libverify.so` / `Linking java` |
| 7. 优化收尾 | AddPackagesAttribute、记耗时 | `Compiling 4 files for BUILD_JIGSAW_TOOLS` |

几个关键观察：

1. **阶段 3、4、5 是并行的**——日志里 interim 编译器、HotSpot C++ 编译、Java 模块编译的输出大量交错出现。这不是日志乱序，是 make `-j 32` 真的在三路并行。

2. **阶段 6 横跨整个后半段**——native 库的编译和 Java 模块编译交错，因为 native 库的 JNI 头文件依赖 Java 类的编译产物（`javac -h` 生成 .h），而 native 库本身又被 Java 类通过 `System.loadLibrary` 调用。

3. 日志里 `Linking java` 是一个里程碑——它标志着 33 个启动器中第一个被链接出来。后续的 javac、jcmd、jps 等都跟着 java 之后陆续链接。

最终日志输出耗时统计：

```
----- Build times -------
Start 2026-06-25 10:17:21
End   2026-06-25 10:18:52

00:01:31 TOTAL
-------------------------
```

中间那行空行不是格式错误——是 `ls build_time_diff_* | grep -v _TOTAL` 的输出为空。OpenJDK 默认不开 `LOG_PROFILE_TIMES`，所以拿不到分阶段耗时，只有总的 1 分 31 秒。如果想要每个阶段单独花了多久，configure 时加 `--with-log-profile-times`，build-times/ 下会出现 `build_time_diff_<target>` 文件，那行空行才会被填充。

下面按这 8 个阶段的执行顺序逐个展开。每个阶段讲三件事：做了什么、产出在哪里、和 `src/` 源码是什么关系。

## 阶段一：造工具

### bootstrap 悖论

JDK 包含 `javac`——编译 Java 源代码的编译器。但 `javac` 本身是 `.java` 写的，编译它需要一个 Java 编译器。鸡生蛋问题。

OpenJDK 的解法是分两步：

1. 用一个已有的 JDK（叫 **bootstrap JDK**，configure 时 `--with-boot-jdk` 指定的）编译出一套临时编译器
2. 用这套临时编译器编译真正的 JDK

这套临时编译器叫 **interim langtools**——临时语言工具。它只存在于 `build/buildtools/interim_langtools_modules/` 下，不会进入最终的 JDK 镜像。编译完 JDK 之后就被丢弃了。

### 第一把锤子：BUILD_TOOLS_LANGTOOLS

日志里第一个实际编译出现得很早：

```
Compiling 8 files for BUILD_TOOLS_LANGTOOLS
```

8 个 `.java` 文件——这是构建系统自己用的工具类（`compileproperties`、`propertiesparser` 等），用 bootstrap JDK 的 javac 编译。精简后的编译命令：

```
<bootstrap-jdk>/bin/javac -g \
    -J-XX:+UseSerialGC -J-Xms32M -J-Xmx512M -J-XX:TieredStopAtLevel=1 \
    -source 9 -target 9 \
    -Xlint:all,-deprecation,-options -Werror \
    -d buildtools/langtools_tools_classes
```

几个细节值得注意：

- `-source 9 -target 9`——用 Java 9 语法编译。OpenJDK 11 的构建系统自身工具用 Java 9 而不是 11，因为 bootstrap JDK 可能是 9 或 10，不能假设它支持 11。
- `-J-XX:+UseSerialGC -J-Xms32M -J-Xmx512M -J-XX:TieredStopAtLevel=1`——给 javac 进程本身的 JVM 选项：单线程 GC、512MB 堆上限、只跑 C1（快速启动，不跑 C2 优化）。构建工具不需要高性能，启动快就行。
- `-Werror`——警告视为错误。OpenJDK 构建系统的铁律：任何警告都不能通过。

这 8 个文件编译完之后，产出到 `buildtools/langtools_tools_classes/`——一个很小的 class 目录，后续的构建步骤会用它。

### 模块重命名：interim 机制

有了 BUILD_TOOLS_LANGTOOLS 这把小锤子，make 开始造 interim langtools。但这里有个问题——interim javac 的源码就是 `src/java.compiler/` 和 `src/jdk.compiler/` 下的源码，和最终 JDK 里的 `java.compiler`、`jdk.compiler` 模块是同一份代码。如果直接编译，模块名会冲突——同一个模块名在 classpath 里出现两次。

解法是用 sed **改模块名**。日志里紧接着出现：

```
Generating module-info.java for java.compiler.interim
+ /usr/bin/sed -e 's/java.compiler\([,; ]\)/java.compiler.interim\1/g' \
    -e 's/jdk.compiler\([,; ]\)/jdk.compiler.interim\1/g' \
    -e 's/jdk.javadoc\([,; ]\)/jdk.javadoc.interim\1/g' \
    src/java.compiler/share/classes/module-info.java \
    > buildtools/gensrc/java.compiler.interim/module-info.java
```

这条 sed 命令把 `module-info.java` 里的模块名加一个 `.interim` 后缀：`java.compiler` → `java.compiler.interim`、`jdk.compiler` → `jdk.compiler.interim`、`jdk.javadoc` → `jdk.javadoc.interim`。这样临时编译器的模块名和最终的模块名就不冲突了。

同样的 sed 还对时区数据库工具做了类似处理——把 `java.time.zone` 包改成 `build.tools.tzdb`：

```
/usr/bin/sed -e "s/package java.time.zone/package build.tools.tzdb/" \
    < src/java.base/share/classes/java/time/zone/ZoneRules.java \
    > buildtools/interim_tzdb_classes/build/tools/tzdb/ZoneRules.java
```

4 个时区相关的 `.java` 文件被改了包名，让构建工具能复用 JDK 源码里的时区实现，而不需要单独维护一份。

### 四套 interim 模块

模块名改好之后，make 用 bootstrap JDK 的 javac 编译四套 interim 模块。日志里依次出现：

```
Compiling 117 files for BUILD_java.compiler.interim
Compiling 394 files for BUILD_jdk.compiler.interim
Compiling 299 files for BUILD_jdk.javadoc.interim
Compiling 188 files for BUILD_jdk.rmic.interim
```

| 模块 | 文件数 | 是什么 |
|------|-------|--------|
| java.compiler.interim | 117 | JSR 269（注解处理 API） |
| jdk.compiler.interim | 394 | javac 编译器本体 |
| jdk.javadoc.interim | 299 | javadoc 文档生成器 |
| jdk.rmic.interim | 188 | RMI stub 编译器（老工具，兼容用） |

`jdk.compiler.interim` 是最关键的一个——它就是 javac 本体。394 个 `.java` 文件编译出来之后，存在 `buildtools/interim_langtools_modules/jdk.compiler.interim/` 下。后面编译真正的 JDK Java 源码时，make 会用 `--module-path buildtools/interim_langtools_modules` 来调用这个 interim javac。

这就是 bootstrap 悖论的解法：**用 bootstrap JDK 编译出 interim javac，再用 interim javac 编译真正的 JDK**。两阶段自举。

interim javac 的编译命令（精简后）：

```
<bootstrap-jdk>/bin/javac -g \
    -J-XX:+UseSerialGC -J-Xms32M -J-Xmx512M -J-XX:TieredStopAtLevel=1 \
    -source 9 -target 9 -Werror \
    --module-path buildtools/interim_langtools_modules \
    --add-exports java.base/sun.reflect.annotation=jdk.compiler.interim \
    --add-exports java.base/jdk.internal.jmod=jdk.compiler.interim \
    ...
```

注意 `--module-path buildtools/interim_langtools_modules`——interim 模块之间互相依赖，用 module-path 让它们能看到彼此。`--add-exports` 打开了 JDK 内部 API 给 interim javac 用（`sun.reflect.annotation`、`jdk.internal.jmod` 等内部包）。

四套 interim 模块编译完后，产出到：

```
$ ls build/*/buildtools/interim_langtools_modules/
java.compiler.interim  jdk.compiler.interim  jdk.javadoc.interim

$ ls build/*/buildtools/interim_rmic_modules/
jdk.rmic.interim
```

### ADLC：HotSpot 的"编译器的编译器"

Java 侧的临时编译器（interim javac）造好了。但 HotSpot 侧也需要一个工具——**ADLC**（Architecture Description Language Compiler）。

ADLC 是 HotSpot 的代码生成器——它把 `.ad` 架构描述文件翻译成 C++ 代码。这个工具本身是 C++ 写的，所以需要先用 g++ 编译它。

日志里紧接着出现 HotSpot 侧的开场：

```
Building JVM variant 'server' with features 'aot cds cmsgc compiler1 compiler2 dtrace epsilongc g1gc graal jfr jni-check jvmci jvmti management nmt parallelgc serialgc services vm-structs zgc'
```

这一行列出了你编译的 JVM 变体（`server`）包含的 21 个 feature。每个 feature 对应一组 `.cpp` 文件和编译宏：

| feature | 含义 |
|---------|------|
| aot | Ahead-Of-Time 编译（jaotc 预编译） |
| cds | Class Data Sharing（类数据共享，启动加速） |
| cmsgc | CMS 垃圾回收器（已 deprecated） |
| compiler1 | C1 JIT 编译器（快速编译） |
| compiler2 | C2 JIT 编译器（极致优化） |
| dtrace | DTrace/USDT 探针 |
| epsilongc | Epsilon GC（No-op GC，测试用） |
| g1gc | G1 垃圾回收器（默认 GC） |
| graal | Graal JIT 编译器（用 Java 写的 JIT） |
| jfr | Java Flight Recorder（飞行记录器） |
| jvmci | JVM Compiler Interface（JVM 编译器接口，Graal 的入口） |
| jvmti | JVM Tool Interface（调试/监控接口） |
| zgc | ZGC（低延迟垃圾回收器） |
| | management / nmt / parallelgc / serialgc / services / vm-structs |

这些 feature 决定了后续编译 928 个 `.cpp` 时哪些文件会编进去、哪些 `-D` 宏会打开。比如 `zgc` 对应 `-DINCLUDE_ZGC=1`，`cmsgc` 对应 `-DINCLUDE_CMSGC=1`。

然后 ADLC 本身开始编译。日志宣告：

```
Creating hotspot/variant-server/tools/adlc/adlc from 13 file(s)
```

13 个 `.cpp` → 13 个 `.o` → 链接成 `adlc` 二进制。编译记录：

```
Compiling adlparse.cpp (for adlc)
Compiling archDesc.cpp (for adlc)
Compiling arena.cpp (for adlc)
...
```

链接命令（`BUILD_ADLC_link.cmdline`）：

```
/usr/bin/g++ -o .../hotspot/variant-server/tools/adlc/adlc \
    .../adlc/objs/adlparse.o \
    .../adlc/objs/archDesc.o \
    .../adlc/objs/arena.o \
    .../adlc/objs/dfa.o \
    .../adlc/objs/dict2.o \
    .../adlc/objs/filebuff.o \
    .../adlc/objs/forms.o \
    .../adlc/objs/formsopt.o \
    .../adlc/objs/formssel.o \
    .../adlc/objs/main.o \
    .../adlc/objs/opcodes.o \
    .../adlc/objs/output_c.o \
    .../adlc/objs/output_h.o
```

注意和后面 libjvm.so 的链接命令不同——ADLC 是一个**可执行文件**（`-o adlc`，不是 `-shared`），不依赖任何 JDK 库，只依赖 g++ 标准库。它是一个独立的命令行工具，在下一阶段被调用来从 `.ad` 文件生成 C++ 代码。

ADLC 的源码在 `src/hotspot/share/adlc/`——13 个 `.cpp`：

```
$ ls src/hotspot/share/adlc/*.cpp
adlparse.cpp    forms.cpp       main.cpp      output_c.cpp
archDesc.cpp    formsopt.cpp    opcodes.cpp   output_h.cpp
arena.cpp       formssel.cpp
dict2.cpp
dfa.cpp
filebuff.cpp
```

这是一套完整的解析器——`adlparse.cpp` 是词法/语法分析器、`archDesc.cpp` 是架构描述的内存表示、`output_c.cpp` 和 `output_h.cpp` 是代码生成器（输出 `.cpp` 和 `.hpp`）、`dfa.cpp` 是决策有限自动机（DFA，用于指令选择）。

### 总结

这一阶段产出了两把"锤子"：

| 工具 | 语言 | 源码 | 产出 | 用途 |
|------|------|------|------|------|
| interim javac | Java | `src/jdk.compiler/` 等（394 文件） | `buildtools/interim_langtools_modules/` | 编译真正的 JDK Java 源码 |
| ADLC | C++ | `src/hotspot/share/adlc/`（13 文件） | `hotspot/variant-server/tools/adlc/adlc` | 从 `.ad` 生成 HotSpot C++ 代码（下一节详细展示） |

下一阶段，make 就要用这两把锤子去**生成代码**——ADLC 把 `.ad` 架构描述文件翻译成 `ad_x86.hpp` 和 `ad_x86.cpp`（共 11 个文件、179923 行 C++ 代码），JFR 工具从 `metadata.xml` 生成事件类，JVMTI 工具从 `jvmti.xml` 生成接口代码。

### 补充：feature 宏从哪里来

上面 21 个 feature 列出了你的 JVM 变体包含哪些模块。但它们在源码里是怎么控制的？答案是**编译期宏**。

每个 feature 对应一个 `-D` 编译选项。比如 `compiler1` 对应 `-DCOMPILER1`，`zgc` 对应 `-DINCLUDE_ZGC=1`。这些宏不是写死在 HotSpot 源码里的——它们由 make 构建系统在编译时刻注入。

链路是这样的：

1. configure 阶段，`make/autoconf/jvm-features.m4` 根据 `--with-jvm-variants=server` 和平台探测结果，决定哪些 feature 开启、哪些关闭
2. configure 把结果写入 spec.gmk：`JVM_FEATURES_server := aot cds cmsgc compiler1 compiler2 ...`
3. `make/hotspot/lib/JvmFeatures.gmk` 把每个 feature 翻译成 g++ 编译选项：`compiler1` → `-DCOMPILER1`，`zgc` → `-DINCLUDE_ZGC=1`
4. 最终的 `abstractCompiler.o.cmdline` 里就能看到这些 `-D` 宏：

```
$ grep "COMPILER1\|COMPILER2\|INCLUDE_ZGC" .../libjvm/objs/abstractCompiler.o.cmdline
-DCOMPILER1 -DCOMPILER2 -DINCLUDE_ZGC=1
```

所以你在 HotSpot 源码里看到 `#ifdef COMPILER1` 时，这个宏的定义不是来自任何 `.h` 文件——它是 make 在命令行里 `-D` 注入的。源码跳转跳不过去是正常的，因为源头在 build 系统里，不在源码树里。要知道哪些 feature 被开启了，最快的方法是看你编出来的 `.o.cmdline` 文件。下一阶段编译 libjvm.so 时会详细展示这个。
