# make 到底做了什么

上一章的最后，你跑了 `make JOBS=32 LOG=debug`，1 分 31 秒后 `build/` 目录下多了一个 311MB 的 JDK。

本章逐项拆开这 1 分 31 秒：make 按什么顺序做了什么、每个阶段产出了什么、那些产物和你的 `src/` 源码之间是什么关系。

---

## 1 分 31 秒里发生了什么

先回顾上一章那条命令：

```
$ cd /data/workspace/jdk11u
$ make JOBS=32 LOG=debug
```

`make` 后面没跟任何目标。那它默认做了什么？

### 默认目标：exploded-image

答案在 `make/Main.gmk:1187`：

```makefile
default: $(DEFAULT_MAKE_TARGET)
```

`DEFAULT_MAKE_TARGET` 的值不是写死在 Makefile 里的——它在 configure 阶段由 autoconf 决定。看 `make/autoconf/basic.m4:498-510`：

```m4
AC_ARG_WITH(default-make-target, [AS_HELP_STRING([--with-default-make-target],
    [set the default make target @<:@exploded-image@:>@])])
if test "x$with_default_make_target" = "x" \
    || test "x$with_default_make_target" = "xyes"; then
  DEFAULT_MAKE_TARGET="exploded-image"
elif test "x$with_default_make_target" = "xno"; then
  AC_MSG_ERROR([--without-default-make-target is not a valid option])
else
  DEFAULT_MAKE_TARGET="$with_default_make_target"
fi
```

`@<:@exploded-image@:>@` 是 autoconf 转义后的方括号，意思是默认值 `exploded-image`。如果你 configure 时加 `--with-default-make-target=docs-image`，默认目标就变成了文档镜像。你没有加这个参数，所以 configure 把 `DEFAULT_MAKE_TARGET="exploded-image"` 写进了 spec.gmk：

```
$ grep DEFAULT_MAKE_TARGET build/*/spec.gmk
DEFAULT_MAKE_TARGET:=exploded-image
```

make 启动时读取 spec.gmk，把 `$(DEFAULT_MAKE_TARGET)` 展开成 `exploded-image`。所以你敲的 `make` 等价于 `make exploded-image`。

`exploded-image` 是什么？看 `Main.gmk:1089`：

```makefile
exploded-image: exploded-image-base release-file
```

它依赖两个东西：`exploded-image-base`（把所有产物组装成 jdk/ 目录）和 `release-file`（写 release 文件）。最终的 `build/*/jdk/` 目录——311MB、33 个启动器、38 个 .so、71 个模块——就是 `exploded-image` 的产物。

它和 `make jdk-image` 不同。`jdk-image` 会跑 jlink 把 71 个模块打包成单一的 `lib/modules` 文件、生成 `images/jdk/`——那是分发给用户的版本。`exploded-image` 是开发者的版本：模块是目录展开的（`jdk/modules/java.base/` 下直接放 .class 文件），改一行 Java 源码后下次 make 只编译那一个文件，然后直接覆盖到 jdk/modules/——不用 jlink 重新打包，迭代最快。

`Main.gmk:1188-1192` 还列了几个别名：

```makefile
jdk: exploded-image
images: product-images
docs: docs-image
bundles: all-bundles
all: all-images
```

`make jdk` 等价于 `make exploded-image`（旧式叫法）；`make images` 是打包发布版；`make all` 是所有产物（含 docs、test image）。

### 两层 make 嵌套

日志第一行不是直接开编，而是一个**包装层** make：

```
make[1]: Entering directory '/data/workspace/jdk11u'
FindJtregGroups /data/workspace/jdk11u/test/hotspot/jtreg
```

`make[1]`（带方括号的是子 make）执行 `make/Init.gmk`，它做三件事：

1. **扫描测试组**——调用 `FindJtregGroups` 5 次，分别扫 `test/hotspot/jtreg`、`test/jdk`、`test/langtools`、`test/nashorn`、`test/jaxp`。每次用 `sed` 解析 `TEST.ROOT` 和 `TEST.groups` 文件，提取所有测试组名。这些组名会传给后续的测试目标，但你这次没跑测试——这个扫描只是 Init.gmk 的固定动作。

2. **生成 main target list**——日志行 23：`Generating main target list`。行 24 输出一行超长的 `ALL_MAIN_TARGETS := all all-bundles all-bundles-only ...`——几百个 target 名字，包括每个模块的 clean、java、libs、launchers 等子目标。这个变量会被 `make help` 用到。

3. **准备 build.log 和计时目录**——行 29-52：把上次的 `build.log` 改名为 `build.log.old`，新建空的 `failure-logs/`、`build-times/`、`javacservers/` 目录。行 44 在 `build-times/build_time_start_TOTAL` 写入开始时间戳（`2026 06 25 10 17 21 37041`，37041 是当天零点到现在的秒数）。

做完这些准备工作，Init.gmk 在行 53-65 发起真正的构建——一个 **`gmake` 子调用**：

```
( cd /data/workspace/jdk11u && \
    /usr/bin/nice /usr/bin/gmake \
        -r -R -I /data/workspace/jdk11u/make/common \
        SPEC=/data/workspace/jdk11u/build/.../spec.gmk \
        MAKE_LOG_FLAGS="" LOG_LEVEL=debug \
        -Onone -j 32 \
        -f make/Main.gmk default \
    > >(/usr/bin/tee -a .../build.log) 2> >(/usr/bin/tee -a .../build.log >&2) && wait || \
    ( exitcode=$? && \
      printf "\nERROR: Build failed..." && \
      cd /data/workspace/jdk11u && /usr/bin/gmake -r -R ... \
          -j 1 -f make/Init.gmk HAS_SPEC=true on-failure ; \
      exit $exitcode ) )
```

几个关键参数：

- **`-r -R`**——禁用 make 的所有内置规则和内置变量。OpenJDK 的构建系统自己定义一切，不依赖 make 的默认行为（比如 `%.o: %.c` 这种隐式规则）。这避免了"我以为没生效的规则突然生效"的诡异 bug。
- **`-I /data/workspace/jdk11u/make/common`**——指定 include 搜索路径，所有 `include Utils.gmk` 之类的指令都从这里找。
- **`SPEC=.../spec.gmk`**——把 configure 产出的 spec.gmk 路径作为变量传给 make。Main.gmk 第一件事就是 `include $(SPEC)`，把所有 configure 探测出的变量（编译器路径、平台标识、版本号、CFLAGS 等）加载进来。
- **`-Onone`**——禁用 make 的输出同步（output sync）。OpenJDK 用自己的 `tee` 管道来收集日志，不用 make 4.0+ 的 `--output-sync`。
- **`-j 32`**——32 路并行。
- **`-f make/Main.gmk default`**——用 Main.gmk 作为入口 Makefile，构建 `default` 目标。

注意末尾的 `|| ( exitcode=$? ... -j 1 -f make/Init.gmk HAS_SPEC=true on-failure )`——这是失败回退机制。如果 32 路并行构建中某个模块失败，make 会用 `-j 1`（单线程）重新跑一次 `Init.gmk on-failure` 目标。这个目标会读取 `failure-logs/` 下的失败记录，把错误信息整理成人类可读的报告。你这次构建没失败，所以这段没执行。

日志行 65 是这条 gmake 命令真正执行：

```
+ cd /data/workspace/jdk11u
+ /usr/bin/nice /usr/bin/gmake -r -R -I .../make/common SPEC=.../spec.gmk ... -Onone -j 32 -f make/Main.gmk default
```

行 68 进入 `gmake[2]`——这是真正的 Main.gmk 的执行。从此往下 55,000 多行日志，都是 `gmake[2]` 干的事。

### 八阶段流水线 + 日志头尾

行 37 的宣告：

```
Building target 'default (exploded-image)' in configuration 'linux-x86_64-normal-server-slowdebug'
```

1 分 31 秒后，行 55279 的收尾：

```
Finished building target 'default (exploded-image)' in configuration 'linux-x86_64-normal-server-slowdebug'
```

中间这 55,242 行被 make 拆成了若干阶段。Main.gmk 用 `DeclareRecipesForPhase` 宏定义了 10 层构建依赖，按实际执行顺序是：

| 阶段 | 做什么 | 日志起始行 | 关键标记 |
|------|--------|-----------|---------|
| 0. 包装层 | Init.gmk 扫测试组、准备日志 | 1 | `make[1]: Entering directory` |
| 1. 造工具 | bootstrap JDK 编译 interim javac、ADLC | 437 | `Compiling 8 files for BUILD_TOOLS_LANGTOOLS` |
| 2. 生成代码 | ADLC/JFR/JVMTI/DTrace 四套生成器 | 742 | `Building JVM variant 'server' with features ...` |
| 3. interim 编译器 | 用 bootstrap JDK 编译 interim langtools | 1543 | `Compiling 117 files for BUILD_java.compiler.interim` |
| 4. 编译 libjvm.so | 928 个 .cpp → libjvm.so | 4870 | `Creating libjvm.so from 928 file(s)` |
| 5. 编译 Java 类 | 71 个模块的 .java → .class | 13389 | `Compiling 3043 files for java.base` |
| 6. 编译原生库 + 启动器 | 35 个 .so + 33 个启动器 | 33048 | `Linking libverify.so` / 行 37241 `Linking java` |
| 7. 优化收尾 | AddPackagesAttribute、记耗时 | 55206 | `Compiling 4 files for BUILD_JIGSAW_TOOLS` |

几个关键观察：

1. **阶段 3、4、5 是并行的**——interim 编译器（行 1543-8878）、HotSpot C++ 编译（行 1590-30259）、Java 模块编译（行 13389-49370）的行号范围大量重叠。这不是日志乱序，是 make `-j 32` 真的在三路并行。

2. **阶段 6 横跨整个后半段**——native 库的编译（行 30269-55100）和 Java 模块编译交错，因为 native 库的 JNI 头文件依赖 Java 类的编译产物（`javac -h` 生成 .h），而 native 库本身又被 Java 类通过 `System.loadLibrary` 调用。

3. **行 37241 的 `Linking java`** 是一个里程碑——它标志着 33 个启动器中第一个被链接出来。后续的 javac、jcmd、jps 等都跟着 java 之后陆续链接。

最终行 55264-55269 输出耗时统计：

```
----- Build times -------
Start 2026-06-25 10:17:21
End   2026-06-25 10:18:52

00:01:31 TOTAL
-------------------------
```

中间那行空行不是格式错误——是 `ls build_time_diff_* | grep -v _TOTAL` 的输出为空。OpenJDK 默认不开 `LOG_PROFILE_TIMES`，所以拿不到分阶段耗时，只有总的 1 分 31 秒。如果想要每个阶段单独花了多久，configure 时加 `--with-log-profile-times`，build-times/ 下会出现 `build_time_diff_<target>` 文件，那行空行才会被填充。
