# make 到底做了什么

上一章的最后，你跑了 `make JOBS=32 LOG=debug`，1 分 31 秒后 `build/` 目录下多了一个 311MB 的 JDK。

本章逐项拆开这 1 分 31 秒：make 按什么顺序做了什么、每个阶段产出了什么、那些产物和你的 `src/` 源码之间是什么关系。

---

## 1 分 31 秒里发生了什么

先回顾上一章那条命令：

```
$ cd /data/workspace/jdk11u
$ make JOBS=32 LOG=debug
```

`make` 后面没跟任何目标。那它默认做了什么？

### 你敲 make，make 读的是哪个文件

`make` 后面没跟任何目标。make 的默认行为是找当前目录下名为 `Makefile`（或 `makefile`、`GNUmakefile`）的文件。你根目录下确实有一个 `Makefile`——但它不是真正的构建逻辑。文件头注释自己说得很清楚：

```
###
### This file is just a very small wrapper needed to run the real make/Init.gmk.
### It also performs some sanity checks on make.
###
```

这个 `Makefile` 做三件事：

1. **检查是不是 GNU Make**——`.TEST_FOR_NON_GNUMAKE:sh=echo You are not using GNU Make/gmake...`，Solaris 的 `/usr/bin/make` 会执行这行报错。GNU Make 直接跳过。
2. **检查 make 版本**——`ifeq ($(strip $(foreach v, 3.81% 3.82% 4.%,$(filter $v,$(MAKE_VERSION)))),)` 要求版本 ≥ 3.81。
3. 算出 `topdir` 后 `include $(topdir)/make/Init.gmk`——把控制权交给 `make/Init.gmk`。

根目录 Makefile 全文就 12 行有效代码，没有任何构建规则——它只是个**门卫 + 跳板**。

### Init.gmk：为什么要被调用两次

`make/Init.gmk` 有 364 行，是整个构建的调度中心。但它有一个反直觉的设计——**同一个文件被 make 调用了两次**，走完全不同的代码分支。为什么要这样？

核心矛盾是：**Init.gmk 需要 spec.gmk 里的变量才能工作，但 spec.gmk 的路径它一开始不知道**。

spec.gmk 是 configure 产出的，路径是 `build/linux-x86_64-normal-server-slowdebug/spec.gmk`——这个 `linux-x86_64-normal-server-slowdebug` 是你 configure 时选的配置名，不同配置名路径不同。Init.gmk 不能把这个路径写死，必须运行时去 `build/*/` 下扫描找到它。

但找到了还不够——Init.gmk 想用 spec.gmk 里的变量（比如 `JOBS`、`SPEC`、`TOPDIR`），必须先 `include` 它。而 include 之前，make 的变量作用域里还没有这些值。所以解法是：

1. **第一次调用**：不 include spec.gmk。只扫描 `build/*/` 找到 spec.gmk 的路径，然后**带上这个路径重新调用自己**。
2. **第二次调用**：这次知道 spec.gmk 在哪了，`include $(SPEC)` 把变量加载进来，然后真正开始工作。

Init.gmk 用一个叫 `HAS_SPEC` 的变量区分这两次调用——第一次它是空的，第二次它是 `true`。

#### 第一次调用：找 SPEC

根目录 Makefile `include` Init.gmk 时，`HAS_SPEC` 是空的。Init.gmk:42 检测到这一点，走第一分支。这个分支做三件事：

1. **扫描测试组**——日志开头就是 5 次 `FindJtregGroups` 调用：

```
make[1]: Entering directory '/data/workspace/jdk11u'
FindJtregGroups /data/workspace/jdk11u/test/hotspot/jtreg
+ /usr/bin/sed -n -e 's/\[\(.*\)\]/\1/g' -e 's/^groups\w*=//p' .../test/hotspot/jtreg/TEST.ROOT
+ /usr/bin/sed -n -e 's/^\#.*//g' -e 's/\([^ ]*\)\w*=.*/\1/gp' .../test/hotspot/jtreg/TEST.groups ...
+ /usr/bin/sort -u
FindJtregGroups /data/workspace/jdk11u/test/jdk
+ /usr/bin/sed ... .../test/jdk/TEST.ROOT
FindJtregGroups /data/workspace/jdk11u/test/langtools
FindJtregGroups /data/workspace/jdk11u/test/nashorn
FindJtregGroups /data/workspace/jdk11u/test/jaxp
```

5 次分别扫 `test/hotspot/jtreg`、`test/jdk`、`test/langtools`、`test/nashorn`、`test/jaxp`，每次用 `sed` 解析 `TEST.ROOT` 和 `TEST.groups` 提取测试组名。你这次没跑测试——这个扫描只是 Init.gmk 的固定动作，给 `make test` 预备的。

2. **找 SPEC 文件**——Init.gmk 在 `build/*/` 下找 `spec.gmk`：

```makefile
ifeq ($(wildcard $(SPEC)), )
  main_targets_file := $(firstword $(wildcard $(build_dir)/*/make-support/main-targets.gmk))
  ifneq ($(main_targets_file), )
    SPEC := $(patsubst %/make-support/main-targets.gmk, %/spec.gmk, $(main_targets_file))
  else
    SPEC := $(firstword $(all_spec_files))
```

如果你 configure 过多次（多个配置目录），它会列出所有 spec。你只有一个配置 `linux-x86_64-normal-server-slowdebug`，所以找到 `build/linux-x86_64-normal-server-slowdebug/spec.gmk`。

3. **生成 main target list**——日志里紧接着输出：

```
Generating main target list
ALL_MAIN_TARGETS := all all-bundles all-bundles-only all-images all-only bootcycle-images ...
```

后面跟着几百个 target 名字——包括每个模块的 clean、java、libs、launchers 等子目标。这个变量是给 `make help` 用的，让 `make help` 知道有哪些目标可构建。

做完这三件事，第一分支发起自调用——重新调用自己，但这次带上 `SPEC=.../spec.gmk` 和 `HAS_SPEC=true`：

```
$(MAKE) -r -R -j 1 -f make/Init.gmk SPEC=.../spec.gmk HAS_SPEC=true main
```

注意 `-j 1`（单线程）——这只是准备阶段，不需要并行。日志里紧接着回显了这条命令，然后再次进入目录——这是第二次进入：

```
Running make as 'make JOBS=32 LOG=debug'
make[1]: Entering directory '/data/workspace/jdk11u'
```

#### 第二次调用：真正工作

`HAS_SPEC=true` 了，Init.gmk 走 `else` 分支。第一件事就是：

```makefile
include $(SPEC)
```

**这是整个构建过程中第一次有人读取 spec.gmk**。spec.gmk 是 configure 的产物——里面记录了所有探测结果：编译器路径、平台标识、版本号、CFLAGS、BOOT_JDK 等。include 它之后，Init.gmk 就有了所有需要的变量。

有了变量之后，Init.gmk 提供 `main` 目标。这个 `main` 目标的 recipe 是整个构建的真正发起者——它调用 Main.gmk 开始 32 路并行编译。

日志里紧接着就是这条命令的真实执行（从 Init.gmk 的 recipe 展开而来）：

```
( cd /data/workspace/jdk11u && \
    /usr/bin/nice /usr/bin/gmake \
        -r -R -I /data/workspace/jdk11u/make/common \
        SPEC=/data/workspace/jdk11u/build/.../spec.gmk \
        LOG_LEVEL=debug -Onone -j 32 \
        -f make/Main.gmk default \
    > >(/usr/bin/tee -a .../build.log) 2> >(/usr/bin/tee -a .../build.log >&2) || \
    ( exitcode=$? && ... on-failure ) )
```

逐个参数拆开看：

| 参数 | 值 | 作用 |
|------|-----|------|
| `nice` | `/usr/bin/nice` | 降低进程优先级，构建不抢占系统 |
| `-r -R` | | 禁用 make 所有内置规则和变量，OpenJDK 自己定义一切 |
| `-I` | `.../make/common` | include 搜索路径，`include Utils.gmk` 之类从这里找 |
| `SPEC=` | `.../spec.gmk` | 把 spec.gmk 路径传给 Main.gmk（Main.gmk 第一件事是 include 它） |
| `-Onone` | | 禁用 make 4.0+ 的输出同步，OpenJDK 用自己的 tee 管道收集日志 |
| `-j 32` | | 32 路并行 |
| `-f` | `make/Main.gmk` | 用 Main.gmk 作为入口——这是第三次文件跳转 |
| `default` | | 构建 default 目标（= exploded-image） |
| `> >(tee...)` | | 所有输出 tee 到 build.log |

末尾的 `|| ( exitcode=$? ... on-failure )` 是失败回退——如果 32 路并行构建中某个模块失败，make 会用单线程重新跑 `Init.gmk on-failure` 目标，整理错误信息。你这次没失败，这段没执行。

这条命令真正执行时，日志记录的是：

```
+ cd /data/workspace/jdk11u
+ /usr/bin/nice /usr/bin/gmake -r -R -I .../make/common SPEC=.../spec.gmk ... -Onone -j 32 -f make/Main.gmk default
```

紧接着日志里出现 `gmake[2]: Entering directory`——这是 Main.gmk 的执行。从此往下 55,000 多行日志，都是 `gmake[2]` 干的事。

### Main.gmk：真正的构建入口

Main.gmk 第一件事就是 `include $(SPEC)`——再次读 spec.gmk（Init.gmk 第二分支已经 include 过一次，这里再 include 一次，确保所有变量在 Main.gmk 作用域内可见）。

然后它定义所有构建目标。你关心的默认目标在 `Main.gmk:1187`：

```makefile
default: $(DEFAULT_MAKE_TARGET)
```

`DEFAULT_MAKE_TARGET` 的值不是写死在 Main.gmk 里的——它在 configure 阶段由 autoconf 决定。看 `make/autoconf/basic.m4:498-510`：

```m4
AC_ARG_WITH(default-make-target, [AS_HELP_STRING([--with-default-make-target],
    [set the default make target @<:@exploded-image@:>@])])
if test "x$with_default_make_target" = "x" \
    || test "x$with_default_make_target" = "xyes"; then
  DEFAULT_MAKE_TARGET="exploded-image"
elif test "x$with_default_make_target" = "xno"; then
  AC_MSG_ERROR([--without-default-make-target is not a valid option])
else
  DEFAULT_MAKE_TARGET="$with_default_make_target"
fi
```

`@<:@exploded-image@:>@` 是 autoconf 转义后的方括号，意思是默认值 `exploded-image`。如果你 configure 时加 `--with-default-make-target=docs-image`，默认目标就变成了文档镜像。你没有加这个参数，所以 configure 把 `DEFAULT_MAKE_TARGET="exploded-image"` 写进了 spec.gmk：

```
$ grep DEFAULT_MAKE_TARGET build/*/spec.gmk
DEFAULT_MAKE_TARGET:=exploded-image
```

Main.gmk include spec.gmk 后，`$(DEFAULT_MAKE_TARGET)` 展开成 `exploded-image`。所以你敲的 `make` 等价于 `make exploded-image`。

### exploded-image 是什么

看 `Main.gmk:1089`：

```makefile
exploded-image: exploded-image-base release-file
```

它依赖两个东西：`exploded-image-base`（把所有产物组装成 jdk/ 目录）和 `release-file`（写 release 文件）。最终的 `build/*/jdk/` 目录——311MB、33 个启动器、38 个 .so、71 个模块——就是 `exploded-image` 的产物。

它和 `make jdk-image` 不同。`jdk-image` 会跑 jlink 把 71 个模块打包成单一的 `lib/modules` 文件、生成 `images/jdk/`——那是分发给用户的版本。`exploded-image` 是开发者的版本：模块是目录展开的（`jdk/modules/java.base/` 下直接放 .class 文件），改一行 Java 源码后下次 make 只编译那一个文件，然后直接覆盖到 jdk/modules/——不用 jlink 重新打包，迭代最快。

`Main.gmk:1188-1192` 还列了几个别名：

```makefile
jdk: exploded-image
images: product-images
docs: docs-image
bundles: all-bundles
all: all-images
```

`make jdk` 等价于 `make exploded-image`（旧式叫法）；`make images` 是打包发布版；`make all` 是所有产物（含 docs、test image）。

### 完整的文件跳转链路

回顾一下，从你敲 `make` 到 Main.gmk 真正开始构建，经过了 4 次文件跳转 + 3 次 make 调用：

```
你敲 make JOBS=32 LOG=debug
  ↓ make 找当前目录的 Makefile
根目录 Makefile（wrapper，检查 GNU Make 版本）
  ↓ include make/Init.gmk
make/Init.gmk 第一次（HAS_SPEC 为空）
  扫测试组、找 SPEC、生成 main target list
  ↓ $(MAKE) -j 1 -f make/Init.gmk SPEC=... HAS_SPEC=true main
make/Init.gmk 第二次（HAS_SPEC=true）
  include $(SPEC) 读 spec.gmk
  提供 main 目标
  ↓ $(NICE) $(MAKE) -j 32 -f make/Main.gmk default
make/Main.gmk
  include $(SPEC) 再读 spec.gmk
  default: $(DEFAULT_MAKE_TARGET) → exploded-image
```

日志里对应三层 `Entering directory`：
- 第一次 `make[1]: Entering`——进 Init.gmk（无 SPEC）
- 第二次 `make[1]: Entering`——进 Init.gmk（带 SPEC）
- `gmake[2]: Entering`——进 Main.gmk

紧接着日志输出 `Building target 'default (exploded-image)'`——这是 Init.gmk 的 main 目标 recipe 打印的，宣告准备工作完成、正式开始构建。

1 分 31 秒后，日志收尾：

```
Finished building target 'default (exploded-image)' in configuration 'linux-x86_64-normal-server-slowdebug'
```

中间这 55,000 多行都是 Main.gmk 干的事。

### 八阶段流水线 + 日志头尾

Main.gmk 用 `DeclareRecipesForPhase` 宏定义了 10 层构建依赖，按实际执行顺序是：

| 阶段 | 做什么 | 关键标记 |
|------|--------|---------|
| 0. 包装层 | Init.gmk 扫测试组、准备日志 | `make[1]: Entering directory` |
| 1. 造工具 | bootstrap JDK 编译 interim javac、ADLC | `Compiling 8 files for BUILD_TOOLS_LANGTOOLS` |
| 2. 生成代码 | ADLC/JFR/JVMTI/DTrace 四套生成器 | `Building JVM variant 'server' with features ...` |
| 3. interim 编译器 | 用 bootstrap JDK 编译 interim langtools | `Compiling 117 files for BUILD_java.compiler.interim` |
| 4. 编译 libjvm.so | 928 个 .cpp → libjvm.so | `Creating libjvm.so from 928 file(s)` |
| 5. 编译 Java 类 | 71 个模块的 .java → .class | `Compiling 3043 files for java.base` |
| 6. 编译原生库 + 启动器 | 35 个 .so + 33 个启动器 | `Linking libverify.so` / `Linking java` |
| 7. 优化收尾 | AddPackagesAttribute、记耗时 | `Compiling 4 files for BUILD_JIGSAW_TOOLS` |

几个关键观察：

1. **阶段 3、4、5 是并行的**——日志里 interim 编译器、HotSpot C++ 编译、Java 模块编译的输出大量交错出现。这不是日志乱序，是 make `-j 32` 真的在三路并行。

2. **阶段 6 横跨整个后半段**——native 库的编译和 Java 模块编译交错，因为 native 库的 JNI 头文件依赖 Java 类的编译产物（`javac -h` 生成 .h），而 native 库本身又被 Java 类通过 `System.loadLibrary` 调用。

3. 日志里 `Linking java` 是一个里程碑——它标志着 33 个启动器中第一个被链接出来。后续的 javac、jcmd、jps 等都跟着 java 之后陆续链接。

最终日志输出耗时统计：

```
----- Build times -------
Start 2026-06-25 10:17:21
End   2026-06-25 10:18:52

00:01:31 TOTAL
-------------------------
```

中间那行空行不是格式错误——是 `ls build_time_diff_* | grep -v _TOTAL` 的输出为空。OpenJDK 默认不开 `LOG_PROFILE_TIMES`，所以拿不到分阶段耗时，只有总的 1 分 31 秒。如果想要每个阶段单独花了多久，configure 时加 `--with-log-profile-times`，build-times/ 下会出现 `build_time_diff_<target>` 文件，那行空行才会被填充。

下面按这 8 个阶段的执行顺序逐个展开。每个阶段讲三件事：做了什么、产出在哪里、和 `src/` 源码是什么关系。

## 阶段一：造工具

### bootstrap 悖论

JDK 包含 `javac`——编译 Java 源代码的编译器。但 `javac` 本身是 `.java` 写的，编译它需要一个 Java 编译器。鸡生蛋问题。

OpenJDK 的解法是分两步：

1. 用一个已有的 JDK（叫 **bootstrap JDK**，configure 时 `--with-boot-jdk` 指定的）编译出一套临时编译器
2. 用这套临时编译器编译真正的 JDK

这套临时编译器叫 **interim langtools**——临时语言工具。它只存在于 `build/buildtools/interim_langtools_modules/` 下，不会进入最终的 JDK 镜像。编译完 JDK 之后就被丢弃了。

### 第一把锤子：BUILD_TOOLS_LANGTOOLS

日志里第一个实际编译出现得很早：

```
Compiling 8 files for BUILD_TOOLS_LANGTOOLS
```

8 个 `.java` 文件——这是构建系统自己用的工具类（`compileproperties`、`propertiesparser` 等），用 bootstrap JDK 的 javac 编译。精简后的编译命令：

```
<bootstrap-jdk>/bin/javac -g \
    -J-XX:+UseSerialGC -J-Xms32M -J-Xmx512M -J-XX:TieredStopAtLevel=1 \
    -source 9 -target 9 \
    -Xlint:all,-deprecation,-options -Werror \
    -d buildtools/langtools_tools_classes
```

几个细节值得注意：

- `-source 9 -target 9`——用 Java 9 语法编译。OpenJDK 11 的构建系统自身工具用 Java 9 而不是 11，因为 bootstrap JDK 可能是 9 或 10，不能假设它支持 11。
- `-J-XX:+UseSerialGC -J-Xms32M -J-Xmx512M -J-XX:TieredStopAtLevel=1`——给 javac 进程本身的 JVM 选项：单线程 GC、512MB 堆上限、只跑 C1（快速启动，不跑 C2 优化）。构建工具不需要高性能，启动快就行。
- `-Werror`——警告视为错误。OpenJDK 构建系统的铁律：任何警告都不能通过。

这 8 个文件编译完之后，产出到 `buildtools/langtools_tools_classes/`——一个很小的 class 目录，后续的构建步骤会用它。

### 模块重命名：interim 机制

有了 BUILD_TOOLS_LANGTOOLS 这把小锤子，make 开始造 interim langtools。但这里有个问题——interim javac 的源码就是 `src/java.compiler/` 和 `src/jdk.compiler/` 下的源码，和最终 JDK 里的 `java.compiler`、`jdk.compiler` 模块是同一份代码。如果直接编译，模块名会冲突——同一个模块名在 classpath 里出现两次。

解法是用 sed **改模块名**。日志里紧接着出现：

```
Generating module-info.java for java.compiler.interim
+ /usr/bin/sed -e 's/java.compiler\([,; ]\)/java.compiler.interim\1/g' \
    -e 's/jdk.compiler\([,; ]\)/jdk.compiler.interim\1/g' \
    -e 's/jdk.javadoc\([,; ]\)/jdk.javadoc.interim\1/g' \
    src/java.compiler/share/classes/module-info.java \
    > buildtools/gensrc/java.compiler.interim/module-info.java
```

这条 sed 命令把 `module-info.java` 里的模块名加一个 `.interim` 后缀：`java.compiler` → `java.compiler.interim`、`jdk.compiler` → `jdk.compiler.interim`、`jdk.javadoc` → `jdk.javadoc.interim`。这样临时编译器的模块名和最终的模块名就不冲突了。

同样的 sed 还对时区数据库工具做了类似处理——把 `java.time.zone` 包改成 `build.tools.tzdb`：

```
/usr/bin/sed -e "s/package java.time.zone/package build.tools.tzdb/" \
    < src/java.base/share/classes/java/time/zone/ZoneRules.java \
    > buildtools/interim_tzdb_classes/build/tools/tzdb/ZoneRules.java
```

4 个时区相关的 `.java` 文件被改了包名，让构建工具能复用 JDK 源码里的时区实现，而不需要单独维护一份。

### 四套 interim 模块

模块名改好之后，make 用 bootstrap JDK 的 javac 编译四套 interim 模块。日志里依次出现：

```
Compiling 117 files for BUILD_java.compiler.interim
Compiling 394 files for BUILD_jdk.compiler.interim
Compiling 299 files for BUILD_jdk.javadoc.interim
Compiling 188 files for BUILD_jdk.rmic.interim
```

| 模块 | 文件数 | 是什么 |
|------|-------|--------|
| java.compiler.interim | 117 | JSR 269（注解处理 API） |
| jdk.compiler.interim | 394 | javac 编译器本体 |
| jdk.javadoc.interim | 299 | javadoc 文档生成器 |
| jdk.rmic.interim | 188 | RMI stub 编译器（老工具，兼容用） |

`jdk.compiler.interim` 是最关键的一个——它就是 javac 本体。394 个 `.java` 文件编译出来之后，存在 `buildtools/interim_langtools_modules/jdk.compiler.interim/` 下。后面编译真正的 JDK Java 源码时，make 会用 `--module-path buildtools/interim_langtools_modules` 来调用这个 interim javac。

这就是 bootstrap 悖论的解法：**用 bootstrap JDK 编译出 interim javac，再用 interim javac 编译真正的 JDK**。两阶段自举。

interim javac 的编译命令（精简后）：

```
<bootstrap-jdk>/bin/javac -g \
    -J-XX:+UseSerialGC -J-Xms32M -J-Xmx512M -J-XX:TieredStopAtLevel=1 \
    -source 9 -target 9 -Werror \
    --module-path buildtools/interim_langtools_modules \
    --add-exports java.base/sun.reflect.annotation=jdk.compiler.interim \
```

### ADLC：HotSpot 的"编译器的编译器"

Java 侧的临时编译器（interim javac）造好了。但 HotSpot 侧也需要一个工具——**ADLC**（Architecture Description Language Compiler）。

ADLC 是 HotSpot 的代码生成器——它把 `.ad` 架构描述文件翻译成 C++ 代码。这个工具本身是 C++ 写的，所以需要先用 g++ 编译它。

### 为什么要生成 C++ 代码，而不是直接手写

你可能会问：ADLC 的输入只有 ~13000 行 `.ad` 文件（`x86_64.ad` + `x86.ad`），为什么不让 HotSpot 开发者直接手写那 179923 行 C++ 代码？答案有三个。

**第一，数量级差距太大了。** 13000 行 DSL 被 ADLC 放大到 179923 行 C++——放大 7.7 倍。这不是简单的"把 DSL 翻译成 C++ 语法"，而是 ADLC 从 DSL 里推导出很多额外的数据结构和校验代码。看一下输出文件的内容：

| 输出文件 | 行数 | 内容 |
|---------|------|------|
| `ad_x86.cpp` | 56612 | 指令选择的匹配实现——JIT 编译器决定"这条 Java 操作应该变成哪条 x86 指令" |
| `ad_x86.hpp` | 49184 | 所有指令操作数的枚举和机器指令的 C++ 类定义 |
| `ad_x86_format.cpp` | 25675 | 调试输出——把你 JIT 生成的机器码格式化成人可读的汇编 |
| `dfa_x86.cpp` | 22702 | 决定性有限自动机——指令选择的匹配引擎，数千个状态节点 |
| `ad_x86_expand.cpp` | 10929 | 复杂指令的分解——一条逻辑指令拆成多条硬件指令 |
| 其余 6 个文件 | ~5500 | peephole 优化、pipeline 调度、clone 操作、misc |

**第二，指令选择 DFA 手写几乎不可能正确。** `dfa_x86.cpp` 是一个**决定性有限自动机**的 C++ 实现——22702 行，由 ADLC 的 `dfa.cpp` 模块从 `.ad` 文件中的指令模式**自动推导**出来。指令选择和 DFA 的关系是：每条 x86 指令有一个"匹配模式"（比如"这条指令需要两个寄存器操作数、产生一个 flags 输出"），JIT 编译器在生成机器码时，要从几百条 x86 指令里选出"最适合当前 Java 操作码的那条"——比如 `iadd` 在 x86 上可以用 `add_reg_reg`、`add_reg_mem`、`add_mem_reg`、`add_imm_reg` 等等。DFA 就是做这个"最合适匹配"的引擎。每条指令的匹配规则在 `.ad` 文件里用几行 DSL 描述，ADLC 自动计算出所有指令模式之间的冲突和优先级，生成 22702 行状态转移代码——如果手写，几千个状态节点的转移表，改一条规则就要全盘重算。

**第三，一份 DSL 出多份代码，改一处自动同步。** `.ad` 文件里改一条指令的寄存器约束，ADLC **同时更新** 6 个输出文件：匹配实现（`ad_x86.cpp`）、类定义（`ad_x86.hpp`）、格式化输出（`ad_x86_format.cpp`）、DFA 状态表（`dfa_x86.cpp`）、指令展开（`ad_x86_expand.cpp`）、pipeline 调度。如果手写，开发者必须人工找到 6 个文件中所有和这条指令相关的代码，逐个修改。这不仅工作量巨大，更危险的是——**漏改一处，JIT 编译器就会悄悄生成错误的机器码**。代码看起来编译通过了，跑起来就崩，而且 debug 极难（崩溃发生在生成的机器码里，不是在你的 C++ 源码里）。

`.ad` 文件里描述一条 x86 指令长什么样呢？以 `Add` 指令为例（简化）：

```
instruct addI_rReg(rRegI dst, rRegI src)
%{
  match(Set dst (AddI dst src));
  format %{ "addl    $dst, $src" %}
  ins_encode %{ __ addl($dst$$Register, $src$$Register); %}
%}
```

四行 DSL，描述了：这条指令叫 `addI_rReg`、它匹配"两个寄存器相加"的模式、汇编输出是 `addl r1, r2`、机器码编码是 `__ addl()`。ADLC 从这四行自动生成：指令类（C++ class）、DFA 匹配状态、调试格式输出、寄存器分配信息。这四行 DSL 只需要写一次，ADLC 确保 6 个输出文件全部一致。

这和 `#ifdef COMPILER1` 那个宏不同——那个是"开关"，用来选择编译哪些代码。ADLC 是"生成器"，用来产生新的代码。开关只能决定"编不编这段代码"，生成器直接创造之前不存在的代码。

### 补充：feature 宏从哪里来

上面 21 个 feature 列出了你的 JVM 变体包含哪些模块。但它们在源码里是怎么控制的？答案是**编译期宏**。

每个 feature 对应一个 `-D` 编译选项。比如 `compiler1` 对应 `-DCOMPILER1`，`zgc` 对应 `-DINCLUDE_ZGC=1`。这些宏不是写死在 HotSpot 源码里的——它们由 make 构建系统在编译时刻注入。

链路是这样的：

1. configure 阶段，`make/autoconf/jvm-features.m4` 根据 `--with-jvm-variants=server` 和平台探测结果，决定哪些 feature 开启、哪些关闭
2. configure 把结果写入 spec.gmk：`JVM_FEATURES_server := aot cds cmsgc compiler1 compiler2 ...`
3. `make/hotspot/lib/JvmFeatures.gmk` 把每个 feature 翻译成 g++ 编译选项：`compiler1` → `-DCOMPILER1`，`zgc` → `-DINCLUDE_ZGC=1`
4. 最终的 `abstractCompiler.o.cmdline` 里就能看到这些 `-D` 宏：

```
$ grep "COMPILER1\|COMPILER2\|INCLUDE_ZGC" .../libjvm/objs/abstractCompiler.o.cmdline
-DCOMPILER1 -DCOMPILER2 -DINCLUDE_ZGC=1
```

所以你在 HotSpot 源码里看到 `#ifdef COMPILER1` 时，这个宏的定义不是来自任何 `.h` 文件——它是 make 在命令行里 `-D` 注入的。源码跳转跳不过去是正常的，因为源头在 build 系统里，不在源码树里。要知道哪些 feature 被开启了，最快的方法是看你编出来的 `.o.cmdline` 文件。下一阶段编译 libjvm.so 时会详细展示这个。

### ADLC：把 DSL 翻译成 C++

上一节造 ADLC 工具时讲过：它的输入是 `.ad` 架构描述文件，输出是 11 个 C++ 文件。现在看实际的输入和输出。

#### 输入长什么样

`x86_64.ad`（13326 行）和 `x86.ad`（9834 行）是入口。ADLC 调用前，先用 gawk 把它们合并成 `all-ad-src.ad`：

```
/usr/bin/gawk '
  { if (CUR_FN != FILENAME) {
      CUR_FN=FILENAME; NR_BASE=NR-1; need_lineno=1 }
    if (need_lineno && $0 !~ /\/\//) {
      print "\n\n#line " (NR-NR_BASE) " \"" FILENAME "\"";
      need_lineno=0 };
    print
  }' x86_64.ad x86.ad > all-ad-src.ad
```

合并目的是让 `.ad` 文件之间的关系可追踪——`x86_64.ad` 通过 `source_hpp %{ #include "x86.ad" %}` 引用 `x86.ad`，ADLC 需要看到完整的指令集。

来看看 `.ad` 文件实际的内容。`x86_64.ad` 开头是寄存器定义块：

```
// AMD64 Architecture Description File

//----------REGISTER DEFINITION BLOCK------------------------------------------
// "reg_def"  name ( register save type, C convention save type,
//                   ideal register type, encoding );
// Register Save Types:
// NS  = No-Save    SOC = Save-On-Call
// SOE = Save-On-Entry   AS  = Always-Save

register %{
// General Registers
```

这不是 C 语言，是 HotSpot 自创的 DSL（Domain Specific Language）。`register %{...%}` 定义了 x86_64 平台的所有寄存器：rax、rbx、rcx、rdx 等等以及它们的保存类型。ADLC 会从这里生成 `enum MachOperands`——上一节在 `ad_x86.hpp` 里看到的那个几十行的操作数枚举。

文件末尾是指令的 peephole 优化规则：

```
peephole
%{
  peepmatch (loadI storeI);
  peepconstraint (1.src == 0.dst, 1.mem == 0.mem);
  peepreplace (storeI(1.mem 1.mem 1.src));
%}
```

这条规则说：如果遇到"load 后立刻 store 到同一个内存位置"，可以直接用 store 的值跳过 load——典型的 peephole 优化。这些规则被 ADLC 编译进 `ad_x86_peephole.cpp`（7454 字节）。

#### ADLC 怎么生成代码

合并后的 `all-ad-src.ad` 传给 ADLC 二进制：

```
./adlc -q -T -DLINUX=1 -D_GNU_SOURCE=1 -g -DAMD64=1 -D_LP64=1 \
    all-ad-src.ad \
    -c ad_x86.cpp \
    -h ad_x86.hpp \
    -a dfa_x86.cpp \
    -v adGlobals_x86.hpp
```

4 个输出标志：
- `-c`：主实现文件——指令匹配和代码生成
- `-h`：主头文件——MachOperands 枚举和所有 MachNode 类定义
- `-a`：DFA（确定性有限自动机）——指令选择的状态机
- `-v`：全局变量定义

ADLC 的初始输出还要经过一次 gawk 后处理——把占位符 `#line 999999` 替换成正确的行号：

```
/usr/bin/gawk '
  BEGIN { print "#line 1 \"ad_x86.cpp\""; }
  /^#line 999999$/ { print "#line " (NR+1) " \"ad_x86.cpp\""; next }
  { print }' ad_x86.cpp > adfiles/ad_x86.cpp
```

所有 11 个文件都经过这个后处理流程。

#### 输出长什么样

所有生成文件第一行（版权头之后）都有：

```
// Machine Generated File.  Do Not Edit!
```

以 `ad_x86.hpp` 为例（49184 行，2.0MB），开头是：

```cpp
#ifndef GENERATED_ADFILES_AD_HPP
#define GENERATED_ADFILES_AD_HPP

#include "memory/allocation.hpp"
#include "code/nativeInst.hpp"
#include "opto/machnode.hpp"

enum MachOperands {
  /*    0 */  UNIVERSE,
  /*    1 */  LABEL,
  /*    2 */  SREGI,
  /*    3 */  SREGP,
  /*    4 */  SREGF,
  /*    5 */  SREGD,
  ...
};
```

这个 `enum MachOperands` 就是从 `.ad` 的 `register %{...%}` 块推导出来的。每个条目对应一个寄存器或操作数类型。`UNIVERSE` 是全局集合、`LABEL` 是跳转标号、`SREGI`/`SREGP`/`SREGF`/`SREGD`/`SREGL` 分别是栈寄存器、指针、浮点、双精度、长整型——都是 x86 指令集的操作数类型。

再打开 `ad_x86.cpp`（56612 行，1.8MB）：

```cpp
// Machine Generated File.  Do Not Edit!

#include "precompiled.hpp"
#include "adfiles/adGlobals_x86.hpp"
#include "adfiles/ad_x86.hpp"
```

注意它 `#include "adfiles/ad_x86.hpp"`——头文件和实现文件都是同一个目录下的生成产物。`precompiled.hpp` 是 HotSpot 的预编译头。

#### 11 个文件的清单

ADLC 从 23160 行 `.ad` DSL 生成了 179923 行 C++ 代码：

| 文件 | 行数 | 大小 | 内容 |
|------|------|------|------|

| ad_x86_expand.cpp | ~10929 | 356KB | 宏指令展开 |
| ad_x86_pipeline.cpp | ~5250 | 315KB | 流水线调度 |

### JFR：用 XML 描述事件

JFR 的代码生成器和 ADLC 完全不同——它不是 C++ 写的命令行工具，而是 Java 写的，用 bootstrap JDK 来跑。

#### 输入长什么样

JFR 的输入是 `src/hotspot/share/jfr/metadata/metadata.xml`（1168 行）——一个声明式的 XML，描述 JVM 支持的所有事件类型：

```xml
<Metadata>
  <Event name="ThreadStart" category="Java Application"
         label="Java Thread Start" thread="true" stackTrace="true">
    <Field type="Thread" name="thread" label="New Java Thread" />
    <Field type="Thread" name="parentThread" label="Parent Java Thread" />
  </Event>

  <Event name="ThreadEnd" category="Java Application"
         label="Java Thread End" thread="true" startTime="false">
    <Field type="Thread" name="thread" label="Java Thread" />
  </Event>

  <Event name="ThreadSleep" category="Java Application"
         label="Java Thread Sleep" thread="true" stackTrace="true">
    <Field type="long" contentType="millis" name="time" label="Sleep Time" />
  </Event>
```

每个 `<Event>` 描述了一个 JFR 事件：它的名字、分类、包含哪些字段。字段有类型（`Thread`、`long`、`Class` 等）和内容类型（`millis`、`nanos`、`address` 等）。文件末尾还定义了 XML Schema 类型映射：

```xml
<XmlType name="long" javaType="long" parameterType="i8" fieldType="i8"/>
<XmlType name="string" javaType="java.lang.String" parameterType="const char*" fieldType="const char*"/>
<XmlContentType name="millis" annotation="jdk.jfr.Timespan(MILLISECONDS)" />
<XmlContentType name="address" annotation="jdk.jfr.MemoryAddress" />
```

这些映射告诉代码生成器：XML 里写 `type="long"` 时要生成 C++ 的 `i8`（8 字节 int），`type="string"` 时要生成 `const char*`（C 字符串）。

#### 工具怎么跑

生成器 `build.tools.jfr.GenerateJfrFiles` 是一个 Java 类，它的调用命令记录了完整过程：

```
<bootstrap-jdk>/bin/java -XX:+UseSerialGC -Xms32M -Xmx512M \
    -cp buildtools/tools_classes \
    build.tools.jfr.GenerateJfrFiles \
    src/hotspot/share/jfr/metadata/metadata.xml \
    src/hotspot/share/jfr/metadata/metadata.xsd \
    gensrc/jfrfiles
```

三个参数：`metadata.xml`（事件定义）、`metadata.xsd`（XSD schema，用于校验 XML 格式）、输出目录 `jfrfiles/`。日志里对应的标记：

```
Generating jfrEventClasses.hpp
```

#### 输出长什么样

5 个 `.hpp` 文件。每个文件首行都是：

```cpp
/* AUTOMATICALLY GENERATED FILE - DO NOT EDIT */
```

最大的 `jfrEventClasses.hpp`（340KB）——包含所有事件的结构体定义：

```cpp
struct JfrStructVirtualSpace
{
 private:
  u8 _start;
  u8 _committedEnd;
  u8 _committedSize;
  u8 _reservedEnd;
  u8 _reservedSize;

 public:
  void set_start(u8 new_value) { this->_start = new_value; }
  void set_committedEnd(u8 new_value) { this->_committedEnd = new_value; }
  // ...

  template <typename Writer>
  void writeData(Writer& w) {
    w.write(_start);
    w.write(_committedEnd);
    w.write(_committedSize);
    w.write(_reservedEnd);
    w.write(_reservedSize);
  }
};
```

注意这个 struct 是如何从 XML 生成的：XML 里的每个 `<Field>` 变成了 private 成员变量 + setter + writeData 中的一个 `w.write()` 调用。一份 XML 描述了"这个事件有哪些字段"，代码生成器自动产出了存储、设置、序列化的完整 C++ 代码。

`jfrEventIds.hpp` 是事件的枚举：

```cpp
enum JfrEventId {
  _jfreventbase = (NUM_RESERVED_EVENTS-1),
  JfrThreadStartEvent,
  JfrThreadEndEvent,
  JfrThreadSleepEvent,
  JfrThreadParkEvent,
  JfrJavaMonitorEnterEvent,
  JfrJavaMonitorWaitEvent,
  ...
};
```

每个 `<Event name="ThreadStart">` 在 XML 里变成了 `JfrThreadStartEvent` 枚举值。代码生成器自动给每个事件分配了唯一的 ID。

### JVMTI：一份 XML，多种输出

JVMTI 是所有生成器里最特别的——**同一份 XML 输入，经过不同的 XSLT 样式表，产出完全不同的输出文件**：C++ 实现骨架、C 头文件、甚至 HTML 文档。

#### 输入长什么样

`src/hotspot/share/prims/jvmti.xml` 不是普通的 XML 文件——它用 DTD 自描述结构，头部还声明了 XSLT 样式表：

```xml
<?xml-stylesheet type="text/xsl" href="jvmti.xsl"?>
<!DOCTYPE specification [
   <!ELEMENT specification (title, copyright, intro*, functionsection,
                            errorsection, eventsection, datasection,
                            issuessection, changehistory)>
   <!ELEMENT function (synopsis, typedef*, description?, origin,
                         (capabilities|eventcapabilities), parameters, errors)>
   ...
]>
```

这个文件是 JVMTI 规范的**权威来源**——14993 行。所有 JVMTI 函数的签名、参数、错误码、capability 需求都在这里定义。这是一个典型的"单一事实来源"（Single Source of Truth）模式。

#### 工具怎么跑

生成器是一个 Java 类 `jvmtiGen`，接受 `-XSL` 参数指定 XSLT 样式表：

```
<bootstrap-jdk>/bin/java -cp tools/jvmti jvmtiGen \
    -IN src/hotspot/share/prims/jvmti.xml \
    -XSL src/hotspot/share/prims/jvmtiEnter.xsl \
    -OUT jvmtiEnter.cpp \
    -PARAM interface jvmti
```

同一份 `jvmti.xml`，用不同的 `.xsl` 产出不同的产物：

| XSLT 文件 | 输出 | 内容 |
|-----------|------|------|
| `jvmtiEnter.xsl` | jvmtiEnter.cpp (225KB) | JVMTI 函数入口桩代码——所有 `jvmti_*` 函数的 C++ 实现骨架 |
| `jvmtiEnterTrace.xsl` | jvmtiEnterTrace.cpp (635KB) | 带 trace 的版本——每个函数调用前后都打 LOG |
| `jvmtiHpp.xsl` | jvmtiEnv.hpp (15KB) | `JvmtiEnv` 类的头文件声明 |
| `jvmtiH.xsl` | jvmti.h (82KB) | 对外的 C 头文件——JVMTI agent 开发者的接口 |
| `jvmti.xsl` | jvmti.html (1.4MB) | **HTML 规范文档**——给 JVMTI agent 开发者看的 API 参考 |

最反直觉的是最后一行——**HTML 文档也是生成的**。OpenJDK 不单独维护一份 JVMTI 文档。规范和实现从同一份 XML 产生，保证文档和代码永远一致。如果你在 `jvmti.xml` 里改一个函数的参数，ADLC 不会参与（ADLC 只处理 `.ad`），但 jvmtiGen 会同时更新 C++ 实现骨架和 HTML 文档——没有"文档忘记更新"的问题。

#### 输出长什么样

`jvmtiEnter.cpp` 第 24 行标注：

```cpp
 // AUTOMATICALLY GENERATED FILE - DO NOT EDIT
```

然后是正式内容——比如 `_error_names` 数组：

```cpp
const char* JvmtiUtil::_error_names[] = {
  "JVMTI_ERROR_NONE",
  NULL, NULL, NULL, ...
};
```

这个数组的每一项对应 XML 里定义的一个错误码。`NULL` 表示这个错误码编号还没有对应的错误名——代码生成器按 JVMTI 规范分配编号，留了扩展位。

`jvmti.h` 是 agent 开发者用的 C 头文件：

```c
 /* AUTOMATICALLY GENERATED FILE - DO NOT EDIT */
#ifndef _JAVA_JVMTI_H_
```

agent 开发者写 `#include "jvmti.h"` 时，这个文件里的函数签名和 `jvmti.xml` 完全一致——因为它就是从同一份 XML 生成的。

### DTrace：`.d` 生成 C 宏

DTrace 的风格又不一样——它不用自研工具，直接调系统的 `/usr/bin/dtrace`。

#### 输入长什么样

`src/hotspot/os/posix/dtrace/hotspot.d` 用 D 语言描述探针：

```c
provider hotspot {
  probe class__loaded(char*, uintptr_t, void*, uintptr_t);
  probe class__unloaded(char*, uintptr_t, void*, uintptr_t);
  probe class__initialization__required(char*, uintptr_t, void*, intptr_t);
  probe vm__init__begin();
  probe vm__init__end();
  probe vmops__request(char*, uintptr_t, int);
  ...
};
```

每个 `probe` 声明了一个探测点——JVM 源码里可以插入 `HOTSPOT_CLASS_LOADED(...)` 这样的宏调用来触发热点。参数类型是 D 语言的（`char*`、`uintptr_t`），不是 C 语言的。

#### 工具怎么跑

```
/usr/bin/dtrace -h -o hotspot.h -s support/dtrace/hotspot.h.d
```

分两步：先用 `gcc -E` 把原始 `.d` 文件预处理成 `.h.d`（中间文件，存在 support/dtrace/ 下），再用系统 `dtrace` 命令 `-h` 生成 C 头文件。

Linux 上系统 DTrace 不可用怎么办？生成的代码用 **SystemTap SDT** 兼容层。看看实际输出：

#### 输出长什么样

`hotspot.h` 首行：

```c
/* Generated by the Systemtap dtrace wrapper */
```

下面是用 `<sys/sdt.h>` 实现的探针宏：

```c
#include <sys/sdt.h>

/* HOTSPOT_CLASS_LOADED ( char *, uintptr_t, void *, uintptr_t ) */
__extension__ extern unsigned short hotspot_class__loaded_semaphore
    __attribute__ ((unused)) __attribute__ ((section (".probes")));

#define HOTSPOT_CLASS_LOADED(arg1, arg2, arg3, arg4) \
DTRACE_PROBE4 (hotspot, class__loaded, arg1, arg2, arg3, arg4)
```

每条 `.d` 探针变成两个 C 内容：一个 `__attribute__((section(".probes")))` 的 semaphore 变量（用于探针启用/关闭检查），和一个 `DTRACE_PROBE` 宏。运行时 SystemTap 和 bpftrace 可以扫描 `.probes` 段找到所有探针点——这就是为什么 Linux 上也能用 DTrace 探针：不是原生 DTrace，而是 SystemTap SDT 兼容层。

### JDK 侧：数据驱动的 .java 生成

HotSpot 侧的生成器都在 `hotspot/variant-server/gensrc/` 下，产出 C++ 代码。但 `support/gensrc/` 下有 17 个模块、2323 个 `.java` 文件、总共 ~48MB——它们是用 Java 写的、给 JDK 标准库用的。

#### 最大的一坨：jdk.localedata（31MB，1423 个 .java）

这个模块没有手写代码——全部从 **CLDR**（Common Locale Data Repository，Unicode 国际化数据库）生成。

比如 `CurrencyNames_en_GB.java`：

```java
package sun.util.resources.ext;

import java.util.ListResourceBundle;

public final class CurrencyNames_en_GB extends sun.util.resources.LocaleNamesBundle {
    protected final Object[][] getContents() {
        return new Object[][] {
            { "EUR", "\u20AC" },
            { "GBP", "\u00A3" },
        };
    }
}
```

英国使用欧元吗？（实际是英镑为主，这个文件只是列出英国相关的货币名。）1423 个类似文件覆盖了几百种语言——没有人手写这些文件，全部由 CLDR 数据驱动生成。

#### java.base 的生成代码（4.7MB，221 个 .java）

这个模块的生成代码种类最多：

1. **CharacterData 系列**——`CharacterData00.java` ~ `CharacterData0E.java`、`CharacterDataLatin1.java`——每行都标注 `generated AUTOMATICALLY from a template file`。这些文件包含 Unicode 字符属性表（大小写转换规则、数字值、方向性等），从 Unicode 标准数据库生成。

2. **VarHandle 系列**——`VarHandleInts.java`、`VarHandleBytes.java` 等 15 个文件。VarHandle 是 Java 9 引入的内存访问 API，每种基本类型（int、long、byte、double...）需要一套变异实现，但算法相同——从模板生成而不是手写 15 遍。

3. **UnixConstants.java**——文件开头标注 `AUTOMATICALLY GENERATED FILE - DO NOT EDIT`。内容是 C 系统头文件的常量翻译：

```java
class UnixConstants {
    private UnixConstants() { }
    static final int O_RDONLY = 00 ;
    static final int O_WRONLY = 01 ;
    static final int O_RDWR = 02 ;
    static final int O_APPEND = 02000 ;
    ...
}
```

这些值从 C 头文件（`fcntl.h`）提取，保证 Java 侧的 `O_RDONLY` 和底层系统调用完全一致。JDK 11 在编译时跑 C 预处理器提取这些常量，而不是把值写死在 Java 源码里——不同平台的值可能不同（比如 AIX 和 Linux 的 `O_APPEND` 值就不一样）。

4. **version.java**——`javac -version` 输出的版本号：

```java
public final class version extends java.util.ListResourceBundle {
    protected final Object[][] getContents() {
        return new Object[][] {
            { "full", "11.0.32-internal+0-adhoc.root.jdk11u" },
            { "jdk", "11.0.32" },
            { "release", "11.0.32-internal" },
        };
    }
}
```

版本号在 configure 阶段写入 spec.gmk，构建时注入到这个文件——不是手写的。

#### jdk.charsets（4.4MB，102 个 .java）

字符编码器从 IANA 标准映射表生成。以 `IBM037.java` 为例——它包含一个 256 条目的编码表（`b2cTable`），把 IBM EBCDIC 037 编码的每个字节映射到 Unicode 字符：

```java
private final static String b2cTable =
    "\u00D8\u0061\u0062\u0063\u0064\u0065\u0066\u0067" +  // 0x80-0x87
    "\u0068\u0069\u00AB\u00BB\u00F0\u00FD\u00FE\u00B1" +  // 0x88-0x8f
    ...  // 共 256 个条目
    ;
```

这 256 个条目如果手写，只写错一个字符就会导致某些 EBCDIC 文本解码错误——几乎不可能人工排查。从标准映射表生成确保 100% 正确。

### 总结：为什么要生成代码

截至这一节结束，make 已经完成了 4 套 HotSpot 生成器 + JDK 侧 17 个模块的代码生成。回头问一个问题——**如果手写会怎样**？

| 生成器 | 手写代价 |
|--------|---------|
| ADLC | 179923 行 C++，包括 DFA 状态机（22702 行）、指令匹配（56612 行）。改一条指令的寄存器约束要同步 6 个文件——漏改一处，JIT 编译器悄悄生成错误机器码，debug 几乎不可能。 |
| JFR | 每个事件 struct 的字段/setter/writeData 方法全手写。新增一个 JFR 事件意味着在 C++ 里加 struct、加 setter、加序列化代码、加 enum 值——一个改动同步 4 处。 |
| JVMTI | 14993 行规范要手写 C++ 实现骨架 + C 头 + HTML 文档。3 处内容如果手写，规范更新后很容易出现文档和代码不一致。 |
| JDK locale | 1423 个 ResourceBundle 类，CLDR 数据升级要逐文件改。 |

共同模式：用声明式描述"是什么"，让工具生成"怎么做"。生成代码的头部都明确标注 `DO NOT EDIT`——不是建议，是声明：你改这些文件不会生效，下一次 make 会覆盖。

## 阶段三：编译 Java 类

代码生成完了。make 现在开始编译 Java 源代码——71 个模块的 `.java` 变成 `.class`。

但别以为这是简单的 `javac *.java`。OpenJDK 用的是**两阶段自举**：第一阶段造出来的 interim javac 现在登场——它用 bootstrap JDK 启动，编译真正的 JDK Java 源码。

### 不是用你系统的 javac

看 java.base 的编译命令（`_the.java.base_batch.cmdline`），精简后保留关键参数：

```
<bootstrap-jdk>/bin/java \
    -Xms64M -Xmx1600M -XX:ThreadStackSize=1536 \
    --limit-modules java.base,jdk.zipfs,java.compiler.interim,jdk.compiler.interim,jdk.javadoc.interim \
    --add-modules java.compiler.interim,jdk.compiler.interim,jdk.javadoc.interim \
    --module-path buildtools/interim_langtools_modules \
    -m jdk.compiler.interim/com.sun.tools.javac.Main \
    -g -source 11 -target 11 --doclint-format html5 -encoding ascii \
    -Xlint:all -Werror \
    --module-source-path \
        "support/gensrc/*:src/*/linux/classes:src/*/unix/classes:src/*/share/classes" \
    --system none -implicit:none \
    -d jdk/modules \
    -h support/headers.java.base.tmp \
    @_the.java.base_batch.tmp
```

几个关键点：

1. **启动者是 bootstrap JDK 的 java，不是 interim javac 直接跑**。`--limit-modules` 把 bootstrap JDK 的模块限定到最小（`java.base,jdk.zipfs`），然后把 `--module-path buildtools/interim_langtools_modules` 的 interim 模块加进来。bootstrap JDK 只提供 JVM 运行时，编译器的 class 全来自 interim 模块。`-m jdk.compiler.interim/com.sun.tools.javac.Main` 调用的是上一阶段造出来的 interim javac。

2. **`--system none`**——这是最关键的一个参数。正常情况下 javac 会从系统 JDK 的模块路径里找 `java.lang.String` 等标准类的定义——但编译 JDK 自身时，`java.lang.String` 还没被编译出来，不能从外面偷一个。`--system none` 关闭系统模块查找，javac 只从 `--module-source-path` 中读取源码——自己编自己。

3. **`-implicit:none`**——不隐式编译依赖。javac 默认遇到 `import java.util.List` 时会自动去找 `java.util.List` 的源码编译它——但在编译 JDK 时，模块依赖的编译顺序是 make 在外部规划好的，javac 只管编译传给它的文件列表（`@_the.java.base_batch.tmp` 里那 3043 个文件）。

4. **`-Xlint:all -Werror`**——所有 lint 检查全开，警告即错误。整个 71 模块的编译通过了零警告——这不是运气好，是 OpenJDK 的硬性代码质量约束。

5. **`-h support/headers.java.base.tmp`**——生成 JNI C 头文件。java.base 里有很多 `native` 方法声明，`javac -h` 从 `.class` 里反向提取C 函数签名，生成 `java_lang_Object.h`、`java_lang_System.h` 等头文件。**这一步必须在 C++ 编译之前完成**——因为后续编译 `libjvm.so` 时需要 include 这些头文件。

### 3043 个源文件从哪里来

java.base 是最大的模块，它编译的 3043 个文件来自四个路径，按优先级排序：

| 来源 | 数量 | 占比 | 说明 |
|------|------|------|------|
| share/classes | 2737 | 89.9% | 跨平台通用代码——大部分 JDK 标准库 |

### module-info.java 不是单一文件

Java 9 引入模块系统后，每个模块必须有一个 `module-info.java`。但 OpenJDK 的 `module-info.java` 不是手写完就行的——它要经过构建系统的**合并处理**。

拿 java.base 举例——它实际有三个输入源：

1. **share/classes/module-info.java**（385 行）——跨平台主体，手写的模块声明。包含所有 `exports`、`uses`、`provides` 声明。中间有一段注释很关键：

```java
// additional qualified exports may be inserted at build time
// see make/gensrc/GenModuleInfo.gmk
```

2. **unix/classes/module-info.java.extra**（1 行有效内容）：

```java
exports sun.nio.cs to java.desktop;
```

这个 `.extra` 文件是平台条件导出——在 Linux/macOS 上，`java.base` 需要把 `sun.nio.cs` 包额外暴露给 `java.desktop`（桌面字体需要字符集编码能力）。Windows 上没有这个额外的导出——Windows 有自己的字符集提供方式。

3. **生成的 module-info.java**（342 行）——构建系统用 `GenModuleInfo.gmk` 把 share 版本 + 平台 `.extra` 合并：

```java
// source file: file:///.../src/java.base/share/classes/module-info.java
//              file:///.../src/java.base/unix/classes/module-info.java.extra

module java.base {
    ...
    exports sun.nio.cs to java.desktop, jdk.charsets;
    ...
}
```

注意合并效果：share 版本原本只有 `exports sun.nio.cs to jdk.charsets;`，genmdule-info.gmk 把 `.extra` 里的 `exports sun.nio.cs to java.desktop;` 合并进来，生成最终文件里的 `exports sun.nio.cs to java.desktop, jdk.charsets;`——两个目标模块用逗号合并到一条声明里。

这种"主体 + 平台补丁"的设计意味着：同一个模块在不同平台上编译出不同的 module-info.java。Linux 上 java.base 导出 `sun.nio.cs` 给 `java.desktop`，Windows 上不导出——但是同一份源码。

### 73 → 71：消失的 2 个模块

`src/` 下有 73 个模块目录（`java.*` 22 个 + `jdk.*` 51 个），但 `build/*/jdk/modules/` 下只有 71 个。少的 2 个：

| 模块 | 平台 | 内容 |
|------|------|------|
| jdk.crypto.mscapi | Windows | MSCAPI 密钥库封装——调用 Windows CryptoAPI |
| jdk.crypto.ucrypto | Solaris | Oracle Solaris 硬件加密加速框架（Ucrypto） |

它们有一个共同特征：`src/` 下**没有 `share/` 目录**。两个模块的全部源码都在平台子目录里（`windows/` 和 `solaris/`）。构建系统在模块发现阶段根据当前平台过滤——Linux 上根本找不到这两个模块的任何源码。

mscapi 模块提供 `sun.security.mscapi.SunMSCAPI`——Windows 上的 RSA/SHA/ECDSA 签名和密钥存储（调用原生 `advapi32.dll`）。ucrypto 模块提供 Solaris SPARC 上的硬件加速加密（调用 `libucrypto.so`）。你在 Linux x86_64 上自然用不到它们。

### 零警告意味着什么

整个 71 模块的编译，javac 输出零警告。不是"警告不多"——是零。

所有模块编译都带 `-Xlint:all -Werror`——所有 lint 检查全开，警告即错误。71 个 `_the.*_batch.log` 文件全是 0 字节。编译成功意味着 OpenJDK 的 Java 源码没有触发任何 lint 警告，包括类型安全性、deprecation、序列化、try-with-resources 等所有检查。

这是一个硬性代码质量约束——比绝大多数商业项目的 lint 配置严格得多。

### 一个 `.cpp` 是怎么编译的

打开第一个被编译的文件 `abstractCompiler.o.cmdline`，看看 g++ 到底收到了什么命令。这是一行长达几千字符的完整命令行，按功能分组后是这样。

**预处理宏**——告诉编译器这段代码运行在什么平台上：

```
-D__STDC_FORMAT_MACROS  -D__STDC_LIMIT_MACROS  -D__STDC_CONSTANT_MACROS
-D_GNU_SOURCE -D_REENTRANT
-DLIBC=gnu -DSUPPORTS_CLOCK_MONOTONIC -DLINUX
-DVM_LITTLE_ENDIAN -D_LP64=1
-DASSERT -D_NMT_NOINLINE_
-DTARGET_ARCH_x86 -DAMD64 -DHOTSPOT_LIB_ARCH='"amd64"'
-DCOMPILER1 -DCOMPILER2
-DDTRACE_ENABLED -DINCLUDE_SHENANDOAHGC=0
```

`-DCOMPILER1 -DCOMPILER2` 同时定义——server 变体编译了 C1 和 C2 两个 JIT。`-DINCLUDE_SHENANDOAHGC=0` 说明 Shenandoah 在本构建中关闭——22 个 feature 里确实没有它。`-DASSERT` 是 slowdebug 的标志——`assert()` 在 product build 里会被 `-DNDEBUG` 关掉。

**C++ 语言控制**——HotSpot 用 C++98，但禁了很多标准特性：

```
-std=gnu++98
-fno-rtti              [禁用运行时类型信息]
-fno-exceptions        [禁用 C++ 异常]
-fvisibility=hidden    [默认隐藏所有符号]
```

这三个禁用的原因很关键：
- **`-fno-rtti`**：HotSpot 有自己的类型系统（Klass/Oop 层级），不需要 C++ 的 `typeid`/`dynamic_cast`。禁用 RTTI 节省每个对象的 vtable 空间。
- **`-fno-exceptions`**：HotSpot 用 `setjmp/longjmp` 加上资源区域（ResourceArea）自己管理异常路径。C++ 异常的 unwind 信息最少要几千字节每个函数，928 个文件加起来就是几十 MB。
- **`-fvisibility=hidden`**：默认不导出符号。配合 `-version-script=mapfile` 只导出白名单里的 JNI 入口，库的内部实现全藏起来。

**优化与代码生成**——slowdebug 的特殊配置：

```
-O0                          [不优化]
-fno-strict-aliasing         [允许 reinterpret_cast 类型的指针转换]
-fno-omit-frame-pointer       [保留帧指针，gdb 栈回溯可用]
-fcheck-new                  [检查 new 返回值非空]
-fstack-protector            [栈 canary 保护]
-fno-delete-null-pointer-checks  [不优化掉空指针检查]
-fno-lifetime-dse            [禁用死存储消除]
-fPIC -pipe -m64
```

`-O0` 是 slowdebug 的核心——release build 用 `-O2` 甚至 `-O3`，编译结果小很多但几乎不可调试。`-fno-delete-null-pointer-checks` 和 `-fno-lifetime-dse` 是专门针对 HotSpot 的——GCC 的默认优化有时会假设"你不可能在 placement new 之后还用旧的 this 指针"，但 HotSpot 的内存管理恰好依赖这种行为。

**警告选项**——比绝大多数项目都严格：

```
-Wall -Wextra -Wformat=2 -Wpointer-arith -Wsign-compare
-Wunused-function -Wundef -Wunused-value -Woverloaded-virtual -Wreturn-type
-Werror
-Wno-unused-parameter -Wno-unused
```

`-Werror` — 警告即错误。和 Java 编译一样，整个 HotSpot 的 C++ 代码也在零警告约束下。

**包含路径**——一个 `.cpp` 能看到的所有头文件目录：

```
-I gensrc/adfiles          [ADLC 生成的 ad_* 文件——阶段二的产物]
-I src/hotspot/share       [跨平台核心代码]
-I src/hotspot/os/linux    [Linux 平台层]
-I src/hotspot/os/posix    [POSIX 通用层]
-I src/hotspot/cpu/x86     [x86 CPU 层]
-I src/hotspot/os_cpu/linux_x86  [Linux+x86 交叉层]
-I support/modules_include/java.base  [JNI 头文件——阶段三的 javac -h 产物]
```

第一个 `-I` 是指向 `gensrc/adfiles/`——阶段二的 ADLC 产物。`abstractCompiler.cpp` 不直接 include ad 文件，但如果它调用的某个函数声明在了 `ad_x86.hpp` 里，编译器需要通过这个路径找到它。

### 链接：928 个 `.o` 拼成 `libjvm.so`

编译全部 928 个 `.cpp` 后，make 执行链接。`BUILD_LIBJVM_link.cmdline` 只有一行：

```
/usr/bin/g++ -Wl,--hash-style=both -Wl,-z,defs -Wl,-z,relro -Wl,-z,now
    -Wl,-z,noexecstack -Wl,-O1 -m64 -shared
    -Wl,-version-script=.../libjvm/mapfile
    -Wl,-soname=libjvm.so
    -o support/modules_libs/java.base/server/libjvm.so
    @.../libjvm/objs/_BUILD_LIBJVM_objectfilenames.txt
    -lm -ldl -lpthread
```

逐项拆开：

| 选项 | 作用 |
|------|------|
| `-Wl,--hash-style=both` | 同时生成 `.hash` 和 `.gnu.hash`，兼容旧 ld.so + 快速查找 |
| `-Wl,-z,defs` | 所有符号必须可解析——拒绝未定义引用 |
| `-Wl,-z,relro -Wl,-z,now` | Full RELRO——GOT 在加载后只读，防 GOT 覆写攻击 |
| `-Wl,-z,noexecstack` | 栈不可执行——NX 保护 |
| `-Wl,-O1` | 链接器优化——GC 无用段、合并相同函数 |
| `-m64 -shared` | 64 位共享库 |
| `-Wl,-version-script=mapfile` | 版本脚本控制导出符号（关键参数） |
| `-Wl,-soname=libjvm.so` | SONAME——运行时 dlopen 匹配此名 |
| `@_BUILD_LIBJVM_objectfilenames.txt` | response file——把 928 个 `.o` 路径传给 g++ |
| `-lm -ldl -lpthread` | 数学库、动态加载、POSIX 线程 |

关键参数是 `-Wl,-version-script=mapfile`。这个 mapfile 决定了外部程序（包括启动器 libjli）能看到 libjvm.so 的哪些符号。

### mapfile：5050 行的导出白名单

libjvm.so 的 mapfile 有 5050 行，结构是：

```
SUNWprivate_1.1 {
  global:
    AsyncGetCallTrace;
    jio_fprintf;  jio_printf;  jio_snprintf;  jio_vfprintf;  jio_vsnprintf;
    JNI_CreateJavaVM;  JNI_GetCreatedJavaVMs;  JNI_GetDefaultJavaVMInitArgs;
    JVM_FindClassFromBootLoader;  JVM_GetVersionInfo;
    JVM_Clone;  JVM_DoPrivileged;  JVM_handle_linux_signal;
    ...
    _ZTV10BarrierSet;  _ZTV11CollectedHeap;  ...     [C++ mangled vtable]
    gHotSpotVMStructs;  gHotSpotVMTypes;  ...         [Serviceability 元数据]
  local:
    *;
};
```

导出符号分四类：
1. **公开 JNI/JVM API**——约 174 个 `JVM_*` 函数 + 3 个 `JNI_*` 入口 + 几个 jio 函数。`JNI_CreateJavaVM` 是 libjli 用 `dlsym` 找的入口——不在白名单里，JVM 就启动不了。
2. **DTrace/AsyncGetCallTrace**——异步采样 profiling 钩子。
3. **C++ mangled 名**——以 `_ZTV` 开头的 4800 多个符号。为什么导出 vtable？因为 libjvmti agent、libjfr.so、libattach.so 等原生库需要直接访问 HotSpot 内部的 C++ 对象。vtable 让它们能找到方法入口——比走 JNI 快。
4. **Serviceability 元数据**——`gHotSpotVMStructs`、`gHotSpotVMTypes` 等全局变量。Serviceability Agent（HSDB、jstack、jmap）通过这些变量在外部进程读取 HotSpot 的内部结构——不需要 JVM 启动。

`local: *;` 一行屏蔽所有其他符号。配合 `-fvisibility=hidden` 编译选项，形成双保险——编译期和链接期都确保 libjvm.so 只暴露必要接口。

```
-DVERSION_BUILD=0
```
