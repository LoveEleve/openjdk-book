# Ch0.2 编译你自己的 JDK

## configure 是什么

上节 clone 下来的源码，现在要想办法让它跑起来。

编译任何东西的基本流程都一样：先检查你的机器上有没有需要的工具，再动手编译。JDK 的编译也不例外——`configure` 就是一个检测脚本。

它的全名叫 `autoconf`，本质上是在问你：

- 你用的是 gcc 还是 clang？版本够不够？（`make/autoconf/toolchain.m4:56`——最低 gcc 4.8）
- 有没有 X11 头文件？有没有 CUPS？（如果你的机器没有，可以跳过 GUI 相关的编译）
- 你的系统是 Linux 还是 macOS？CPU 是 x86_64 还是 aarch64？
- 你要编译什么级别的 JVM？slowdebug（带完整调试符号，适合本书）还是 release？

这一堆问题回答完之后，configure 把检测结果写到一个叫 `spec.gmk` 的文件里（`make/autoconf/spec.gmk.in:26-31`），然后整个 `make` 命令照着这个文件来编译。所以 configure 可以理解成一张体检表——体检通过了，才能开始动手。

## 跑 configure

在 `/data/workspace/jdk11u` 下面执行：

```
$ bash configure \
    --with-debug-level=slowdebug \
    --with-jvm-variants=server \
    --with-boot-jdk=/tmp/TencentKona-11.0.31.b1 \
    --with-native-debug-symbols=none \
    --with-memory-size=32768 \
    --disable-precompiled-headers \
    --disable-javac-server
```

每个选项是干什么的：

`--with-debug-level=slowdebug`

这是全书最重要的选项。它决定了三件事：编译优化级别设为 `-O0`（不优化，GDB 能看到每个变量）、断言 `ASSERT` 开启（`make/hotspot/lib/JvmFlags.gmk:85`：`#define ASSERT`）、完整符号表（`flags-cflags.m4:121`：`-g`）。这三个加起来，「用 GDB 看到 JVM 内部状态」才成为可能。

如果改成 `release`，`-O2` 优化会干掉很多中间变量，GDB 打过去什么也看不到。所以本书每一个 GDB 输出都基于 slowdebug——这是刻在环境声明里的。

`--with-jvm-variants=server`

JVM 有几种编译变体：server（默认，带 C2 JIT）、client（只有 C1）、minimal（没有 JIT，只有解释器）。选 server 是因为本书要讲 C2。

`--with-boot-jdk=/tmp/TencentKona-11.0.31.b1`

编译 JDK 需要先有一个能跑的 JDK。为什么？因为 Java 编译器 `javac` 本身是用 Java 写的——你需要一个已有的 JDK 来编译新的 javac，然后用刚编译出来的 javac 去编译 java.base、java.compiler 等模块。这叫做自举（bootstrap）。一般选 JDK 11 就行，版本要和你要编译的源码一致。

`--with-native-debug-symbols=none`

调试符号要不要单独拆成一个 `.debuginfo` 文件？选 `none` 的意思是直接嵌在二进制里（libjvm.so 会变大，但 GDB 加载时自动找到）。选 `external` 的话符号表会独立成一个文件，方便生产环境删掉。

`--with-memory-size=32768`

告诉构建系统你这台机器有 32GB 内存，它会在编译 Java 类库时合理分配并行度。

`--disable-precompiled-headers`

HotSpot 源码量很大（`src/hotspot/share/` 下 25 个子目录），预编译头能加速编译但容易导致奇怪的问题。关掉最稳。

`--disable-javac-server`

javac 编译服务器模式在某些机器上会卡住。关掉也用不了多几秒。

## configure 跑完之后

输出大概长这样：

```
A new configuration has been successfully created in
  /data/workspace/jdk11u/build/linux-x86_64-normal-server-slowdebug
  using configure arguments '--with-debug-level=slowdebug ...'.

Configuration summary:
  Debug level:    slowdebug
  HS debug level: slowdebug
  JVM variants:   server
  JVM features:   aot, cds, cmsgc, compiler1, compiler2, g1gc, jfr, ...
  OpenJDK target: linux-x86_64-normal-server-slowdebug
  Boot JDK:       /tmp/TencentKona-11.0.31.b1
  Tools summary:
    Boot JDK:       TencentKonaJDK 11.0.31
    Toolchain:      gcc (GCC 8.5.0)
    C Compiler:     Version 8.5.0
    C++ Compiler:   Version 8.5.0
  Build performance summary:
    Build jobs:     96
    Memory limit:   32768 MB
```

重点关注几个东西：

**Debug level: slowdebug**

这就是刚才 `--with-debug-level=slowdebug` 生效的结果。它会在 `spec.gmk` 里把 `JVM_CFLAGS` 设成 `-O0 -g -DASSERT` 的组合。后面 GDB 能停住 JVM、能打印变量、能看到完整调用栈，全是因为这一行配置。

**JVM features**

configure 根据你的系统环境和选的 variant 自动决定哪些 JVM 特性要编译进去。`compiler1`、`compiler2`、`g1gc`、`jfr`——这些名字你现在不用全认识，后面每卷展开时，对应的 feature 会在对应章节里讲到。现在只需要知道：configure 帮你选好了。

**Toolchain: gcc (GCC 8.5.0)**

configure 在 `make/autoconf/toolchain.m4` 里跑了一堆检测才找到的。它先通过 `gcc --version` 提取版本号（`toolchain.m4:463-465`），再确认版本 ≥ 4.8（`toolchain.m4:56`），最后设置好编译器和链接器的路径。如果这一步失败，这里会报 `GCC compiler is required`。

**Build jobs: 96**

96 核并行编译。configure 自动检测了你机器的 CPU 数量（`nproc`），用它来决定 `make -j` 的值。这就是为什么全编译 5 分钟就能跑完——如果是单核，得等半小时。

## configure 产出了什么

跑完 configure，根目录下多出一个 `build/` 目录：

```
build/
└── linux-x86_64-normal-server-slowdebug/
    ├── Makefile          ← 顶层的构建入口
    ├── spec.gmk          ← configure 的全部输出（核心文件）
    ├── configure.log     ← configure 的详细日志
    └── ...
```

`spec.gmk` 是核心。它记录了刚才所有检测的结果——GCC 路径、编译 flag、JVM feature 列表、模块列表。`make` 命令跑的时候，第一件事就是读这个文件，然后照单全收。

你可以翻一下 `spec.gmk`：

```
$ grep JVM_CFLAGS build/*/spec.gmk
JVM_CFLAGS := -fno-rtti -fno-exceptions -fvisibility=hidden ...
```

这里面 `-fno-rtti`（禁用 RTTI）、`-fno-exceptions`（禁用 C++ 异常）是 HotSpot 的标志性编译选项。为什么不用？后面编译章节会讲。现在只需要知道：configure 往 spec.gmk 里写的东西，决定了你的 JVM 长什么样。
