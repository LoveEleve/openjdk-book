# Ch0.2 编译你自己的 JDK

## 为什么需要 configure

编译一个 hello.c，你不需要任何前置步骤。`gcc hello.c -o hello`，两秒出结果。因为你知道这台机器上装了 gcc。

但如果你的程序引了 pthread：

```c
#include <pthread.h>
#include <stdio.h>

void* worker(void* arg) {
    printf("thread\n");
    return NULL;
}

int main() {
    pthread_t t;
    pthread_create(&t, NULL, worker, NULL);
    pthread_join(t, NULL);
    return 0;
}
```

`gcc thread.c -o thread` 在某些机器上会直接报错：

```
/tmp/ccXxxx.o: In function `main':
thread.c:(.text+0x2a): undefined reference to `pthread_create'
collect2: error: ld returned 1 exit status
```

pthread 不是 libc 的一部分。它需要显式传给链接器：`gcc thread.c -o thread -lpthread`。

即使你记住了 `-lpthread`，当你的项目有 1000 个 `.c` 文件、依赖十几个系统库、还要兼容三个平台时，手动敲这些检测命令不可行。你需要一套工具先确认「这台机器能不能编」，再决定「怎么编」。

JDK 的编译依赖远不止 pthread。它还依赖 X11、CUPS、Freetype 等本地库，需要确认 GCC 版本是否符合最低要求（≥ 4.8），需要知道目标 CPU 是 x86_64 还是 aarch64。进入 `src/hotspot` 目录——近千个 `.cpp` 文件，每一个都需要正确的 include 路径和编译宏。进入 `src/`——80 多个 Java 模块，需要先有一个能用的 JDK 来编译 javac 本身，再用刚编出来的 javac 编译整个类库。

这套自动化检测不是靠一个工具完成的，而是一条流水线。我把每个环节拆开讲。

### Autoconf 是什么

如果你写过 Makefile，应该知道一个头疼的问题：怎么让你的编译脚本在不同机器上都能跑？

张三的机器上 gcc 装在 `/usr/bin/gcc`，李四的机器上可能装在 `/usr/local/gcc-8/bin/gcc`。张三的系统有 X11 头文件，李四的没有。张三要编 64 位，王五要编 32 位。你不可能为每一种环境手写一份 Makefile。

GNU 解决这个问题的方式就是 Autoconf。它的思路是「先检测，后编译」——先跑一个脚本摸清这台机器的底细，然后根据底细生成适配的 Makefile。这个检测脚本就是 configure。

但你不需要手写 configure。Autoconf 提供了两样东西：宏语言和代码生成器。

### .m4 文件：宏定义

宏语言叫 m4。你用它描述检测逻辑，写出来的源文件后缀是 `.m4`。比如「检测系统有没有 pthread_create」这个逻辑，在 `libraries.m4` 里会用 Autoconf 提供的宏写成类似这样：

```
AC_CHECK_LIB(pthread, pthread_create, ...)
```

AC_CHECK_LIB 是 Autoconf 内置的宏。它会编译一个测试程序并尝试链接 pthread，成功就说明 libpthread 可用，失败就报错。

JDK 在 `make/autoconf/` 下放了十几组 `.m4` 文件，每组负责一类检测：

工具链相关：`toolchain.m4`。检测 GCC 还是 Clang，提取版本号，校验最低版本。`toolchain.m4:56` 设了 GCC 最低版本 4.8。

系统库相关：`libraries.m4`。检测 pthread、dl、数学库、X11、CUPS、Freetype 等等。`libraries.m4:106` 是 `-lpthread` 的检测点。

编译参数相关：`flags.m4`、`flags-cflags.m4`。根据 debug level 拼装 `-O0`/`-O2`、`-g`、`-DASSERT` 这些 flag。`flags-cflags.m4:121` 是 `-g` 的来源。

平台检测：`platform.m4`。识别 Linux/macOS/Windows，x86_64/aarch64。

JDK 专有：`boot-jdk.m4`（自举 JDK 检测）、`hotspot.m4`（JVM 特性检测）、`jdk-options.m4`（JDK 专用配置项）。

### configure.ac：总装模板

这些 `.m4` 文件不是独立运行的。它们被一份叫 `configure.ac` 的总装模板通过 `m4_include` 指令拉进来：

```m4
# configure.ac 中的 include 列表（部分）
m4_include([toolchain.m4])
m4_include([libraries.m4])
m4_include([flags.m4])
m4_include([platform.m4])
...
```

`configure.ac` 还定义了检测流程——先初始化环境变量（`BASIC_INIT`），再检测平台（`PLATFORM_SETUP`），再检测工具链（`TOOLCHAIN_SETUP`），再检测系统库（`LIB_SETUP`），逐个环节串起来。

这个流程对应我们手动编译的思考过程：先看看这台机器是 Linux 还是别的（平台），再看看装了哪个编译器（工具链），再确认需要的库都在不在（库检测）。

### 生成 configure 脚本

`.m4` 是源文件，`configure.ac` 是总装模板。把它们喂给 Autoconf，吐出来的是 `generated-configure.sh`——4.5MB，**135,856 行** shell 代码。

这个数字值得停一下。源头的 `configure.ac` 只有三百多行，加上十几个 `.m4` 文件，被 Autoconf 展开成了 13 万行。这和 gcc 把几十行 C 编译成几千条机器指令是一回事——你写的是高度浓缩的宏，机器吐出来的是展开后的完整代码。

打开 `generated-configure.sh`，第二行就写明了它的用途：

```
# Guess values for system-dependent variables and create Makefiles.
# Generated by GNU Autoconf 2.71 for OpenJDK openjdk.
```

「猜测系统相关的各种变量值，然后生成 Makefile」——这就是它干的全部事情。具体来说：

- 文件里有 **2144 处** `checking ...`。你跑 configure 时屏幕上哗哗滚的 `checking for gcc... yes`、`checking whether the C compiler works... yes` 这些行，每一行都对应这里的一个检测点。
- 前面分析的那些 `.m4` 检测逻辑全被展开进了这一个文件：gcc 识别（77 处 `Free Software Foundation` 的 grep）、Boot JDK 探测（1373 处 `BOOT_JDK`）、系统库、平台架构。
- 所有检测跑完，结果由它写进 `spec.gmk`（文件里 16 处引用）。

一句话：`generated-configure.sh` 才是真正跑这 2144 个检测、把结果写进 `spec.gmk` 的脚本。但你并不直接运行它——后面会看到，你敲的 `configure` 命令要经过两层包装，才最终调到它。

这个脚本不是仓库里现成的。jdk11u 特意没有提交它——你 clone 下来，`make/autoconf/` 下根本找不到 `generated-configure.sh`。第一次跑 configure 时，驱动器才现场调用 Autoconf 把它生成出来。下面这三行就是第一次运行时打印的：

```
Runnable configure script is not present
Generating runnable configure script at build/.configure-support/generated-configure.sh
Using autoconf at /usr/bin/autoconf [autoconf (GNU Autoconf) 2.71]
```

它告诉你：找不到现成的 `generated-configure.sh`，正在用 autoconf 2.71 现场生成，生成完缓存在 `build/.configure-support/` 下，之后（只要你不改 `.m4` 源文件）就不用再生成了。

你可能会问：早期的 OpenJDK 不是把 `generated-configure.sh` 直接提交进 git、clone 下来就有吗？没错，以前是这样。但有一次改动把它从仓库里删掉了，改成每次本地用 autoconf 现场生成。这件事不用信我，自己验三条命令就清楚了。

先确认这个脚本确实不在仓库里：

```
$ ls make/autoconf/generated-configure.sh
ls: cannot access 'make/autoconf/generated-configure.sh': No such file or directory
```

再看 git 历史，它是被哪条提交删掉的：

```
$ git log --oneline -1 -- make/autoconf/generated-configure.sh
a4630a5b3b 8195689: Remove generated-configure.sh and instead use autoconf
```

提交信息写得明明白白：「Remove generated-configure.sh and instead use autoconf」——移除预生成脚本，改用 autoconf 现场生成。

最后，那个第一次 configure 时冒出来的 `build/` 目录，为什么 GitHub 上从来看不到？因为它在 `.gitignore` 里：

```
$ head -1 .gitignore
/build/
```

`.gitignore` 第一行就是 `/build/`。它是本地编译输出目录，永远不会提交。所以你在 GitHub 上看不到 build/ 完全正常——它只在你本地跑过 configure 之后才出现，里面装的是 `generated-configure.sh`、稍后的 `spec.gmk`、`Makefile`，以及 make 编译出来的全部产物。

### 入口、跳板和驱动器

你实际运行的不是 `generated-configure.sh`，而是项目根目录下的 `configure`。从你敲命令到真正开始检测，中间隔着两层包装。

第一层，根目录的 `configure`，只有 35 行，是个跳板：

```bash
# jdk11u/configure（完整代码）

# 1. 算出顶层目录的绝对路径
this_script_dir=`dirname $0`
this_script_dir=`cd $this_script_dir > /dev/null && pwd`

# 2. 调用第二层脚本，传 CHECKME 做身份验证
bash ${-:+-$-} -c ". $this_script_dir/make/autoconf/configure" \
    $this_script_dir/configure CHECKME $this_script_dir "$@"
```

它算出项目根目录的绝对路径，然后调用第二层 `make/autoconf/configure`，并传一个 `CHECKME` 令牌。

第二层，`make/autoconf/configure`（12KB），是个驱动器。它开头先验证 CHECKME 令牌：

```bash
# make/autoconf/configure 开头 4 行
if test "x$1" != xCHECKME; then
  echo "ERROR: Calling this wrapper script directly is not supported."
  echo "Use the 'configure' script in the top-level directory instead."
  exit 1
fi
```

确认是从顶层 `configure` 调过来的之后，它检查 `generated-configure.sh` 在不在——不在就调 autoconf 生成（就是上一节那三行打印），在了就直接运行它。第三层 `generated-configure.sh` 才是真正干活的脚本。

所以完整的调用链是三层：

```
configure（1.6KB 跳板）
  └→ make/autoconf/configure（12KB 驱动器：验证 CHECKME + 确保 generated-configure.sh 存在）
       └→ generated-configure.sh（4.6MB：autoconf 真正产物，干所有检测）
```

那个 `CHECKME` 令牌的意义就在这里：你不能绕过顶层 `configure` 直接运行第二层脚本，否则 TOPDIR 路径就丢了。

### 为什么必须 bash configure

你可能注意到，运行 configure 得写成 `bash configure`，直接 `./configure` 会报错。原因在文件权限上。

看一眼根目录 `configure` 的权限：

```
$ ls -l configure
-rw-r--r-- 1 root root 1649 Jun 21 18:35 configure
```

`-rw-r--r--`——可读可写，但没有任何执行位（没有 `x`）。这就解释了三种敲法的不同结果：

- 敲 `./configure`：内核要直接 exec 这个文件，但它没有执行权限，报 `Permission denied`（权限不够）
- 敲 `configure`（不带 `./`）：shell 去 `$PATH` 里找，当前目录不在 PATH 上，报 `command not found`（没有这个命令）
- 敲 `bash configure`：真正被 exec 的是 `bash`（它有执行权限），bash 只需要读这个文件然后解释执行。文件有读权限（`r--`），所以能跑

关键区别：直接执行（`./configure`）需要文件本身有 `x` 位；用解释器执行（`bash configure`）只需要 `x` 在 bash 上、`r` 在脚本上。

> 在官方 OpenJDK 仓库里 `configure` 是带执行位的（755），可以直接 `./configure`。我们这份 checkout 是 644，执行位在拷贝/解压过程中丢了。`chmod +x configure` 之后 `./configure` 也能直接跑。

### 跑完之后

configure 跑完，所有检测结果写入 `spec.gmk`。make 随后读这份清单，照单编译。

### Autoconf 实践：一个最小的例子

上面说的 m4、`configure.ac`、Autoconf 这些概念，用一个小项目跑一遍就清楚了。

我们在 `demo-autoconf/` 目录下创建这个项目。拿刚才的多线程 C 程序做素材。目标是：在不同机器上，不用手动传 `-lpthread`，让 configure 自动检测 pthread 是否存在，如果存在就自动加上。

先写 `configure.ac`，就 5 行：

```
AC_INIT([hello], [1.0])
AC_PROG_CC
AC_CHECK_LIB([pthread], [pthread_create], [],
             [AC_MSG_ERROR([pthread required])])
AC_CONFIG_FILES([Makefile])
AC_OUTPUT
```

逐行解释：

`AC_INIT([hello], [1.0])` 告诉 Autoconf 这个项目叫 hello，版本 1.0。每个 `configure.ac` 的第一行必须是它。

`AC_PROG_CC` 检测 C 编译器。Autoconf 会依次尝试 gcc、cc、clang，找到就设变量 `CC`。

`AC_CHECK_LIB([pthread], [pthread_create], [], [AC_MSG_ERROR([pthread required])])` 检测 pthread 库。前两个参数是库名和函数名。第三个参数是成功时的动作（空，默认把 `-lpthread` 加到 `LIBS` 变量）。第四个参数是失败时的动作（报错退出）。这就是我们手动加 `-lpthread` 的自动化版本。

`AC_CONFIG_FILES([Makefile])` 告诉 Autoconf：检测完用结果填充 `Makefile.in` 模板，生成 `Makefile`。

`AC_OUTPUT` 全部完成，输出结果。

再写一个 `Makefile.in` 模板。`@CC@` 和 `@LIBS@` 是占位符，configure 会用检测结果替换它们：

```
CC = @CC@
CFLAGS = -g -O2
LIBS = @LIBS@
hello: hello.c
	$(CC) $(CFLAGS) -o hello hello.c $(LIBS)
```

三样东西齐了：`hello.c`（源码）、`configure.ac`（检测逻辑）、`Makefile.in`（编译模板）。运行 Autoconf：

```
$ autoconf
```

5 行的 `configure.ac` 被 Autoconf 展开成了 4329 行的 `configure` 脚本。这和我们刚才分析的 JDK configure 生成机制完全一致——`.m4` 宏语言被翻译成了可执行的 shell 代码。

跑这个生成的脚本：

```
$ ./configure
checking for gcc... gcc
checking whether the C compiler works... yes
checking for pthread_create in -lpthread... yes
configure: creating ./config.status
config.status: creating Makefile
```

关键的两行：configure 找到了 gcc，确认了 pthread 可用。打开生成的 `Makefile`：

```
CC = gcc
CFLAGS = -g -O2
LIBS = -lpthread
hello: hello.c
	$(CC) $(CFLAGS) -o hello hello.c $(LIBS)
```

`@CC@` 被替换成了 `gcc`，`@LIBS@` 被替换成了 `-lpthread`。这两个值不是手动填的——是 configure 在这台机器上实际检测出来的。

最后编译运行：

```
$ make
gcc -g -O2 -o hello hello.c -lpthread
$ ./hello
thread
hello from autoconf demo
```

5 行 `configure.ac`，驱动了一套跨机器的自动检测系统。JDK 的编译用的完全一样的机制——只是 `configure.ac` 不是 5 行，是三百多行；检测的也不止 pthread，还有 GCC/MSVC/Xcode 工具链、X11/CUPS/Freetype 图形库、CPU 架构、操作系统版本。

理解了这个小例子，后面看 JDK 的 configure 输出，每一行 `checking` 你都知道它在做什么。

## 执行 configure

### 怎么知道有哪些选项

configure 支持几十个选项。你不用记——`bash configure --help` 会全部列出来。

输出分成几大块。前几块是所有 Autoconf 项目都长一样的标准模板，最后两块才是 JDK 专有的：

| 块 | 内容 | 来源 |
|---|---|---|
| Configuration | `--help`、`--version`、`--quiet`、`--cache-file` 等元选项 | Autoconf 标准 |
| Installation directories | `--prefix`、`--bindir` 等安装路径（JDK 基本用不上） | Autoconf 标准 |
| System types | `--build`、`--host`、`--target`（交叉编译） | Autoconf 标准 |
| Optional Features | `--enable-X` / `--disable-X` 一族 | JDK 的 `.m4` |
| Optional Packages | `--with-X` / `--without-X` 一族 | JDK 的 `.m4` |

JDK 自己的选项全在最后两块——也就是前面 `.m4` 文件里那些 `AS_HELP_STRING` 展开出来的。

### 为什么只带一部分选项

每个选项后面方括号里都有一个 `[默认值]`。这是看懂 configure 命令的关键：configure 对每个选项都有合理默认，你只在想偏离默认时才需要写它。

默认值分两种。一种是固定的：

```
--with-debug-level      set the debug level (release, fastdebug, slowdebug, optimized) [release]
--with-jvm-variants     JVM variants ... (server,client,minimal,core,zero,custom) [server]
--disable-precompiled-headers   disable using precompiled headers when compiling C++ [enabled]
--disable-javac-server  disable javac server [enabled]
```

另一种是自动探测的——`[probed]`（探测）、`[guessed]`（猜测）、`[calculated]`（计算）：

```
--with-boot-jdk    path to Boot JDK (used to bootstrap build) [probed]
--with-num-cores   number of cores in the build system [probed]
--with-memory-size memory (in MB) available in the build system [probed]
--with-jobs        number of parallel jobs to let make run [calculated based on cores and memory]
```

`[probed]` 的意思是：你不指定，configure 自己去探。`--with-boot-jdk` 会扫 PATH 和 JAVA_HOME 找现成的 JDK，`--with-num-cores` 读 `/proc/cpuinfo`，`--with-memory-size` 读 `/proc/meminfo`。

所以你只在两种情况下才需要显式写一个选项：你想要的值和默认不一样，或者自动探测靠不住、你想盖掉它。其余几十个选项全用默认——这就是为什么真实的 configure 命令只有一把，不是一整列。

### 这条命令

本书用的 configure 命令：

```bash
bash configure \
    --with-debug-level=slowdebug \
    --with-jvm-variants=server \
    --with-boot-jdk=./jdk-11.0.17 \
    --with-native-debug-symbols=none \
    --with-memory-size=32768 \
    --disable-precompiled-headers \
    --disable-javac-server
```

逐个对照 `[默认值]`，会发现其中有两个其实是多余的——它们把值设成了和默认一样：

`--with-debug-level=slowdebug` —— **必要**。默认是 `release`，但 release 编出来的 JVM 优化全开、没有调试符号，没法用 GDB 调。本书每一个 GDB 输出都依赖 slowdebug，所以必须改。

`--with-jvm-variants=server` —— **多余**。默认就是 `server`，写不写一个样。

`--with-boot-jdk=./jdk-11.0.17` —— **建议写**。默认是 `[probed]`，但如果你 PATH 和 JAVA_HOME 上没有合适的 JDK，探测会失败、configure 直接报错退出。显式指到上一章解压的那个 JDK 更稳。

`--with-native-debug-symbols=none` —— **有用**。默认 `[varying]` 会随 debug level 变，这里钉死成 none——调试符号直接嵌进二进制，不拆成独立的 `.debuginfo` 文件。

`--with-memory-size=32768` —— **多余**。默认 `[probed]`，configure 自己会读 `/proc/meminfo`。

`--disable-precompiled-headers` —— **必要**。默认 `[enabled]`，关掉它。预编译头能加速编译，但跨 GCC 版本容易出诡异问题。

`--disable-javac-server` —— **必要**。默认 `[enabled]`，关掉它。javac 常驻编译服务在某些环境会卡死。

去掉两个多余的，最精简的等价命令是四个选项：

```bash
bash configure \
    --with-debug-level=slowdebug \
    --with-boot-jdk=./jdk-11.0.17 \
    --disable-precompiled-headers \
    --disable-javac-server
```

本书后面沿用完整的七选项写法（多写两个无害，且意图更明确），但你心里要清楚：真正起作用、真正偏离默认的，只有这四个。

上面这些选项里，有三个背后藏着概念，值得展开。

### debug-level：为什么必须 slowdebug

`--with-debug-level` 有四个取值。`jdk-options.m4:53-56` 的注释把它们的区别讲得很干脆：

| 级别 | 优化 | 调试符号 -g | 断言 | HotSpot 内部名 | 场景 |
|------|------|:---:|:---:|------|------|
| release | 全开（-O3/-O2） | ✗ | ✗ | product | 交付给用户 |
| optimized | 全开 | ✗ | ✗ | optimized | 测试实验性功能 |
| fastdebug | 全开 | ✓ | ✓ | fastdebug | 日常开发调试 |
| slowdebug | 全关（-O0） | ✓ | ✓ | debug | 深度调试（本书） |

这张表的每一格都能在源码里找到出处：

- 优化级别在 `flags-cflags.m4:354-377`：slowdebug 把所有优化降到 `-O0`
- 断言宏在 `JvmFlags.gmk:69-86`：fastdebug 和 slowdebug 都定义 `-DASSERT`；release 定义 `-DNDEBUG`（关掉断言）外加 `-DPRODUCT`
- HotSpot 内部名在 `jdk-options.m4:85-91`：release 翻译成 product，slowdebug 翻译成 debug

本书选 slowdebug，就为了表里那两格。`-O0` 保证每个局部变量都不会被优化掉，GDB 打印任何变量都看得到真实值；`-DASSERT` 保证 `#ifdef ASSERT` 包着的检查代码和附加字段都在二进制里，你看到的数据结构是完整的。release 把这两样都拿掉了——优化重排了代码、复用了寄存器、消除了中间变量，GDB 进去只剩一堆对不上号的汇编。

> fastdebug 也开了 `-g` 和 `-DASSERT`，但它的优化是开的（-O2）。本书坚持 slowdebug 而非 fastdebug，就是为了 `-O0`——优化一开，单步调试时行号会乱跳，变量会「被优化掉」。

### jvm-variants：server 是什么

`--with-jvm-variants` 决定编出来的 JVM 带哪些组件。`hotspot.m4:76-82` 列了六种：

- **server**：解释器 + C1 + C2 两个 JIT 编译器（分层编译）
- **client**：解释器 + C1（没有 C2）
- **core**：只有解释器，没有 JIT
- **minimal**：client 的精简版，砍掉可选特性
- **zero**：C++ 写的解释器，没有 JIT
- **custom**：什么默认特性都不带

server 和 client 的唯一区别就是 C2。本书选 server，因为 C2 编译器后面有专门的卷要讲。

选了 server，configure 会自动把一大堆 JVM 特性编进去。`hotspot.m4:546` 定义了 server 变体的特性集：两个编译器（compiler1、compiler2）+ 所有 GC（g1gc、parallelgc、serialgc、cmsgc、epsilongc、shenandoahgc、zgc）+ 诊断设施（jvmti、management、nmt、services、jni-check、vm-structs）+ cds + jvmci/aot/graal。这就是为什么后面 configure 输出里 `JVM features:` 那一行会列出这么长一串——你没手动选，是 server 变体默认带进来的。

### boot-jdk：自举

`--with-boot-jdk` 指向一个已经能用的 JDK。为什么编 JDK 需要先有一个 JDK？因为 `javac` 本身是用 Java 写的。要编出新的 javac，你得先有一个能跑的 javac 来编译它。这种「用旧版本编译新版本」的循环叫**自举**（bootstrap），那个先于目标 JDK 存在的 JDK 就叫 boot JDK。

如果你不写 `--with-boot-jdk`，configure 会自己找。`boot-jdk.m4:319-342` 定义了五级探测顺序，从「最确定」到「最靠猜」：

1. `--with-boot-jdk` 命令行参数（你明确指定的）
2. `$JAVA_HOME` 环境变量
3. macOS 的 `/usr/libexec/java_home`
4. PATH 上 `java`/`javac` 指向的真实 JDK
5. 系统标准安装位置（Linux 的 `/usr/lib/jvm` 等）

每个候选都要过三关：`bin/java` 在、`bin/javac` 在（只有 java 没有 javac，说明那是个 JRE 不是 JDK）、`java -version` 的版本号在可接受范围内。

有个坑要注意：如果你写了 `--with-boot-jdk` 但路径无效，configure **不会**退回去试 JAVA_HOME 或 PATH，而是当场报错退出（`boot-jdk.m4:326-329`）。设计逻辑是：你都明确指定了，那就是你的意图，错了该报错，而不是悄悄用别的顶上。

### configure 的检测顺序

configure 不是随便检测的，有固定顺序。完整流程有几十个宏调用，但你只需要记住四个关键检测，按它们在 `configure.ac` 里的出现顺序：

1. **平台**（`configure.ac:84`）—— 先搞清楚这台机器是什么。Linux 还是 macOS？x86_64 还是 aarch64？后面所有检测都依赖这个。
2. **Boot JDK**（`configure.ac:147`）—— 找到那个用来自举的 JDK。
3. **工具链**（`configure.ac:178`）—— 检测 C/C++ 编译器，确认是 GCC 还是 Clang、版本够不够。
4. **系统库**（`configure.ac:222`）—— pthread、dl、X11、CUPS、Freetype 这些在不在。

这个顺序有它的道理：先知道平台（决定用哪套检测逻辑），再确认能自举（没有 boot JDK，后面 Java 代码全编不了），再检测编译器（C++ 代码靠它），最后查系统库（链接时要用）。任何一步失败，configure 当场停下报错，不会带着残缺的配置往下走。

### 跑起来：第一次大概率会失败

把上面的命令敲下去，在一台刚装好系统的机器上，大概率不会一次成功。我这台就卡住了：

注意 configure 没有自作主张帮你装东西——它只告诉你缺什么、怎么补。错误信息里那条命令就是补法。照着装上（JDK 11 还要 cups、fontconfig、freetype、alsa 几个开发库，一并装了）：

### configure 产出了什么

跑成功后回头看 `build/` 目录，会发现它现在装着两样不同的东西。用 `ls -A`（带上隐藏项）看：

```
$ ls -A build/
.configure-support  linux-x86_64-normal-server-slowdebug
```

第一个 `.configure-support`（点开头，隐藏目录）你已经见过了——前面跑 `--help` 时生成的，里面就一个 4.5MB 的 `generated-configure.sh`。它是 autoconf 现场生成的那个脚本，所有配置共享一份，跟你这次选的 slowdebug、server 没关系。

第二个 `linux-x86_64-normal-server-slowdebug` 才是这次 configure 的成果——一个独立的配置目录。名字由你的选项拼出来：操作系统(linux) - CPU(x86_64) - JDK变体(normal) - JVM变体(server) - debug级别(slowdebug)。如果你换一组选项再 configure 一次（比如把 `--with-debug-level` 从 slowdebug 改成 release），build/ 下会多出一个 `linux-x86_64-normal-server-release/`，两套配置井水不犯河水、互不覆盖。

进到这个配置目录里看：

```
$ ls build/linux-x86_64-normal-server-slowdebug/
bootcycle-spec.gmk  buildjdk-spec.gmk  compare.sh
configure.log       configure-support  Makefile  spec.gmk
```

`spec.gmk` 是核心，35KB，记录了刚才所有检测的结果——编译器路径、每一个编译 flag、JVM 特性列表、模块清单。make 跑起来第一件事就是读它。

随手翻一下里面的 `JVM_CFLAGS`：

```
$ grep "JVM_CFLAGS :=" build/*/spec.gmk
JVM_CFLAGS := ... -pipe -fno-rtti -fno-exceptions -fvisibility=hidden ...
```

`-fno-rtti`（禁用运行时类型识别）和 `-fno-exceptions`（禁用 C++ 异常）是 HotSpot 从 C++ 里剥出来的两个标志性选择——JVM 不用 `dynamic_cast`、不用 `try/catch`，而是自己实现了完整的类型系统和错误处理链。这是后面所有分析的物理基础。

其余几个文件一句话带过：`configure.log`（82KB）是完整检测日志，那 2144 行 `checking` 全在里面；`Makefile` 是 make 的入口；`compare.sh` 是构建结果对比脚本；`bootcycle-spec.gmk`、`buildjdk-spec.gmk` 是自举和交叉编译用的变体配置。

还有一个 `configure-support/` 子目录——注意它没有点，和顶层那个 `.configure-support` 不是一回事。顶层那个装的是「所有配置共享的 autoconf 脚本」，这个装的是「本次配置专属的支持文件」：

```
$ ls build/linux-x86_64-normal-server-slowdebug/configure-support/
classes.jsa  config.log  config.status
```

`config.log` 和 `config.status` 是 autoconf 的标准产物——前者是所有测试编译的原始日志（比 `configure.log` 更底层、更啰嗦），后者是一个能重新生成 spec.gmk/Makefile 的脚本（`make reconfigure` 就靠它）。`classes.jsa` 则是 JDK 特有的：boot JDK 的 CDS（类数据共享）归档，用来加速自举编译期间反复启动的那些 javac/java 进程。

configure 通过了，`spec.gmk` 就绪。下一步 make 读这份清单，把代码变成能跑的 JDK。

## make 是什么

上一步 configure 体检完，在 `build/linux-x86_64-normal-server-slowdebug/` 下留了两样东西：一份 `spec.gmk`（所有检测结果），一个 `Makefile`（构建入口）。现在轮到 make 上场。

make 是一个自动化构建工具。它读一个叫 `Makefile` 的文件，按里面写的规则，把源码一步步编译、链接成最终产物。但 make 真正的价值不在「怎么编」——那不过是几条 gcc 命令——而在判断「该编什么」。

### 只编该编的：增量构建

假设你的项目有 1000 个 `.c` 文件，你只改了其中一个。重新编译时要不要把 1000 个全编一遍？显然不该——只编那一个、再重新链接就够了。

但「哪些该重编」靠人脑记不住，尤其文件之间还有依赖（改一个头文件，所有引用它的 `.c` 都得重编）。make 就是来管这个的。它的判断标准简单粗暴：**比时间戳**。一个目标文件如果比它依赖的源文件旧（或者根本不存在），就重新生成；否则跳过。

这就是为什么 JDK 全量编译要几分钟，但你改一行 HotSpot 源码后重编只要十几秒——make 只编了那一个变化的文件，其余几百个 `.o` 原封不动。

### 一条规则的三要素

Makefile 的核心是「规则」。一条规则长这样：

```
目标: 依赖1 依赖2 ...
	recipe（生成目标的命令）
```

三个要素：

- **目标（target）**：要生成的东西，通常是个文件名（比如 `hello`、`main.o`）
- **依赖（prerequisites）**：生成这个目标需要哪些文件。make 拿目标和依赖比时间戳，决定要不要重新生成
- **recipe**：实际执行的 shell 命令，告诉 make 怎么从依赖造出目标

有个经典的坑：**recipe 必须用 TAB 缩进，不能用空格**。这是 make 几十年的历史包袱，用空格会报 `missing separator` 错误。

### 变量和自动变量

Makefile 支持变量，避免重复：

```
CC = gcc
CFLAGS = -Wall
```

用 `$(CC)`、`$(CFLAGS)` 引用，改编译器只动一处。

还有几个常用的「自动变量」，在 recipe 里指代当前规则的目标和依赖：`$@` 是当前目标，`$<` 是第一个依赖，`$^` 是全部依赖。

另外，第一个出现的目标是「默认目标」——你敲 `make` 不带参数时编的就是它。不对应实际文件的目标（比如 `clean`）要声明成 `.PHONY`，否则万一目录里真有个叫 clean 的文件，make 会以为它已经「生成」过了，什么都不做。

### 一个最小的例子

光说不练没用。建一个三文件的小项目：`main.c` 调用 `util.c` 里的函数，两者都包含 `util.h`。

```c
// main.c
#include <stdio.h>
#include "util.h"
int main() {
    printf("1 + 2 = %d\n", add(1, 2));
    return 0;
}
```

```c
// util.c
#include "util.h"
int add(int a, int b) { return a + b; }
```

配一个 Makefile（recipe 行都是 TAB 缩进）：

```
CC = gcc
CFLAGS = -Wall

hello: main.o util.o
	$(CC) $(CFLAGS) -o hello main.o util.o

main.o: main.c util.h
	$(CC) $(CFLAGS) -c main.c

util.o: util.c util.h
	$(CC) $(CFLAGS) -c util.c

.PHONY: clean
clean:
	rm -f hello *.o
```

它描述了三条规则：`hello` 依赖两个 `.o`，每个 `.o` 依赖对应的 `.c` 和 `util.h`。第一次 `make`：

```
$ make
gcc -Wall -c main.c
gcc -Wall -c util.c
gcc -Wall -o hello main.o util.o
$ ./hello
1 + 2 = 3
```

三步都跑了：编 main.o、编 util.o、链接 hello。现在见证 make 的核心——再敲一次 `make`：

```
$ make
make: 'hello' is up to date.
```

什么也没编。所有目标都比依赖新，make 判断无事可做。

改一下 `util.c`（这里用 `touch` 模拟修改，只更新时间戳），再 `make`：

```
$ touch util.c
$ make
gcc -Wall -c util.c
gcc -Wall -o hello main.o util.o
```

**只重编了 util.o，再重新链接**——main.o 没动，因为 main.c 没变。这就是增量构建。

再看依赖追踪。这次改的是头文件 `util.h`（main.o 和 util.o 都依赖它）：

```
$ touch util.h
$ make
gcc -Wall -c main.c
gcc -Wall -c util.c
gcc -Wall -o hello main.o util.o
```

**两个目标文件都重编了**——因为规则里写明了它们都依赖 `util.h`，头文件一变，引用它的全得重来。

最后 `make clean`（那个 `.PHONY` 目标）：

```
$ make clean
rm -f hello *.o
```

JDK 的编译就是这套机制放大到极致。make 读 configure 生成的那个 `Makefile`（外加 `spec.gmk` 里几万个变量），按依赖关系把近千个 `.cpp` 和成千上万个 `.java` 一步步编出来。

这里要澄清一个常见误解：make 自己**不编译**任何东西，它是个调度器。`.cpp` 文件它调 `g++`，`.java` 文件它调 `javac`——就像你 demo 里 recipe 写的 `$(CC) $(CFLAGS) -c main.c`，真正跑的是 gcc，make 只负责判断「该编什么、按什么顺序、调用哪个工具」。（顺带一提：编 Java 用的 `javac`，正是前面 boot JDK 自举出来的那个——这条线后面真跑 make 时会看到。）

库之间也是这样一层层依赖：libjvm.so 在最底层，libjava.so 依赖它，libzip.so、libnio.so 又依赖 libjava——你刚在玩具项目里看到的「改一个头文件、引用它的全得重来」，在 JDK 里就是一长串库的级联重编。

这套增量构建，正是本书核心方法论的引擎：「改一行源码 → 重编 → 用 GDB 验证」之所以能几秒一轮，靠的就是 make 只动你改过的那个文件。后面有专门一章会把这个循环打磨到极致——换上比 make 更快的增量工具，让你改一行 HotSpot 代码、十几秒就在 GDB 里看到效果。这里先把这个伏笔记下。

下一节，我们把标准的 make 完整跑一遍，把 JDK 真正编出来。
