# Ch0.2 编译你自己的 JDK

## 从一个 C 程序说起

编译一个 hello.c，你我都会：

```
$ gcc hello.c -o hello
```

两步：你没有先检查机器上有没有 gcc，就直接运行了。因为你知道装了。

但如果你的 hello.c 引了 pthread.h：

```c
#include <pthread.h>
#include <stdio.h>

void* worker(void* arg) {
    printf("thread\n");
    return NULL;
}

int main() {
    pthread_t t;
    pthread_create(&t, NULL, worker, NULL);
    pthread_join(t, NULL);
    return 0;
}
```

这时候直接 gcc hello.c -o hello 会报 undefined reference to pthread_create。因为 pthread 不是 libc 的一部分，需要显式告诉编译器去链接 -lpthread。

如果你的程序有 1000 个 .c 文件，依赖十几个系统库，还要兼容 Linux、macOS、Windows 三个平台——你不可能手动敲这些检测和链接参数。你需要一套自动化的「先确认你的环境能不能编，再决定怎么编」的工具。

这就是 configure。

JDK 的源码规模远超「1000 个 .c」——src/hotspot 下近千个 .cpp，src/ 下 80 多个 Java 模块，依赖的本地库涵盖了 X11、CUPS、Freetype 等等。configure 就是这套自动化检测的入口：它在这台机器上逐一确认编译器、头文件、链接库的可用性，然后把结果写入一个叫 spec.gmk 的配置清单。

整个流程和你编译一个多线程 C 程序是一样的，只是每一步都被自动化了：

| 你手动做的 | configure 在做什么 |
|-----------|-------------------|
| which gcc | 检测 GCC 路径和版本（toolchain.m4:463-465） |
| man pthread_create | 确认 libpthread 是否可用（libraries.m4:106：-lpthread） |
| gcc -I/usr/local/include | 设置 JNI 头文件的 include 路径（flags-cflags.m4:470） |
| gcc -g -O0 | 根据 debug level 选优化级别（JvmFlags.gmk:85：slowdebug = -O0 -g -DASSERT） |

## 执行 configure

在 /data/workspace/jdk11u 下执行：

```
$ bash configure \
    --with-debug-level=slowdebug \
    --with-jvm-variants=server \
    --with-boot-jdk=./jdk-11.0.17 \
    --with-native-debug-symbols=none \
    --with-memory-size=32768 \
    --disable-precompiled-headers \
    --disable-javac-server
```

**--with-debug-level=slowdebug**

这是全书最关键的编译选项。它同时踢动了三个开关：编译优化设为零（-O0，每个局部变量都不会被优化掉）、断言宏开启（JvmFlags.gmk:85：-DASSERT，所有 #ifdef ASSERT 代码块都被保留进二进制）、完整调试符号（flags-cflags.m4:121：-g）。三者合在一起的结果就是后面章节用 GDB 能看到的全部信息——变量值、函数名、源码行号。

如果换成 release，这些全部消失。-O2 优化会内联掉短函数、重用寄存器、消除冗余变量，GDB 打断点进去只能看到被折叠后的汇编。

**--with-jvm-variants=server**

HotSpot 支持三类编译变体，这是通过条件编译在同一个源码树上产出的：

- server：C2（Opto）编译器启用，#define COMPILER2 被注入，src/hotspot/share/opto/ 被编译（本书专讲）
- client：仅 C1 编译器，src/hotspot/share/c1/ 被编译
- minimal：没有 JIT，仅解释器运行

本书选 server 因为 JIT 编译器的所有卷（C1/C2/CodeCache）都依赖于 COMPILER2 宏。

**--with-boot-jdk=./jdk-11.0.17**

javac 本身是用 Java 写的。编译 JDK 11 需要一个已经能跑的 JDK 11。这个先于目标 JDK 存在的 JDK 称为 boot JDK——自举。它先帮你编译出 javac，然后刚编出来的 javac 再去编译整个 Java 类库。

**--with-native-debug-symbols=none**

调试符号（-g 的产出）存储在哪里。none 表示直接嵌入在 .so 和可执行文件中。external 则是剥离为独立的 .debuginfo 文件，适合生产环境部署时把它们扔掉。

**--with-memory-size=32768**

告诉 make 这条机器有 32GB 内存。Java 类库编译阶段的 javac 并行度和 GC 堆大小会据此调整，防止编译过程因 OOM 而杀掉进程。

**--disable-precompiled-headers**

HotSpot 使用预编译头来加速编译（把 25 个公共头文件预先编译成 .pch）。但跨 GCC 版本的预编译头兼容性不稳定。关掉最稳妥。

**--disable-javac-server**

JDK 的构建系统默认启动一个常驻的 javac 编译服务器来避免每次 javac 进程启动的开销。某些系统环境下这个服务器会卡死，直接关掉。

## configure 的输出

跑完之后，关键输出如下：

```
A new configuration has been successfully created in
  /data/workspace/jdk11u/build/linux-x86_64-normal-server-slowdebug
  using configure arguments '--with-debug-level=slowdebug ...'.

Configuration summary:
  Debug level:    slowdebug
  HS debug level: slowdebug
  JVM variants:   server
  JVM features:   aot, cds, cmsgc, compiler1, compiler2, g1gc, jfr, ...
  OpenJDK target: linux-x86_64-normal-server-slowdebug
  Boot JDK:       ./jdk-11.0.17
  Tools summary:
    Boot JDK:       jdk-11.0.17
    Toolchain:      gcc (GCC 8.5.0)
    C Compiler:     Version 8.5.0
    C++ Compiler:   Version 8.5.0
  Build performance summary:
    Build jobs:     96
    Memory limit:   32768 MB
```

关键解读：

**Debug level: slowdebug** — 刚才 --with-debug-level 的结果，直接写进了 spec.gmk 的 JVM_CFLAGS，展开就是 -O0 -g -DASSERT。

**JVM features** — configure 根据 variant 选择和系统环境自动决定的特性集。compiler2 就是 server variant 带来的；g1gc、jfr 都是默认启用。如果特意选了 --with-jvm-features=-g1gc，这里就不会出现。

**Toolchain: gcc (GCC 8.5.0)** — toolchain.m4:447-455 通过 $COMPILER --version | grep "Free Software Foundation" 确认是 GCC，toolchain.m4:463-465 从版本里提取出主次版本号，toolchain.m4:56 校对最低版本（≥ 4.8）。

**Build jobs: 96** — configure 自动读取 nproc 的结果。这就是为什么 make 能并行起 96 个编译任务，全量编译几分钟就能跑完。

## configure 产出了什么

根目录下多出一个 build/ 目录，唯一重要的文件是 spec.gmk：

```
build/
└── linux-x86_64-normal-server-slowdebug/
    ├── Makefile
    ├── spec.gmk          ← configure 的全部结果
    └── configure.log
```

spec.gmk 是一个 Make 可引用的变量清单，包含了刚才所有检测的结果。举个例子：

```
$ grep JVM_CFLAGS build/*/spec.gmk
JVM_CFLAGS := -fno-rtti -fno-exceptions -fvisibility=hidden ...
```

这里的 -fno-rtti（禁用 C++ 运行时类型识别）和 -fno-exceptions（禁用 C++ 异常）是 HotSpot 从 C++ 中剥离出来的两个核心设计选择——JVM 不用 C++ 的 dynamic_cast 和 try/catch，而是自己实现了完整的类型系统和错误处理链。这是后面所有分析的物理基础。

configure 通过了，spec.gmk 就绪。接下来 make 命令会读取这一份配置清单，调用子 Makefile 将代码转换为二进制文件。
